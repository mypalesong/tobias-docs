Pull Request (PR)

Pull Request는 GitHub에서 코드 리뷰와 협업을 위한 핵심 기능입니다.

Pull Request란?​

Pull Request(PR)는 내가 작성한 코드를 원본 저장소에 병합해달라고 요청하는 것입니다. 코드 리뷰, 토론, 자동 테스트를 거쳐 코드 품질을 보장합니다.

기본 워크플로우​

1. 브랜치 생성 및 작업​

# 1. 최신 코드 가져오기
git checkout main
git pull

# 2. 새 브랜치 생성
git checkout -b feature/user-authentication

# 3. 작업 수행
# ... 코드 작성 ...

# 4. 커밋
git add .
git commit -m "feat: add user authentication"

# 5. 푸시
git push -u origin feature/user-authentication

2. GitHub에서 PR 생성​

웹 UI 사용​

  1. GitHub 저장소 페이지 방문
  2. "Compare & pull request" 버튼 클릭
  3. 제목과 설명 작성
  4. "Create pull request" 클릭

GitHub CLI 사용​

# PR 생성
gh pr create --title "Add user authentication" --body "Implements JWT-based authentication"

# 또는 대화형으로
gh pr create

# 웹 브라우저로 열기
gh pr create --web

좋은 PR 작성법​

PR 제목​

# 형식
<type>: <description>

# 예시
feat: add user authentication
fix: resolve login redirect issue
docs: update API documentation
refactor: simplify validation logic

PR 설명 템플릿​

## 변경 사항
- JWT 기반 인증 구현
- 로그인/로그아웃 엔드포인트 추가
- 보호된 라우트를 위한 미들웨어 추가

## 테스트
- [ ] 로그인 성공 시나리오
- [ ] 잘못된 자격 증명 처리
- [ ] 토큰 만료 처리
- [ ] 보호된 라우트 접근 제어

## 스크린샷
(해당되는 경우)

## 관련 Issue
Closes #123
Related to #456

## 체크리스트
- [x] 코드가 스타일 가이드를 따름
- [x] 테스트 추가/업데이트됨
- [x] 문서가 업데이트됨
- [x] 모든 테스트가 통과함

PR 리뷰​

리뷰어로서​

# PR 체크아웃
gh pr checkout 123

# 또는
git fetch origin pull/123/head:pr-123
git checkout pr-123

# 변경사항 확인
git diff main

# 로컬에서 테스트
npm test
npm start

리뷰 코멘트 작성​

좋은 리뷰 코멘트:

✅ 이 부분은 별도 함수로 추출하면 재사용성이 높아질 것 같습니다.
✅ 에러 핸들링이 잘 되어 있네요! 👍
✅ 여기서 null 체크를 추가하는 게 어떨까요?

피해야 할 리뷰 코멘트:

❌ 이건 왜 이렇게 했어요?
❌ 이 코드는 별로네요.
❌ 다시 작성하세요.

리뷰 승인/거절​

# 웹에서 리뷰
gh pr review 123 --approve
gh pr review 123 --request-changes
gh pr review 123 --comment

# 코멘트와 함께
gh pr review 123 --approve --body "LGTM! 좋은 구현입니다."

PR 업데이트​

피드백 반영​

# 1. 피드백에 따라 코드 수정
# ... 수정 작업 ...

# 2. 커밋
git add .
git commit -m "fix: address review comments"

# 3. 푸시 (자동으로 PR 업데이트됨)
git push

커밋 정리 (Squash)​

# 마지막 3개 커밋을 하나로 합치기
git rebase -i HEAD~3

# 에디터에서:
# pick -> s (squash)로 변경

# 강제 푸시 (주의!)
git push --force-with-lease

PR 병합 방식​

1. Merge Commit (기본)​

# 모든 커밋 히스토리 보존
gh pr merge 123 --merge

커밋 히스토리:

* Merge pull request #123
|\
| * feat: add feature C
| * feat: add feature B
| * feat: add feature A
|/
* Previous commit

2. Squash and Merge​

# 모든 커밋을 하나로 합침
gh pr merge 123 --squash

커밋 히스토리:

* feat: add features A, B, C (#123)
* Previous commit

3. Rebase and Merge​

# 커밋을 재배치
gh pr merge 123 --rebase

커밋 히스토리:

* feat: add feature C
* feat: add feature B
* feat: add feature A
* Previous commit

Draft PR​

개발 중인 코드를 미리 공유할 때 유용합니다.

# Draft PR 생성
gh pr create --draft --title "WIP: Add new feature"

# Draft에서 Ready로 변경
gh pr ready 123

PR 템플릿​

.github/PULL_REQUEST_TEMPLATE.md 생성​

## 변경 사항 요약
<!-- 이 PR에서 무엇을 변경했는지 간단히 설명하세요 -->

## 변경 타입
- [ ] 버그 수정
- [ ] 새로운 기능
- [ ] 성능 개선
- [ ] 리팩토링
- [ ] 문서 업데이트
- [ ] 테스트 추가

## 테스트
<!-- 어떻게 테스트했는지 설명하세요 -->

## 스크린샷
<!-- UI 변경이 있다면 스크린샷을 첨부하세요 -->

## 관련 Issue
<!-- Closes #123 -->

## 체크리스트
- [ ] 코드가 스타일 가이드를 따름
- [ ] 자체 리뷰를 수행함
- [ ] 복잡한 코드에 주석을 추가함
- [ ] 문서를 업데이트함
- [ ] 변경사항이 경고를 생성하지 않음
- [ ] 테스트를 추가함
- [ ] 새 테스트와 기존 테스트가 모두 통과함

자동화​

GitHub Actions와 연동​

# .github/workflows/pr-checks.yml
name: PR Checks

on:
  pull_request:
    branches: [ main ]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - name: Run tests
        run: npm test

  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - name: Run linter
        run: npm run lint

실습 예제​

예제 1: 첫 PR 만들기​

# 1. Fork (GitHub 웹에서)

# 2. Clone
git clone https://github.com/YOUR_USERNAME/repository.git
cd repository

# 3. 브랜치 생성
git checkout -b fix/typo-in-readme

# 4. 수정
echo "Fixed typo" >> README.md
git add README.md
git commit -m "docs: fix typo in README"

# 5. 푸시
git push -u origin fix/typo-in-readme

# 6. PR 생성
gh pr create --title "Fix typo in README" --body "Fixed a spelling mistake"

예제 2: 리뷰 코멘트 반영​

# 1. PR 브랜치 확인
git checkout feature-branch

# 2. 코멘트에 따라 수정
# ... 수정 작업 ...

# 3. 커밋
git add .
git commit -m "fix: address review comments"

# 4. 푸시 (PR 자동 업데이트)
git push

# 5. 리뷰어에게 알림
gh pr comment 123 --body "리뷰 코멘트 반영했습니다!"

예제 3: 충돌 해결​

# 1. main 브랜치 최신화
git checkout main
git pull

# 2. 기능 브랜치로 전환
git checkout feature-branch

# 3. main을 병합
git merge main
# 또는 rebase
git rebase main

# 4. 충돌 해결
# ... 충돌 파일 수정 ...
git add .
git commit  # merge의 경우
# 또는
git rebase --continue  # rebase의 경우

# 5. 푸시
git push --force-with-lease

고급 팁​

대용량 PR 분할​

# 큰 기능을 여러 작은 PR로 분할
git checkout -b feature/part-1
# ... 일부 작업 ...
git commit -m "feat: implement part 1"
gh pr create

git checkout -b feature/part-2
# ... 나머지 작업 ...
git commit -m "feat: implement part 2"
gh pr create

PR 라벨 활용​

# 라벨 추가
gh pr edit 123 --add-label "bug,high-priority"

# 마일스톤 추가
gh pr edit 123 --milestone "v2.0"

# 리뷰어 할당
gh pr edit 123 --add-reviewer username

PR 체크리스트 자동화​

# .github/pr-labeler.yml
bug:
  - 'fix/*'
  - 'hotfix/*'

feature:
  - 'feat/*'
  - 'feature/*'

documentation:
  - 'docs/*'

일반적인 실수​

1. ❌ 너무 큰 PR​

  • 문제: 리뷰하기 어렵고 병합이 오래 걸림
  • 해결: 작은 단위로 분할

2. ❌ 불명확한 설명​

  • 문제: 리뷰어가 맥락을 이해하지 못함
  • 해결: 상세한 설명과 컨텍스트 제공

3. ❌ 테스트 없음​

  • 문제: 버그 발생 가능성 증가
  • 해결: 항상 테스트 포함

4. ❌ main에 직접 푸시​

  • 문제: 리뷰 없이 병합됨
  • 해결: 브랜치 보호 규칙 설정
