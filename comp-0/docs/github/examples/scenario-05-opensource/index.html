시나리오 5: 오픈소스 기여하기

2단계: Issue 생성​

Title: Bug: DatePicker shows wrong month when selecting last day

## Description
When selecting the last day of a month (e.g., Jan 31), the picker
sometimes shows the next month (Feb) instead.

## Steps to Reproduce
1. Open datepicker
2. Select January 31st
3. Close and reopen picker
4. Bug: Shows February instead of January

## Expected Behavior
Should show January with 31st selected

## Actual Behavior
Shows February

## Environment
- react-awesome-datepicker: v3.2.1
- React: 18.2.0
- Browser: Chrome 120

## Possible Fix
I think the issue is in `src/utils/dateCalculations.js` line 45.
I'd like to fix this if you approve.

## Additional Context
(Screenshot attached)

6단계: 버그 수정​

문제가 있는 코드 발견 (src/utils/dateCalculations.js):

// 🐛 버그가 있는 코드
export function normalizeDate(date) {
  const normalized = new Date(date);
  normalized.setDate(1); // ❌ 항상 1일로 설정하는 게 문제!
  return normalized;
}

수정:

// ✅ 수정된 코드
export function normalizeDate(date) {
  // 원래 날짜 유지
  return new Date(date);
}

7단계: 테스트 추가​

새 테스트 작성 (src/utils/__tests__/dateCalculations.test.js):

describe('normalizeDate', () => {
  // 기존 테스트...

  // 새로 추가한 테스트
  it('should preserve the last day of month', () => {
    const jan31 = new Date(2024, 0, 31); // Jan 31, 2024
    const normalized = normalizeDate(jan31);

    expect(normalized.getMonth()).toBe(0); // January
    expect(normalized.getDate()).toBe(31); // 31st
  });

  it('should handle February edge cases', () => {
    const feb29 = new Date(2024, 1, 29); // Feb 29, 2024 (leap year)
    const normalized = normalizeDate(feb29);

    expect(normalized.getMonth()).toBe(1); // February
    expect(normalized.getDate()).toBe(29);
  });
});

2시간 후, 메인테이너의 리뷰:

@john-maintainer:

Great fix! Just a few minor suggestions:

  1. Could you add a JSDoc comment explaining why we don't normalize anymore?
  2. The test case for leap year is excellent, but could you also add a test for non-leap year February?

민지, 피드백 반영:

/**
 * Normalizes a date object
 * @param {Date} date - The date to normalize
 * @returns {Date} A new Date object preserving the original day
 * @note Previous implementation set day to 1, causing issues with
 *       end-of-month dates. Now preserves the original date.
 */
export function normalizeDate(date) {
  return new Date(date);
}
// 테스트 추가
it('should handle non-leap year February', () => {
  const feb28 = new Date(2023, 1, 28); // Feb 28, 2023 (non-leap)
  const normalized = normalizeDate(feb28);

  expect(normalized.getMonth()).toBe(1);
  expect(normalized.getDate()).toBe(28);
});
# 변경사항 커밋

git commit -m "docs: add JSDoc and additional test case

- Add JSDoc explaining normalization behavior
- Add test for non-leap year February

Co-authored-by: John Maintainer <john@example.com>"

💡 실전 팁​

1. 좋은 첫 이슈 찾기​

GitHub 검색:

- label:"documentation"

2. PR 크기 작게 유지​

✅ Good PR:
- 50-200 lines
- 1가지 목적
- 명확한 설명

❌ Bad PR:
- 1000+ lines
- 여러 기능 동시에
- 설명 부족

3. 메인테이너와 소통​

- 예상 동작
- 실제 동작
- 환경 정보

# PR에서
- 인내심 갖기
- 피드백에 감사 표현
- 건설적인 토론

4. Upstream 동기화 자동화​

# .git/config에 추가
[remote "upstream"]
    url = https://github.com/original/repo.git
    fetch = +refs/heads/*:refs/remotes/upstream/*

# Alias 설정
git config alias.sync '!git fetch upstream && git merge upstream/main'

# 사용
git sync
