시나리오 11: Code Review 문화 만들기

팀에 건강한 코드 리뷰 문화를 정착시킵니다.

📌 상황​

코드 리뷰가 형식적으로만 이루어지고 있습니다. "LGTM"만 남기고 제대로 보지 않는 팀원들. 준호가 이를 개선하려 합니다.

🎯 목표​

  • 효과적인 코드 리뷰 방법
  • PR 템플릿과 체크리스트
  • 리뷰 자동화 도구 활용
  • 건설적인 피드백 문화

💬 대화​

준호: "코드 리뷰가 너무 형식적이에요. 실제로 문제를 찾아내야 하는데..."

현우: "맞아요. 다들 바빠서 대충 보는 것 같아요."

민지: "저는 리뷰를 어떻게 해야 할지 잘 모르겠어요..."

2단계: 리뷰 자동화​

# .github/workflows/pr-checks.yml
name: PR Checks

on:
  pull_request:

jobs:
  size-label:
    runs-on: ubuntu-latest
    steps:
      - name: Label PR by size
        uses: codelytv/pr-size-labeler@v1
        with:
          xs_label: 'size/XS'
          xs_max_size: 10
          s_label: 'size/S'
          s_max_size: 100
          m_label: 'size/M'
          m_max_size: 500
          l_label: 'size/L'
          l_max_size: 1000
          xl_label: 'size/XL'

  complexity:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - name: Check code complexity
        run: npx complexity-report src/

4단계: 리뷰 가이드라인​

# Code Review 가이드

## 리뷰어가 확인할 것

### 🏗️ 아키텍처
- [ ] 적절한 디자인 패턴 사용
- [ ] 코드 위치가 적절함
- [ ] 불필요한 의존성 없음

### 🐛 버그 가능성
- [ ] Edge case 처리
- [ ] 에러 핸들링
- [ ] Null/Undefined 체크

### 📈 성능
- [ ] 불필요한 연산 없음
- [ ] 메모리 누수 가능성 없음
- [ ] N+1 쿼리 없음

### 🧪 테스트
- [ ] 적절한 테스트 커버리지
- [ ] Edge case 테스트
- [ ] 테스트가 실제 동작 검증

### 📚 코드 품질
- [ ] 읽기 쉬움
- [ ] 적절한 추상화
- [ ] 중복 코드 없음
- [ ] 네이밍이 명확

## 좋은 리뷰 코멘트 예시

✅ "이 부분은 `useMemo`를 사용하면 리렌더링을 줄일 수 있을 것 같아요."

✅ "여기서 user가 null일 때를 처리해야 할 것 같습니다. 5번 라인에서 user.name을 사용하고 있어요."

✅ "좋은 구현입니다! 다만 `getUserData` 함수를 별도 유틸로 분리하면 재사용하기 좋을 것 같아요."

## 피해야 할 코멘트

❌ "왜 이렇게 했어요?"
❌ "다시 작성하세요."

5단계: PR 크기 제한​

# .github/workflows/pr-size-check.yml
name: PR Size Check

on: [pull_request]

jobs:
  check-size:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - name: Check PR size
        run: |
          LINES=$(git diff --stat origin/main... | tail -1 | awk '{print $4+$6}')
          if [ $LINES -gt 500 ]; then
            echo "❌ PR is too large ($LINES lines)"
            echo "Please split into smaller PRs"
            exit 1
          fi

2. 리뷰 통계 추적​

# GitHub CLI로 통계
gh pr list --author=@me --state=merged --json number,reviewDecision

# 리뷰 반응 시간 측정
gh api graphql -f query='
{
  repository(owner:"org", name:"repo") {
    pullRequests(last:10) {
      nodes {
        createdAt
        reviews(first:1) {
          nodes {
            createdAt
          }
        }
      }
    }
  }
}'
