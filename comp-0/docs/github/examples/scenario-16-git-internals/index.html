시나리오 16: Git 내부 동작 이해

Git의 내부 구조를 파헤쳐 진정한 Git 마스터가 됩니다.

📌 상황​

준호가 팀원들에게 "Git 내부 동작 원리" 세미나를 준비합니다. "왜 Git은 이렇게 동작할까?"라는 질문에 답하기 위해 .git 폴더 안을 깊이 탐험합니다.

🎯 목표​

  • .git 폴더 구조 이해
  • Git Objects (blob, tree, commit, tag)
  • Refs와 HEAD의 역할
  • Packfile과 Delta 압축
  • Plumbing vs Porcelain 명령어
  • Git의 성능 비밀

💬 대화​

민지: "Git commit하면 실제로 무슨 일이 일어나요?"

준호: "좋은 질문이에요! .git 폴더 안을 직접 보면서 배워볼까요?"

현우: "Git이 왜 이렇게 빠른지도 궁금해요."

준호: "그것도 내부 구조를 보면 이해할 수 있어요. 시작해봅시다!"

⚡ 실습​

1단계: 빈 저장소로 시작​

# 새 저장소 생성
mkdir git-internals-demo
cd git-internals-demo
git init

# .git 폴더 구조 확인
tree .git

# .git/
# ├── HEAD              # 현재 브랜치 포인터
# ├── config            # 저장소 설정
# ├── description       # 저장소 설명
# ├── hooks/            # Git hooks
# ├── info/             # exclude 등
# ├── objects/          # 모든 Git 객체 저장소
# │   ├── info/
# │   └── pack/
# └── refs/             # 브랜치와 태그 참조
#     ├── heads/        # 로컬 브랜치
#     └── tags/         # 태그

# HEAD 내용 확인
cat .git/HEAD
# ref: refs/heads/main

2단계: Git Objects - Blob​

# 파일 생성
echo "Hello Git" > hello.txt

# Staged 전 확인
ls -la .git/objects/
# info/  pack/  (비어있음)

# Stage
git add hello.txt

# Objects 폴더 확인
ls -la .git/objects/
# 9f/  info/  pack/

# 전체 객체 확인
find .git/objects -type f
# .git/objects/9f/4d96d5b00d98959ea9960f069585ce42b1349a

# 객체 내용 보기
git cat-file -p 9f4d96d5b00d98959ea9960f069585ce42b1349a
# Hello Git

# 객체 타입 확인
git cat-file -t 9f4d96d5
# blob

# 객체 크기
git cat-file -s 9f4d96d5
# 10

중요 개념: Blob은 파일 내용만 저장. 파일명은 저장 안 함!

3단계: Hash 생성 원리​

# Git이 hash를 계산하는 방법
echo -n "Hello Git" | git hash-object --stdin
# 9f4d96d5b00d98959ea9960f069585ce42b1349a

# 직접 계산해보기
(echo -n "blob 10\0"; echo -n "Hello Git") | sha1sum
# 9f4d96d5b00d98959ea9960f069585ce42b1349a

# 과정:
# 1. "blob <크기>\0<내용>" 형식
# 2. SHA-1 해시 계산
# 3. 앞 2자는 폴더명, 나머지 38자는 파일명

4단계: Git Objects - Tree​

# 커밋 생성
git commit -m "First commit"

# 커밋 객체 찾기
git log --oneline
# a1b2c3d First commit

# 커밋 내용 보기
git cat-file -p a1b2c3d
# tree 4d5e6f7...
# author 준호 <junho@example.com> 1234567890 +0900
# committer 준호 <junho@example.com> 1234567890 +0900
#
# First commit

# Tree 객체 보기
git cat-file -p 4d5e6f7
# 100644 blob 9f4d96d5b00d98959ea9960f069585ce42b1349a	hello.txt

# Tree는 파일명과 blob을 매핑!

Tree 구조 시각화:

# 디렉토리 추가
mkdir src
echo "console.log('hi')" > src/app.js
git add .
git commit -m "Add src"

# Tree 구조 확인
git cat-file -p HEAD^{tree}
# 100644 blob 9f4d96d5	hello.txt
# 040000 tree abc1234	src

# src 디렉토리의 tree
git cat-file -p abc1234
# 100644 blob def5678	app.js
commit (HEAD)
  │
  └── tree (root)
        ├── blob (hello.txt)
        └── tree (src/)
              └── blob (app.js)

5단계: Git Objects - Commit​

# 두 번째 커밋
echo "Updated" >> hello.txt
git commit -am "Update hello"

# 새 커밋 확인
git cat-file -p HEAD
# tree xyz789...
# parent a1b2c3d...      # 이전 커밋!
# author 준호 <junho@example.com> 1234567890 +0900
# committer 준호 <junho@example.com> 1234567890 +0900
#
# Update hello

# 커밋 체인 확인
git log --oneline --graph
# * b2c3d4e Update hello
# * a1b2c3d First commit

Commit은 연결 리스트:

[Commit 3] ← HEAD
    ↓ parent
[Commit 2]
    ↓ parent
[Commit 1]
    ↓ parent
  (null)

6단계: Refs와 HEAD​

# HEAD 확인
cat .git/HEAD
# ref: refs/heads/main

# main 브랜치 확인
cat .git/refs/heads/main
# b2c3d4e8f1a2b3c4d5e6f7a8b9c0d1e2f3a4b5c6 (커밋 해시)

# 브랜치 생성
git branch feature
cat .git/refs/heads/feature
# b2c3d4e8f1a2b3c4d5e6f7a8b9c0d1e2f3a4b5c6 (같은 해시!)

# 브랜치 전환
git checkout feature
cat .git/HEAD
# ref: refs/heads/feature

# HEAD가 가리키는 것 확인
git symbolic-ref HEAD
# refs/heads/feature

Detached HEAD:

# 특정 커밋으로 이동
git checkout a1b2c3d

cat .git/HEAD
# a1b2c3d... (커밋 해시 직접!)
# ref가 아니라 해시 - Detached HEAD!

# 다시 브랜치로
git checkout main

7단계: Plumbing 명령어​

# Porcelain (일상 명령어)
git add, git commit, git push, git pull

# Plumbing (저수준 명령어)
git hash-object   # 객체 해시 계산
git cat-file      # 객체 내용 보기
git update-ref    # ref 업데이트
git symbolic-ref  # symbolic ref 조작

# 수동으로 브랜치 생성
git update-ref refs/heads/manual abc1234
# = git branch manual abc1234

# 수동으로 커밋 만들기
# 1. Blob 생성
echo "test" | git hash-object -w --stdin
# 9daeafb9864cf43055ae93beb0afd6c7d144bfa4

# 2. Tree 생성
git mktree << EOF
100644 blob 9daeafb9864cf43055ae93beb0afd6c7d144bfa4	test.txt
EOF
# 4b825dc642cb6eb9a060e54bf8d69288fbee4904

# 3. Commit 생성
git commit-tree 4b825dc6 -m "Manual commit"
# 7c3a8c8...

# 4. 브랜치 연결
git update-ref refs/heads/manual-branch 7c3a8c8

8단계: Packfile과 압축​

# 큰 파일 여러 버전 생성
for i in {1..100}; do
  echo "Version $i: $(cat /dev/urandom | head -c 1000)" > large.txt
  git add large.txt
  git commit -m "Version $i"
done

# Objects 개수
find .git/objects -type f | wc -l
# 303 (많음!)

# 크기
du -sh .git/objects
# 1.2M

# Garbage Collection (Packing)
git gc

# 다시 확인
find .git/objects -type f | wc -l
# 3

ls .git/objects/pack/
# pack-abc123.idx
# pack-abc123.pack

du -sh .git/objects
# 150K (압축됨!)

Delta 압축 확인:

# Packfile 내용 보기
git verify-pack -v .git/objects/pack/pack-*.idx | head -20

# 출력 예시
# abc1234 commit 234 156 12
# def5678 tree   85  90  146
# ghi9012 blob   1000 900 236
# jkl3456 blob   1000 50  1136 1 ghi9012  # Delta!

# jkl3456은 ghi9012의 차이(delta)만 저장
# 원본 1000 bytes → Delta 50 bytes (95% 압축!)

9단계: Reflog - 숨겨진 안전망​

# Reflog 확인
git reflog
# b2c3d4e HEAD@{0}: commit: Update hello
# a1b2c3d HEAD@{1}: commit (initial): First commit

# Reflog 파일 직접 보기
cat .git/logs/HEAD

# Reset 해도 복구 가능
git reset --hard HEAD~10
git reflog
# ...
# a1b2c3d HEAD@{11}: reset: moving to HEAD~10

# 복구
git reset --hard HEAD@{10}

10단계: Git의 성능 비밀​

1. Content-Addressable Storage

# 같은 내용 = 같은 해시 = 한 번만 저장
echo "Same content" > file1.txt
echo "Same content" > file2.txt
git add .

# Blob은 하나만 생성됨!
find .git/objects -type f
# (blob 1개만 추가)

2. Shallow Copy (Branch)

# 브랜치는 단순히 ref 파일 하나 (41 bytes)
git branch new-feature

ls -lh .git/refs/heads/new-feature
# -rw-r--r-- 1 user user 41 Nov 15 10:00 new-feature

# 파일 복사 없음 - 포인터만!

3. Delta Compression

# 99% 유사한 파일 100개 = 1개 + (차이 × 99)

11단계: 실전 디버깅​

# 저장소 상태 검사
git fsck --full
# Checking object directories: 100% (256/256), done.
# Checking objects: 100% (303/303), done.

# 손상된 객체 찾기
git fsck --lost-found

# 객체 참조 추적
git rev-list --objects --all | \
  grep <hash>

# 특정 객체 누가 참조하는지
git show <hash>

민지의 깨달음:

# "이제 이해했어요!"

# git add = blob 생성
# git commit = tree + commit 객체 생성
# git branch = ref 파일 생성 (포인터)
# git checkout = HEAD를 다른 ref로 이동
# git merge = 새 commit 생성 (parent 2개)

✅ 검증​

# Git 저장소 완전 분해
cat > inspect-repo.sh << 'EOF'
#!/bin/bash

echo "=== Git Repository Internals ==="

# HEAD
echo -e "\n📍 HEAD:"
cat .git/HEAD

# Branches
echo -e "\n🌿 Branches:"
ls .git/refs/heads/

# Objects 통계
echo -e "\n📦 Objects:"
git count-objects -v

# Recent commits
echo -e "\n📝 Recent Commits:"
git log --oneline -5

# Objects breakdown
echo -e "\n🔍 Objects by type:"
git cat-file --batch-check='%(objecttype)' --batch-all-objects | \
  sort | uniq -c

EOF

chmod +x inspect-repo.sh
./inspect-repo.sh

출력 예시:

=== Git Repository Internals ===

📍 HEAD:
ref: refs/heads/main

🌿 Branches:
feature
main
manual

📦 Objects:
count: 15
size: 1.2 MiB
in-pack: 300
packs: 1
size-pack: 150 KiB

📝 Recent Commits:
b2c3d4e Update hello
a1b2c3d First commit

🔍 Objects by type:
    150 blob
    100 tree
     50 commit
      1 tag

🎓 배운 내용​

Git Objects 4가지​

# 1. Blob - 파일 내용
git hash-object -w file.txt

# 2. Tree - 디렉토리 구조
git cat-file -p HEAD^{tree}

# 3. Commit - 스냅샷 + 메타데이터
git cat-file -p HEAD

# 4. Tag - 커밋에 이름 붙이기
git tag -a v1.0 -m "Version 1.0"

Git의 3가지 영역​

Working Directory
       ↓ git add (blob 생성)
  Staging Area (.git/index)
       ↓ git commit (tree + commit 생성)
   Repository (.git/objects)

Refs 구조​

.git/
├── HEAD                    # 현재 위치
├── refs/
│   ├── heads/
│   │   ├── main           # 브랜치
│   │   └── feature
│   ├── tags/
│   │   └── v1.0           # 태그
│   └── remotes/
│       └── origin/
│           └── main       # 원격 브랜치

💡 실전 팁​

1. 큰 파일 찾아서 제거​

# 큰 파일 찾기
git rev-list --objects --all | \
  git cat-file --batch-check='%(objecttype) %(objectname) %(objectsize) %(rest)' | \
  awk '/^blob/ {print substr($0,6)}' | \
  sort -k2 -n -r | \
  head -10

# 특정 파일 히스토리에서 제거
git filter-repo --path large-file.zip --invert-paths

2. 손상된 저장소 복구​

# 1. 백업
cp -r .git .git.backup

# 2. 검사
git fsck --full

# 3. 복구
git reflog expire --expire=now --all
git gc --prune=now --aggressive

# 4. 안 되면 클론
git clone --mirror <url> .git.new

3. 성능 최적화​

# 정기적인 GC
git gc --auto

# 적극적인 압축
git repack -a -d --depth=250 --window=250

# 통계
git count-objects -v -H

4. Git 내부로 문제 해결​

# 브랜치 수동 복구
git update-ref refs/heads/lost-branch <commit-hash>

# Tag 수동 생성
git update-ref refs/tags/v1.0 <commit-hash>

# HEAD 수동 이동
git symbolic-ref HEAD refs/heads/main

🎓 퀴즈 타임!​

Q1: git add 하면 무슨 일이 일어날까요?

답 보기
  1. 파일 내용으로 SHA-1 해시 계산
  2. .git/objects/xx/yyyy... 에 blob 객체 생성
  3. .git/index (staging area) 업데이트

Q2: 브랜치를 100개 만들면 디스크 공간이 얼마나 필요할까요?

답 보기

약 4.1KB (브랜치당 41 bytes)

브랜치는 단순히 커밋 해시를 가리키는 파일!

Q3: Git은 변경 사항(delta)을 저장할까요, 전체 스냅샷을 저장할까요?

답 보기

스냅샷을 저장합니다!

하지만:

  • Commit 시: 전체 스냅샷
  • Pack 시: Delta로 압축

"스냅샷으로 저장, Delta로 압축"
