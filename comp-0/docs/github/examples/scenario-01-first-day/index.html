시나리오 1: 민지의 첫 출근

민지가 스타트업 '코드크래프트'에 프론트엔드 개발자로 첫 출근하는 날입니다.

📌 상황 설명​

오늘은 민지의 첫 출근일입니다. 오전 10시, 준호 시니어가 민지를 맞이하며 개발 환경 설정을 도와줍니다. 민지는 학교 프로젝트로 Git을 조금 써봤지만, 실무에서 팀과 협업하는 것은 처음입니다.

🎯 목표​

  • 프로젝트 저장소를 로컬에 설정
  • 첫 번째 작업 수행
  • Pull Request 생성 및 코드 리뷰 받기
  • 팀의 Git 워크플로우 이해

💬 대화​

준호: "민지님 첫 출근 축하해요! 오늘은 개발 환경 세팅하고, 간단한 첫 작업을 해볼 거예요."

민지: "네! 긴장되지만 열심히 배우겠습니다."

준호: "우리 팀은 GitHub Flow를 사용해요. main 브랜치는 항상 배포 가능한 상태를 유지하고, 모든 작업은 브랜치에서 하죠. Pull Request를 통해 코드 리뷰를 받은 후에 병합합니다."

민지: "알겠습니다! 그럼 바로 시작해볼까요?"

⚡ 실습​

1단계: 개발 환경 설정​

준호: "먼저 Git 설정부터 해볼게요."

# Git 사용자 정보 설정
git config --global user.name "민지"
git config --global user.email "minji@codecraft.com"

# 기본 에디터 설정 (VS Code 사용)
git config --global core.editor "code --wait"

# 기본 브랜치 이름 설정
git config --global init.defaultBranch main

# 설정 확인
git config --list

민지: "완료했어요!"

2단계: 프로젝트 저장소 클론​

준호: "좋아요. 이제 우리 프로젝트를 로컬로 가져올게요. GitHub에서 접근 권한은 이미 드렸어요."

# 프로젝트 저장소 클론
git clone https://github.com/codecraft/shop-frontend.git

# 디렉토리 이동
cd shop-frontend

# 저장소 상태 확인
git status

# 브랜치 확인
git branch -a

# 최근 커밋 확인
git log --oneline -5

민지: "와, 코드가 많네요!"

준호: "네, 하지만 걱정 마세요. 천천히 익숙해질 거예요."

3단계: 첫 번째 작업 - 환영 메시지 추가​

준호: "첫 작업은 간단해요. 홈페이지에 환영 메시지를 추가하는 거예요. Issue #42를 할당했으니 확인해보세요."

민지: "확인했어요! 지금 바로 시작할게요."

# 최신 코드 받기
git checkout main
git pull origin main

# 새 브랜치 생성
git checkout -b feature/welcome-message

# 브랜치 확인
git branch
# * feature/welcome-message
#   main

민지: "브랜치를 만들었어요. 이제 코드를 작성하면 되나요?"

준호: "맞아요! VS Code로 파일을 열어볼까요?"

# VS Code 열기
code .

민지, src/components/HomePage.jsx 파일을 수정합니다:

// 수정 전
export default function HomePage() {
  return (
    <div className="home">
      <h1>쇼핑몰</h1>
      <ProductList />
    </div>
  );
}

// 수정 후
export default function HomePage() {
  return (
    <div className="home">
      <h1>쇼핑몰</h1>
      <p className="welcome-message">안녕하세요! 코드크래프트 쇼핑몰에 오신 것을 환영합니다.</p>
      <ProductList />
    </div>
  );
}

CSS도 추가합니다 (src/styles/HomePage.css):

.welcome-message {
  font-size: 1.2rem;
  color: #666;
  margin: 20px 0;
  text-align: center;
}

4단계: 변경사항 확인 및 커밋​

# 변경된 파일 확인
git status

# 변경 내용 상세히 보기
git diff

# 파일 스테이징
git add src/components/HomePage.jsx
git add src/styles/HomePage.css

# 스테이징된 내용 확인
git diff --staged

# 커밋
git commit -m "feat: add welcome message to home page

- Add welcome message paragraph to HomePage component
- Add CSS styling for welcome message
- Closes #42"

준호: "커밋 메시지 잘 작성했네요! Conventional Commits 형식을 따르고, 이슈 번호도 참조했어요."

민지: "감사합니다! 다음은 뭐죠?"

5단계: 푸시 및 Pull Request 생성​

# 브랜치를 원격 저장소에 푸시
git push -u origin feature/welcome-message

준호: "터미널에 PR 생성 링크가 나왔죠? GitHub 웹으로 가서 PR을 만들어볼게요."

GitHub 웹에서:

  1. "Compare & pull request" 버튼 클릭
  2. PR 작성:
## 변경 사항
홈페이지에 환영 메시지를 추가했습니다.

## 스크린샷
(스크린샷 첨부)

## 테스트
- [x] 로컬에서 동작 확인
- [x] 반응형 디자인 확인

## 관련 Issue
Closes #42

## 체크리스트
- [x] 코드가 스타일 가이드를 따름
- [x] 필요한 CSS 추가됨
- [x] 변경사항이 의도한 대로 동작함
  1. "Create pull request" 클릭

민지: "PR 만들었어요!"

6단계: 코드 리뷰 받기​

5분 후, 준호가 리뷰를 남깁니다:

준호의 코멘트:

좋은 첫 PR이에요! 몇 가지 제안사항이 있습니다:

  1. welcome-message 클래스명을 CSS 모듈로 변경하면 어떨까요?
  2. 접근성을 위해 semantic HTML을 사용하면 좋을 것 같아요.

민지: "알겠습니다! 바로 수정할게요."

7단계: 피드백 반영​

# 같은 브랜치에서 작업
git checkout feature/welcome-message

# 파일 수정 후...
git add .
git commit -m "refactor: improve welcome message accessibility

- Use semantic HTML (section tag)
- Convert to CSS module
- Add aria-label for better accessibility"

# 푸시 (PR이 자동으로 업데이트됨)
git push

준호: "완벽해요! Approve 할게요. 수고했어요!"

8단계: PR 병합​

준호가 "Squash and merge" 버튼을 클릭합니다.

# 로컬에서 main 업데이트
git checkout main
git pull origin main

# 작업 브랜치 삭제
git branch -d feature/welcome-message

# 원격 브랜치도 삭제됨 (GitHub에서 자동 삭제 설정됨)

민지: "첫 PR이 병합되었어요! 🎉"

준호: "축하해요! 이제 팀의 일원이에요. 점심 먹고 다음 작업 시작해볼까요?"

✅ 검증​

다음 항목들을 확인해보세요:

# main 브랜치에 변경사항이 반영되었는지 확인
git log --oneline -3

# 원격 저장소와 동기화되었는지 확인
git status

# 로컬 브랜치 정리 확인
git branch
# * main

🎓 배운 내용​

Git 기본 워크플로우​

  1. git clone - 저장소 복제
  2. git checkout -b - 브랜치 생성 및 전환
  3. git add - 변경사항 스테이징
  4. git commit - 커밋 생성
  5. git push - 원격에 푸시
  6. git pull - 최신 코드 가져오기

GitHub 협업​

  • Pull Request 생성
  • 코드 리뷰 받기
  • 피드백 반영
  • PR 병합

커밋 메시지 작성​

<type>: <subject>

<body>

<footer>
  • feat: 새로운 기능
  • fix: 버그 수정
  • refactor: 리팩토링
  • docs: 문서 수정

💡 실전 팁​

1. 자주 Pull 하기​

# 하루 일과 시작 시
git checkout main
git pull

# 작업 브랜치 업데이트
git checkout feature/my-work
git rebase main

2. 커밋 전에 확인​

# 변경사항 확인
git diff

# 스테이징된 내용 확인
git diff --staged

# 린트 실행
npm run lint

# 테스트 실행
npm test

3. PR 크기 작게 유지​

  • 200-400줄 이하 권장
  • 한 가지 목적에 집중
  • 리뷰하기 쉽게

4. 브랜치 명명 규칙​

feature/기능명      # 새 기능
fix/버그명         # 버그 수정
refactor/내용     # 리팩토링
docs/내용         # 문서
