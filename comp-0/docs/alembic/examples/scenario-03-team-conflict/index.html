시나리오 3: 민호와 수진의 충돌 해결

두 개발자가 동시에 마이그레이션을 만들어서 충돌이 발생한 상황입니다.

📌 상황 설명​

같은 프로젝트에서 일하는 민호와 수진. 민호는 결제 시스템을, 수진은 리뷰 시스템을 각자 개발하고 있습니다. 두 사람 모두 마이그레이션을 만들었는데, 이제 브랜치를 병합해야 합니다.

🎯 목표​

  • 마이그레이션 브랜치 충돌 이해
  • 올바른 병합 방법 배우기
  • 마이그레이션 체인 재구성

💬 대화​

민호: "수진님, 제 결제 기능 개발 끝났어요. main에 병합해도 될까요?"

수진: "저도 방금 리뷰 기능 완료했는데요. 같이 병합하면 문제없을까요?"

시니어 재석: "둘 다 마이그레이션을 만들었다면, 충돌이 날 수 있어요. 제가 도와드릴게요."

📊 상황 파악​

민호의 브랜치 (feature/payments)​

# 민호의 마이그레이션
alembic history
# abc123 -> def456 (head), add payments table
# <base> -> abc123, initial schema
# versions/def456_add_payments_table.py
"""add payments table

Revision ID: def456
Revises: abc123  # ← main 브랜치의 마지막 리비전
"""

def upgrade():
    op.create_table('payments',
        sa.Column('id', sa.Integer(), primary_key=True),
        sa.Column('order_id', sa.Integer(), nullable=False),
        sa.Column('amount', sa.Numeric(12, 2), nullable=False),
        sa.Column('status', sa.String(20), default='pending'),
        sa.Column('payment_method', sa.String(50)),
        sa.Column('created_at', sa.DateTime(), server_default=sa.func.now()),
        sa.ForeignKeyConstraint(['order_id'], ['orders.id'])
    )

수진의 브랜치 (feature/reviews)​

# 수진의 마이그레이션
alembic history
# abc123 -> ghi789 (head), add reviews table
# <base> -> abc123, initial schema
# versions/ghi789_add_reviews_table.py
"""add reviews table

Revision ID: ghi789
Revises: abc123  # ← 똑같이 abc123을 참조!
"""

def upgrade():
    op.create_table('reviews',
        sa.Column('id', sa.Integer(), primary_key=True),
        sa.Column('product_id', sa.Integer(), nullable=False),
        sa.Column('user_id', sa.Integer(), nullable=False),
        sa.Column('rating', sa.Integer(), nullable=False),
        # ...
    )

문제 발견!​

main 브랜치:
  abc123 (initial schema)

민호 브랜치:              수진 브랜치:
  abc123 → def456         abc123 → ghi789
    ↓                       ↓
  payments               reviews

둘 다 abc123에서 분기! → Multiple heads 발생!

⚡ 해결 방법​

방법 1: 리베이스 (권장)​

재석: "먼저 한 사람의 브랜치를 main에 병합하고, 다른 사람이 리베이스하는 게 좋아요."

1단계: 민호의 브랜치 먼저 병합​

# main 브랜치로 전환
git checkout main

# 민호의 브랜치 병합
git merge feature/payments

# 마이그레이션 적용
alembic upgrade head

# 히스토리 확인
alembic history
# abc123 -> def456 (head), add payments table
# <base> -> abc123, initial schema

# Git 푸시
git push origin main

2단계: 수진의 브랜치 업데이트​

# 수진의 브랜치로 전환
git checkout feature/reviews

# main 최신 변경사항 가져오기
git fetch origin
git rebase origin/main

# ⚠️ 충돌 발생하지 않음 (마이그레이션 파일은 다름)

3단계: 수진의 마이그레이션 수정​

# 현재 마이그레이션 파일 확인
cat alembic/versions/ghi789_add_reviews_table.py

재석: "수진님의 마이그레이션이 여전히 abc123을 참조하고 있어요. def456으로 바꿔야 해요."

# versions/ghi789_add_reviews_table.py 수정
"""add reviews table

Revision ID: ghi789
Revises: def456  # ← abc123에서 def456으로 변경!
Create Date: 2024-01-22 15:00:00.123456
"""

def upgrade():
    # ... (내용 동일)

4단계: 테스트​

# 최신 main으로 업그레이드
alembic upgrade head
# def456 상태

# 수진의 마이그레이션 적용
alembic upgrade head
# INFO  Running upgrade def456 -> ghi789, add reviews table

# 히스토리 확인
alembic history
# def456 -> ghi789 (head), add reviews table
# abc123 -> def456, add payments table
# <base> -> abc123, initial schema

5단계: 병합 완료​

# Git 커밋
git add alembic/versions/ghi789_add_reviews_table.py
git commit -m "fix: update review migration to follow payments migration"

# main에 병합
git checkout main
git merge feature/reviews
git push origin main

성공!

최종 마이그레이션 체인:
  <base> → abc123 → def456 → ghi789
           initial  payments reviews

방법 2: Merge 명령어 사용​

재석: "또 다른 방법은 Alembic의 merge 명령어를 사용하는 거예요."

Multiple heads 상황​

# 두 브랜치 모두 main에 병합된 상황
git checkout main
git merge feature/payments
git merge feature/reviews

# 마이그레이션 상태 확인
alembic heads
# def456
# ghi789
# (2 heads!)

alembic upgrade head
# ERROR: Multiple heads are present

Merge 마이그레이션 생성​

alembic merge -m "merge payments and reviews branches" def456 ghi789

생성된 파일:

"""merge payments and reviews branches

Revision ID: jkl012
Revises: def456, ghi789
Create Date: 2024-01-22 16:00:00.123456
"""

revision = 'jkl012'
down_revision = ('def456', 'ghi789')  # 두 개의 부모!
branch_labels = None
depends_on = None

def upgrade():
    pass  # 병합만 하므로 비어있음

def downgrade():
    pass
# 병합 마이그레이션 적용
alembic upgrade head

# 히스토리 확인
alembic history --verbose
#       ┌─ def456 (payments)
# abc123 ┤
#       └─ ghi789 (reviews)
#           ↓
#        jkl012 (merge, head)

🎨 복잡한 시나리오: 3명이 동시 작업​

상황​

main: abc123

철수 (feature/cart):     abc123 → aaa111 (장바구니)
영희 (feature/wishlist): abc123 → bbb222 (위시리스트)
민수 (feature/coupons):  abc123 → ccc333 (쿠폰)

순차 병합 전략​

# 1. 철수 먼저 병합
git checkout main
git merge feature/cart
# abc123 → aaa111

# 2. 영희 리베이스
git checkout feature/wishlist
git rebase main
# versions/bbb222: Revises: aaa111로 수정
# abc123 → aaa111 → bbb222

# 3. 민수 리베이스
git checkout feature/coupons
git rebase main
# versions/ccc333: Revises: bbb222로 수정
# abc123 → aaa111 → bbb222 → ccc333

# 최종 체인
alembic history
# abc123 -> aaa111 -> bbb222 -> ccc333 (head)

💡 실전 팁​

1. 작업 전 main 업데이트​

# ✅ 좋은 습관
git checkout main
git pull
git checkout -b feature/new-feature

# 최신 상태에서 마이그레이션 생성
alembic upgrade head
alembic revision --autogenerate -m "add new feature"

2. 마이그레이션 파일 이름 확인​

# down_revision 자동 확인 스크립트
cat alembic/versions/*.py | grep "Revises:"

# 출력:
# Revises: abc123
# Revises: abc123  # ← 중복 발견!

3. 팀 규칙 정하기​

# .github/PULL_REQUEST_TEMPLATE.md
"""
## 마이그레이션 체크리스트

- [ ] 최신 main을 rebase했나요?
- [ ] down_revision이 올바른가요?
- [ ] upgrade/downgrade 테스트했나요?
- [ ] 마이그레이션 메시지가 명확한가요?
"""

🚨 주의사항​

1. Git Rebase 시 주의​

# ❌ 위험: main에서 rebase하면 히스토리가 꼬임
git checkout main
git rebase feature/something  # 하지 마세요!

# ✅ 올바름: feature 브랜치에서 main을 rebase
git checkout feature/something
git rebase main

2. down_revision 수동 수정 필수​

# Rebase 후 반드시 확인!
revision = 'ghi789'
down_revision = 'def456'  # ← 수동으로 업데이트 필요!

3. 여러 사람이 같은 테이블 수정​

# 민호: products 테이블에 discount_rate 추가
# 수진: products 테이블에 is_featured 추가

# ⚠️ 둘 다 add_column('products', ...)
# → 병합 후 확인 필수!

📊 CI/CD 자동화​

GitHub Actions 예제​

# .github/workflows/alembic-check.yml
name: Alembic Migration Check

on: [pull_request]

jobs:
  check-migrations:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v2

      - name: Check for multiple heads
        run: |
          alembic heads | wc -l
          if [ $(alembic heads | wc -l) -gt 1 ]; then
            echo "Multiple heads detected!"
            exit 1
          fi

      - name: Test upgrade
        run: alembic upgrade head

      - name: Test downgrade
        run: alembic downgrade -1

✅ 실습 과제​

두 개의 마이그레이션을 만들어서 충돌을 해결해보세요:

  1. feature-a 브랜치: customers 테이블 추가
  2. feature-b 브랜치: suppliers 테이블 추가
  3. 두 브랜치 병합하기
정답 보기
# 1. 브랜치 A
git checkout -b feature-a
alembic revision -m "add customers table"
# → revises: base
git commit -am "Add customers"

# 2. 브랜치 B (같은 시작점)
git checkout main
git checkout -b feature-b
alembic revision -m "add suppliers table"
# → revises: base (충돌!)
git commit -am "Add suppliers"

# 3. A 먼저 병합
git checkout main
git merge feature-a

# 4. B 리베이스
git checkout feature-b
git rebase main
# → suppliers 마이그레이션의 revises를 customers로 수정!

# 5. B 병합
git checkout main
git merge feature-b

🎓 배운 내용​

충돌 해결 전략​

  1. 순차 병합: 한 명씩 순서대로
  2. Rebase: down_revision 수정
  3. Merge: alembic merge 명령어

예방 방법​

  • 작업 전 main 업데이트
  • 자주 통합
  • PR에서 마이그레이션 체크
