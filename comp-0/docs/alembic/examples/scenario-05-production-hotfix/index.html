시나리오 5: 서연의 긴급 출동

프로덕션에서 발견된 버그를 긴급하게 수정하는 서연의 이야기입니다.

📌 상황 설명​

금요일 오후 5시. 퇴근 준비를 하던 백엔드 개발자 서연에게 긴급 전화가 옵니다. 프로덕션에서 주문이 실패하고 있다는 것! 원인을 파악해보니 어제 배포한 마이그레이션에 문제가 있었습니다.

문제: orders 테이블에 NOT NULL 제약조건을 잘못 추가해서 기존 주문 업데이트가 실패합니다.

🎯 목표​

  • 프로덕션 에러 신속 파악
  • 안전한 핫픽스 배포
  • 최소 다운타임 달성
  • 데이터 손실 방지

💬 대화​

📞 CTO: "서연님! 주문이 안 들어와요! 고객들이 불만을 쏟아내고 있어요!"

서연: "확인해볼게요! (땀)"

서연: (로그 확인) "아... 어제 배포한 마이그레이션이 문제네요. shipping_address를 NOT NULL로 만들었는데, 픽업 주문에는 배송지가 없어서..."

CTO: "얼마나 걸릴까요?"

서연: "10분 안에 고치겠습니다!"

⚡ 긴급 상황 대응​

1단계: 문제 파악 (2분)​

# 프로덕션 서버 접속
ssh production-server

# 에러 로그 확인
tail -f /var/log/app/error.log
ERROR: null value in column "shipping_address" violates not-null constraint
DETAIL: Failing row contains (12345, 100, pickup, null, ...)
# 데이터베이스 스키마 확인
psql -d shop_production

\d orders
#  Column          |  Type   | Nullable
# -----------------+---------+----------
#  shipping_address| varchar |not null ← 문제!

서연: "맞다! 픽업 주문은 배송지가 필요 없는데 NOT NULL로 만들었네..."

2단계: 원인 분석​

# 문제의 마이그레이션 찾기
alembic current
# abc123 (head)

# 히스토리 확인
alembic history -v
# versions/abc123_add_shipping_address.py (문제의 마이그레이션)
"""add shipping address to orders

Revision ID: abc123
Revises: previous
Create Date: 2024-01-24 18:00:00
"""

def upgrade():
    # ❌ 문제: 기본값 없이 NOT NULL
    op.add_column('orders',
        sa.Column('shipping_address', sa.String(500), nullable=False)
    )

def downgrade():
    op.drop_column('orders', 'shipping_address')

3단계: 긴급 수정 방법 결정 (1분)​

옵션 1: 롤백 (빠르지만 위험)

alembic downgrade -1  # shipping_address 컬럼 삭제
# ⚠️ 어제 이후 입력된 배송지 데이터 손실!

옵션 2: 핫픽스 (안전)

# 새 마이그레이션으로 수정
# ✅ 데이터 손실 없음

서연: "옵션 2로 가겠습니다!"

4단계: 핫픽스 마이그레이션 생성 (2분)​

# 로컬 환경
git checkout main
git pull

# 현재 프로덕션과 동기화
alembic upgrade abc123

# 핫픽스 브랜치
git checkout -b hotfix/shipping-address-nullable

# 마이그레이션 생성
alembic revision -m "fix: make shipping_address nullable"
"""fix: make shipping_address nullable

Revision ID: def456
Revises: abc123
Create Date: 2024-01-25 17:10:00
"""

def upgrade():
    # shipping_address를 NULL 허용으로 변경
    op.alter_column('orders', 'shipping_address',
        nullable=True,
        existing_type=sa.String(500)
    )

    # 기본값 설정 (픽업 주문용)
    op.execute('''
        UPDATE orders
        SET shipping_address = 'PICKUP'
        WHERE shipping_address IS NULL
        AND delivery_method = 'pickup'
    ''')

def downgrade():
    # 픽업 주문의 주소를 NULL로
    op.execute('''
        UPDATE orders
        SET shipping_address = NULL
        WHERE shipping_address = 'PICKUP'
    ''')

    # 다시 NOT NULL로 (원래 잘못된 상태)
    op.alter_column('orders', 'shipping_address',
        nullable=False,
        existing_type=sa.String(500)
    )

5단계: 로컬 테스트 (2분)​

# 마이그레이션 테스트
alembic upgrade head

# 테스트 데이터로 검증
psql -d shop_dev
-- 픽업 주문 생성 테스트
INSERT INTO orders (user_id, delivery_method, shipping_address)
VALUES (1, 'pickup', NULL);
-- ✅ 성공!

INSERT INTO orders (user_id, delivery_method, shipping_address)
VALUES (2, 'delivery', '서울시 강남구...');
-- ✅ 성공!

서연: "테스트 통과! 배포합니다!"

6단계: 프로덕션 배포 (3분)​

# Git 커밋
git add alembic/versions/def456_fix_make_shipping_address_nullable.py
git commit -m "hotfix: make shipping_address nullable for pickup orders

Issue: Orders failing because shipping_address was NOT NULL
Fix: Allow NULL for pickup orders, set default 'PICKUP' value

Tested:
- Pickup orders can be created without address
- Delivery orders still require address
- Existing data preserved"

# Push
git push origin hotfix/shipping-address-nullable

# 긴급이므로 PR 없이 main에 병합
git checkout main
git merge hotfix/shipping-address-nullable
git push origin main
# 프로덕션 배포
ssh production-server

cd /var/www/shop
git pull origin main

# 마이그레이션 실행
alembic upgrade head
# INFO  Running upgrade abc123 -> def456, fix: make shipping_address nullable

# 애플리케이션 재시작
sudo systemctl restart shop-api

7단계: 검증 및 모니터링 (2분)​

# 에러 로그 확인
tail -f /var/log/app/error.log
# (더 이상 에러 없음!)

# 주문 생성 테스트
curl -X POST https://api.shop.com/orders \
  -H "Content-Type: application/json" \
  -d '{
    "user_id": 100,
    "delivery_method": "pickup",
    "items": [...]
  }'
# ✅ 200 OK
-- 데이터 확인
psql -d shop_production

SELECT delivery_method, shipping_address, COUNT(*)
FROM orders
WHERE created_at > NOW() - INTERVAL '1 hour'
GROUP BY delivery_method, shipping_address;

#  delivery_method | shipping_address | count
# -----------------+------------------+-------
#  pickup          | PICKUP           |    15
#  delivery        | 서울시...        |    42

📋 사후 분석 (Post-Mortem)​

근본 원인​

# ❌ 원래 마이그레이션 (문제)
def upgrade():
    op.add_column('orders',
        sa.Column('shipping_address', sa.String(500), nullable=False)
        # 기존 행 + 픽업 주문 = 문제!
    )

올바른 방법​

# ✅ 올바른 마이그레이션
def upgrade():
    # 1. NULL 허용으로 추가
    op.add_column('orders',
        sa.Column('shipping_address', sa.String(500), nullable=True)
    )

    # 2. 기존 데이터 처리
    op.execute('''
        UPDATE orders
        SET shipping_address = 'PICKUP'
        WHERE delivery_method = 'pickup'
    ''')

    op.execute('''
        UPDATE orders
        SET shipping_address = '주소 미등록'
        WHERE shipping_address IS NULL
        AND delivery_method = 'delivery'
    ''')

    # 3. 이제 NOT NULL 제약조건 추가 (선택)
    # op.alter_column('orders', 'shipping_address', nullable=False)

🎨 다양한 핫픽스 시나리오​

시나리오 A: 잘못된 제약조건​

# 문제: 가격이 0 이하 불가
CheckConstraint('price > 0')  # 무료 상품 등록 불가!

# 핫픽스
def upgrade():
    op.drop_constraint('check_positive_price', 'products')
    op.create_check_constraint('check_non_negative_price', 'products',
        'price >= 0'  # 0 허용
    )

시나리오 B: 인덱스 누락으로 성능 저하​

# 문제: 사용자별 주문 조회가 매우 느림
# SELECT * FROM orders WHERE user_id = ?  -- Seq Scan!

# 핫픽스
def upgrade():
    op.create_index('idx_orders_user_id', 'orders',
        ['user_id'],
        postgresql_concurrently=True  # 락 없이 생성
    )

시나리오 C: 외래 키 캐스케이드 문제​

# 문제: 사용자 삭제 시 주문까지 삭제됨
ForeignKey('users.id', ondelete='CASCADE')  # 위험!

# 핫픽스
def upgrade():
    # 1. 기존 외래 키 삭제
    op.drop_constraint('fk_orders_user', 'orders', type_='foreignkey')

    # 2. SET NULL로 재생성
    op.create_foreign_key(
        'fk_orders_user',
        'orders', 'users',
        ['user_id'], ['id'],
        ondelete='SET NULL'  # 주문은 유지, user_id만 NULL
    )

    # 3. user_id를 nullable로 변경
    op.alter_column('orders', 'user_id', nullable=True)

✅ 실습 과제​

다음 프로덕션 이슈를 핫픽스하세요:

문제: users 테이블의 email을 유니크 제약조건으로 만들었는데, 중복 이메일이 이미 존재해서 에러 발생

정답 보기
"""hotfix: handle duplicate emails before unique constraint

Revision ID: hotfix001
"""

def upgrade():
    # 1. 중복 이메일 찾기 및 처리
    op.execute('''
        UPDATE users u
        SET email = email || '_' || id
        WHERE id IN (
            SELECT id FROM (
                SELECT id, email,
                ROW_NUMBER() OVER (PARTITION BY email ORDER BY id) as rn
                FROM users
            ) t
            WHERE rn > 1
        )
    ''')

    # 2. 이제 유니크 제약조건 추가
    op.create_unique_constraint('uq_users_email', 'users', ['email'])

def downgrade():
    op.drop_constraint('uq_users_email', 'users', type_='unique')
