요구사항 분석 (Requirements Analysis)

소프트웨어 개발 프로젝트의 성공을 위한 첫 번째 단계인 요구사항 분석에 대해 학습합니다.

1. 요구사항 분석이란?​

1.1 정의​

**요구사항 분석(Requirements Analysis)**은 소프트웨어 시스템이 무엇을 해야 하는지, 어떤 특성을 가져야 하는지를 식별하고 문서화하는 프로세스입니다.

1.2 중요성​

    A[요구사항 분석] --> B[올바른 제품 개발]
    A --> C[비용 절감]
    A --> D[리스크 감소]
    A --> E[이해관계자 만족]

요구사항 분석의 가치
  • 조기 결함 발견: 개발 초기 단계에서 문제를 발견하면 수정 비용이 10배~100배 절감됩니다
  • 명확한 방향성: 프로젝트 목표와 범위를 명확히 정의합니다
  • 효율적 의사소통: 이해관계자 간 공통 이해를 형성합니다

2. 요구사항 유형​

2.1 기능 요구사항 (Functional Requirements)​

시스템이 무엇을 해야 하는지를 정의합니다.

예시:

FR-001: 사용자는 이메일과 비밀번호로 로그인할 수 있어야 한다
FR-002: 시스템은 상품 검색 기능을 제공해야 한다
FR-003: 관리자는 사용자 권한을 변경할 수 있어야 한다
FR-004: 결제 완료 시 사용자에게 확인 이메일을 발송해야 한다

2.2 비기능 요구사항 (Non-Functional Requirements)​

시스템이 어떻게 동작해야 하는지를 정의합니다.

카테고리설명예시
성능 (Performance)응답 시간, 처리량API 응답 시간 < 200ms
확장성 (Scalability)시스템 성장 능력동시 사용자 10,000명 지원
보안 (Security)데이터 보호, 인증HTTPS, OAuth 2.0 사용
가용성 (Availability)시스템 가동 시간99.9% uptime
유지보수성 (Maintainability)수정 용이성모듈화된 아키텍처
사용성 (Usability)사용자 경험3번의 클릭 내 주요 기능 접근

비기능 요구사항 예시:

NFR-001: 메인 페이지 로딩 시간은 3초 이내여야 한다
NFR-002: 시스템은 연중 99.9% 가용성을 유지해야 한다
NFR-003: 모든 개인정보는 AES-256으로 암호화되어야 한다
NFR-004: 모바일 디바이스에서 반응형으로 동작해야 한다
NFR-005: 장애 발생 시 1분 이내 자동 복구되어야 한다
흔한 실수

비기능 요구사항을 간과하면 프로젝트 후반부에 큰 문제가 발생할 수 있습니다:

  • 성능 문제로 인한 사용자 이탈
  • 보안 취약점으로 인한 데이터 유출
  • 확장성 부족으로 인한 아키텍처 재설계

3. 요구사항 수집 방법​

3.1 인터뷰 (Interviews)​

일대일 또는 그룹 인터뷰를 통한 요구사항 수집

## 인터뷰 준비 체크리스트

### 사전 준비
- [ ] 인터뷰 대상자 선정 (이해관계자, 사용자, 도메인 전문가)
- [ ] 질문 리스트 작성
- [ ] 배경 조사 (비즈니스 도메인, 기존 시스템)
- [ ] 인터뷰 시간 및 장소 예약

### 질문 예시
1. 현재 업무 프로세스에서 가장 큰 문제점은 무엇인가요?
2. 새로운 시스템에서 가장 중요한 기능은 무엇인가요?
3. 이상적인 시스템은 어떤 모습인가요?
4. 기존 시스템의 어떤 부분을 개선하고 싶으신가요?
5. 성공의 기준은 무엇인가요?
효과적인 인터뷰 기법
  • 개방형 질문 사용: "어떻게", "왜", "무엇을"로 시작하는 질문
  • 적극적 경청: 말하는 것보다 듣는 데 집중
  • 5 Why 기법: "왜?"를 5번 반복하여 근본 원인 파악
  • 녹음 및 기록: 중요한 내용 누락 방지

4. 사용자 스토리 (User Stories) 작성법​

4.1 사용자 스토리란?​

사용자 관점에서 기능을 설명하는 간결한 문장

As a [역할],
I want [기능],
So that [목적/가치].

4.2 사용자 스토리 작성 예시​

## US-001: 회원 가입
**As a** 신규 방문자
**I want** 이메일로 회원가입을 할 수 있기를
**So that** 쇼핑몰의 회원 전용 혜택을 받을 수 있다

### 인수 기준 (Acceptance Criteria)
- [ ] 이메일 형식 검증 (`@` 포함, 유효한 도메인)
- [ ] 비밀번호는 최소 8자, 영문/숫자/특수문자 포함
- [ ] 중복 이메일 체크
- [ ] 회원가입 완료 시 인증 이메일 발송
- [ ] 개인정보 처리방침 동의 필수

### 스토리 포인트
5 포인트

---

## US-002: 상품 검색
**As a** 고객
**I want** 키워드로 상품을 검색할 수 있기를
**So that** 원하는 상품을 빠르게 찾을 수 있다

### 인수 기준
- [ ] 검색어 자동완성 제공
- [ ] 검색 결과는 관련도 순으로 정렬
- [ ] 검색 결과가 없을 경우 추천 상품 표시
- [ ] 검색 히스토리 저장 (최근 5개)
- [ ] 카테고리, 가격, 브랜드 필터 제공

### 스토리 포인트
8 포인트

## US-003: 장바구니 담기
**As a** 고객
**I want** 상품을 장바구니에 담을 수 있기를
**So that** 나중에 구매할 수 있다

### 인수 기준
- [ ] 비회원도 장바구니 사용 가능 (쿠키 저장)
- [ ] 수량 변경 가능
- [ ] 장바구니 아이콘에 상품 개수 표시
- [ ] 재고 부족 시 경고 메시지
- [ ] 장바구니 내역은 7일간 유지

### 스토리 포인트
3 포인트

4.3 INVEST 원칙​

좋은 사용자 스토리는 INVEST 원칙을 따릅니다:

원칙설명예시
Independent독립적다른 스토리와 의존성 최소화
Negotiable협상 가능구현 방법은 유연하게
Valuable가치 있는사용자에게 명확한 가치 제공
Estimable추정 가능개발 시간 예측 가능
Small작은 단위1-2주 내 완료 가능
Testable테스트 가능명확한 인수 기준
나쁜 사용자 스토리 예시
❌ "시스템 성능 개선"
→ 너무 모호하고 측정 불가능

❌ "데이터베이스를 MySQL로 구축"
→ 사용자 가치가 아닌 기술적 구현

❌ "관리자 기능 개발"
→ 너무 크고 불명확

✅ "As an admin, I want to export user data as CSV,
    so that I can analyze it in Excel"
→ 명확하고 측정 가능하며 가치 있음

5.3 Use Case 명세서​

# Use Case 명세서: 상품 주문하기

## 기본 정보
- **Use Case ID**: UC-005
- **Use Case 명**: 상품 주문하기
- **액터**: 고객 (회원)
- **사전 조건**:
  - 사용자가 로그인된 상태
  - 장바구니에 1개 이상의 상품이 있음
- **사후 조건**:
  - 주문이 생성됨
  - 재고가 차감됨
  - 결제 대기 상태로 전환

## 정상 흐름
1. 고객이 장바구니 페이지에서 "주문하기" 버튼을 클릭한다
2. 시스템은 배송지 정보 입력 페이지를 표시한다
3. 고객이 배송지 정보를 입력한다
4. 시스템은 주문 확인 페이지를 표시한다
5. 고객이 주문 내용을 확인하고 "결제하기" 버튼을 클릭한다
6. 시스템은 주문을 생성하고 결제 페이지로 이동한다

## 대안 흐름
**3a. 기본 배송지가 저장되어 있는 경우**
  - 3a1. 시스템은 저장된 배송지를 자동으로 채운다
  - 3a2. 고객이 배송지를 변경하거나 그대로 사용할 수 있다

**5a. 재고 부족**
  - 5a1. 시스템은 재고 부족 메시지를 표시한다
  - 5a2. 해당 상품을 장바구니에서 제거하거나 수량을 조정한다
  - 5a3. 정상 흐름 4번으로 복귀

## 예외 흐름
**Ex1. 네트워크 오류**
  - Ex1a. 시스템은 오류 메시지를 표시한다
  - Ex1b. 주문 데이터를 로컬에 임시 저장한다

**Ex2. 세션 만료**
  - Ex2a. 시스템은 로그인 페이지로 리다이렉트한다
  - Ex2b. 로그인 후 원래 페이지로 복귀

## 비기능 요구사항
- 주문 생성은 3초 이내에 완료되어야 함
- 동시에 같은 상품을 주문할 경우 재고 충돌 방지

6.1 BRD (Business Requirements Document)​

# 비즈니스 요구사항 문서 (BRD)

## 1. 개요
**프로젝트명**: AI 기반 상품 추천 시스템 도입
**작성일**: 2025-01-15
**작성자**: 김비즈니스 (Product Manager)

## 2. 비즈니스 목표
- 고객 구매 전환율 15% 향상
- 평균 주문 금액 20% 증가
- 고객 재방문율 30% 개선

## 3. 현재 상황 및 문제점
- 고객이 원하는 상품을 찾기 어려워함
- 이탈률이 업계 평균보다 25% 높음
- 크로스셀링/업셀링 기회 상실

## 4. 제안 솔루션
AI 기반 개인화 추천 엔진을 도입하여:
- 고객 구매 이력 기반 맞춤 추천
- 실시간 행동 분석
- 유사 고객 분석

## 5. 성공 지표 (KPI)
| 지표 | 현재 | 목표 | 측정 방법 |
|------|------|------|----------|
| 전환율 | 2.5% | 2.9% | Google Analytics |
| 평균 주문 금액 | 45,000원 | 54,000원 | 주문 데이터 |
| 재방문율 | 30% | 39% | 쿠키 추적 |

## 6. 투자 대비 효과 (ROI)
- 예상 투자: 1억 원
- 예상 연간 수익 증대: 3억 원
- ROI: 200%
- 회수 기간: 4개월

## 7. 위험 요소
- 개인정보 보호 규정 준수 필요

7.2 RICE 스코어링​

Reach × Impact × Confidence ÷ Effort

## RICE 우선순위 계산

| 기능 | Reach | Impact | Confidence | Effort | RICE Score | 순위 |
|------|-------|--------|------------|--------|------------|------|
| 간편 결제 | 5000 | 3 | 100% | 2 | 7500 | 1 |
| 상품 추천 | 8000 | 2 | 80% | 8 | 1600 | 2 |
| 리뷰 시스템 | 3000 | 3 | 90% | 4 | 2025 | 3 |
| 위시리스트 | 2000 | 2 | 100% | 1 | 4000 | 4 |
| 포인트 | 4000 | 1 | 70% | 5 | 560 | 5 |

### 계산 기준
**Reach**: 분기당 영향받는 사용자 수
**Impact**:
  - 3 = Massive (극대)
  - 2 = High (높음)
  - 1 = Medium (중간)
  - 0.5 = Low (낮음)
**Confidence**:
  - 100% = High
  - 80% = Medium
  - 50% = Low
**Effort**: 개발자-월 (Person-Months)

### 우선순위 결정
RICE Score 기준 상위 3개 기능을 우선 개발
RICE vs MoSCoW
  • MoSCoW: 정성적, 빠른 의사결정, 이해관계자 합의
  • RICE: 정량적, 데이터 기반, 객관적 비교
  • 추천: 두 방법을 함께 사용하여 검증

  - 인증된 사용자: `100 req/min`

- API Key 기반 파트너 인증

- 침입 탐지 시스템 (IDS)
