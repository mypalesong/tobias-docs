구현 (Implementation)

소프트웨어 개발의 구현 단계에서 품질과 효율성을 보장하기 위한 모범 사례를 다룹니다.

1. 코딩 표준 (Coding Standards)​

1.1 명명 규칙 (Naming Conventions)​

일관된 명명 규칙은 코드의 가독성과 유지보수성을 크게 향상시킵니다.

// 좋은 예: 명확하고 설명적인 이름
const getUserProfile = async (userId) => {
  const user = await database.users.findById(userId);
  return user;
};

// 나쁜 예: 모호하고 축약된 이름
const getUsrPrf = async (uid) => {
  const u = await db.usr.fbi(uid);
  return u;
};

명명 규칙 가이드라인:

  • 변수/함수: camelCase 사용
  • 클래스/컴포넌트: PascalCase 사용
  • 상수: UPPER_SNAKE_CASE 사용
  • private 필드: _leadingUnderscore 사용

1.2 코드 포맷팅 (Code Formatting)​

자동화된 포맷팅 도구를 사용하여 일관성을 유지합니다.

// .prettierrc
{
  "semi": true,
  "trailingComma": "es5",
  "singleQuote": true,
  "printWidth": 80,
  "tabWidth": 2
}
팁

Prettier + ESLint 조합을 사용하여 포맷팅과 코드 품질을 동시에 관리하세요. pre-commit hook을 설정하면 일관성을 자동으로 보장할 수 있습니다.

1.3 주석 작성 (Writing Comments)​

코드는 자명해야 하지만, 복잡한 로직에는 명확한 주석이 필요합니다.

# 좋은 예: 왜(Why)를 설명하는 주석
def calculate_discount(price, customer_tier):
    # 프리미엄 고객에게는 20% 할인을 적용합니다.
    # 이는 2024년 Q1 마케팅 전략의 일부입니다.
    if customer_tier == "premium":
        return price * 0.8
    return price

# 나쁜 예: 무엇(What)을 반복하는 주석
def calculate_discount(price, customer_tier):
    # customer_tier가 premium이면
    if customer_tier == "premium":
        # price에 0.8을 곱합니다
        return price * 0.8
    # price를 반환합니다
    return price

주석 작성 원칙:

  1. Why, not What: 코드가 무엇을 하는지가 아닌, 왜 그렇게 하는지 설명
  2. TODO/FIXME 태그: 개선이 필요한 부분 명시
  3. API 문서: 공개 함수/클래스에는 JSDoc, docstring 등 사용

1.4 코드 구조화 (Code Organization)​

src/
├── components/          # UI 컴포넌트
│   ├── common/         # 재사용 가능한 공통 컴포넌트
│   └── features/       # 기능별 컴포넌트
├── hooks/              # Custom React hooks
├── utils/              # 유틸리티 함수
├── services/           # API 호출 및 비즈니스 로직
├── constants/          # 상수 정의
└── types/              # TypeScript 타입 정의
경고

하나의 파일에 여러 책임을 담지 마세요. **단일 책임 원칙(SRP)**을 따라 각 모듈이 하나의 명확한 목적을 가지도록 합니다.

2. Git 워크플로우 (Git Workflow)​

2.1 브랜치 전략 (Branching Strategy)​

Git Flow 또는 GitHub Flow를 팀 상황에 맞게 선택합니다.

# GitHub Flow (간단한 프로젝트)
main (production-ready)
  └── feature/user-authentication
  └── fix/login-validation

# Git Flow (복잡한 프로젝트)
main (production)
  └── develop (integration)
      ├── feature/payment-integration
      ├── feature/user-dashboard
      └── release/v1.2.0

브랜치 명명 규칙:

  • feature/기능명: 새로운 기능 개발
  • fix/버그명: 버그 수정
  • hotfix/긴급수정명: 프로덕션 긴급 수정
  • refactor/리팩토링명: 코드 리팩토링
  • docs/문서명: 문서 작업

2.2 커밋 메시지 (Commit Messages)​

명확하고 일관된 커밋 메시지는 프로젝트 이력을 이해하는 데 필수적입니다.

# 좋은 예: Conventional Commits 형식
feat: add user authentication with JWT
fix: resolve memory leak in image processor
docs: update API documentation for v2.0
refactor: simplify error handling logic
test: add unit tests for payment service

# 나쁜 예
update
fix bug
changes
wip

Conventional Commits 형식:

<type>[optional scope]: <description>

[optional body]

[optional footer(s)]

주요 타입:

  • feat: 새로운 기능
  • fix: 버그 수정
  • docs: 문서 변경
  • style: 코드 포맷팅 (기능 변경 없음)
  • refactor: 리팩토링
  • test: 테스트 추가/수정
  • chore: 빌드/설정 변경

2.3 Pull Request 작성​

효과적인 PR은 리뷰 시간을 단축하고 품질을 향상시킵니다.

## 변경 사항 요약
사용자 인증 시스템에 JWT 기반 토큰 인증을 추가했습니다.

## 변경 이유
기존 세션 기반 인증은 마이크로서비스 환경에서 확장성 문제가 있었습니다.

## 주요 변경 내용
- JWT 토큰 생성 및 검증 미들웨어 추가
- 사용자 로그인/로그아웃 API 엔드포인트 구현
- 토큰 갱신 로직 구현

## 테스트 방법
1. `npm install && npm run dev`
2. POST `/api/auth/login`으로 로그인
3. 반환된 토큰으로 보호된 엔드포인트 접근 확인

## 체크리스트
- [x] 단위 테스트 통과
- [x] 통합 테스트 통과
- [x] 문서 업데이트
- [x] Breaking changes 없음
팁

PR은 300줄 이하로 유지하는 것이 이상적입니다. 큰 변경사항은 여러 개의 작은 PR로 분할하세요.

2.4 Git 모범 사례​

# 자주 커밋하기 (작은 단위로)
git add src/auth/login.js
git commit -m "feat: implement login validation"

# 푸시 전에 rebase로 히스토리 정리
git fetch origin
git rebase origin/main

# 민감한 정보는 절대 커밋하지 않기
echo ".env" >> .gitignore
echo "config/secrets.yml" >> .gitignore

# 의미 있는 변경 단위로 staging
git add -p  # 대화형으로 변경사항 선택
경고

절대 git push --force를 공유 브랜치에 사용하지 마세요. 다른 개발자의 작업을 덮어쓸 수 있습니다. 필요한 경우 git push --force-with-lease를 사용하세요.

3. 코드 리뷰 (Code Review)​

3.1 리뷰어 가이드​

효과적인 코드 리뷰는 건설적이고 구체적이어야 합니다.

체크리스트:

  • 기능성: 코드가 의도한 대로 작동하는가?
  • 가독성: 코드가 이해하기 쉬운가?
  • 테스트: 적절한 테스트가 포함되어 있는가?
  • 성능: 성능 이슈가 있는가?
  • 보안: 보안 취약점이 있는가?
  • 일관성: 프로젝트 스타일 가이드를 따르는가?

좋은 리뷰 코멘트 예시:

# 좋은 예: 구체적이고 건설적
❌ "이 부분을 개선해주세요."
✅ "이 반복문은 O(n²) 복잡도를 가집니다. `Map`을 사용하면 O(n)으로 개선할 수 있습니다."

❌ "이름이 이상해요."
✅ "`processData`는 모호합니다. `calculateMonthlyRevenue`처럼 구체적인 이름은 어떨까요?"

❌ "버그가 있어요."
✅ "`userId`가 null일 때 에러가 발생합니다. null 체크를 추가해주세요."

3.2 작성자 가이드​

코드 리뷰를 받는 것도 중요한 기술입니다.

모범 사례:

  1. 자체 리뷰 먼저: PR 생성 전 본인이 먼저 변경사항 검토
  2. 컨텍스트 제공: PR 설명에 충분한 배경 정보 포함
  3. 작게 유지: 한 번에 하나의 문제만 해결
  4. 열린 마음: 피드백을 학습 기회로 활용
// 리뷰 코멘트에 대한 응답 예시
// Reviewer: "에러 처리가 누락되었습니다."

// 수정 전
const data = await fetchUserData(userId);
return data;

// 수정 후
try {
  const data = await fetchUserData(userId);
  return data;
} catch (error) {
  logger.error(`Failed to fetch user data: ${error.message}`);
  throw new UserDataError(`Unable to retrieve user ${userId}`);
}

3.3 리뷰 우선순위​

모든 코멘트가 동일한 중요도를 가지는 것은 아닙니다.

  • 🔴 Critical: 반드시 수정 필요 (보안, 버그)
  • 🟡 Major: 수정 권장 (성능, 가독성)
  • 🟢 Minor: 선택적 개선 (스타일, 제안)
  • 💭 Question: 질문이나 토론
🔴 **Critical**: SQL injection 취약점이 있습니다.
Prepared statement를 사용해주세요.

🟡 **Major**: 이 함수는 너무 길어서 이해하기 어렵습니다.
여러 함수로 분리하는 것이 좋겠습니다.

🟢 **Minor**: 변수명을 `camelCase`로 통일하면 더 일관성 있을 것 같습니다.

💭 **Question**: 이 로직을 선택한 특별한 이유가 있나요?
팁

리뷰 시간 제한: 코드 리뷰는 24시간 이내에 완료하는 것을 목표로 하세요. 빠른 피드백은 개발 속도를 높입니다.

4. 페어 프로그래밍 (Pair Programming)​

4.1 역할 분담​

Driver (운전자)

  • 키보드를 잡고 코드 작성
  • 세부 구현에 집중
  • Navigator의 지시를 따름

Navigator (네비게이터)

  • 전체적인 방향 제시
  • 잠재적 이슈 발견
  • 대안적 접근 방식 제안
# 페어 프로그래밍 예시 시나리오
# Navigator: "먼저 입력 유효성 검사부터 시작합시다."
# Driver: 코드 작성

def process_payment(amount, card_number):
    # Navigator: "금액이 음수인 경우를 체크해야 할 것 같아요."
    if amount <= 0:
        raise ValueError("Amount must be positive")

    # Navigator: "카드 번호 형식도 검증해봅시다."
    if not is_valid_card_number(card_number):
        raise ValueError("Invalid card number")

    # Driver: "결제 처리 로직을 추가할게요."
    return payment_gateway.charge(amount, card_number)

4.2 페어 프로그래밍 패턴​

Ping-Pong 패턴 (TDD와 함께)

  1. Developer A: 실패하는 테스트 작성
  2. Developer B: 테스트를 통과하는 코드 작성
  3. Developer B: 다음 실패하는 테스트 작성
  4. Developer A: 테스트를 통과하는 코드 작성
  5. 반복...
// Developer A: 테스트 작성
test('should calculate total price with tax', () => {
  const result = calculateTotal(100, 0.1);
  expect(result).toBe(110);
});

// Developer B: 구현
function calculateTotal(price, taxRate) {
  return price * (1 + taxRate);
}

// Developer B: 다음 테스트 작성
test('should throw error for negative price', () => {
  expect(() => calculateTotal(-100, 0.1)).toThrow();
});

// Developer A: 구현
function calculateTotal(price, taxRate) {
  if (price < 0) throw new Error('Price cannot be negative');
  return price * (1 + taxRate);
}

5. 리팩토링 (Refactoring)​

5.1 리팩토링 원칙​

리팩토링은 외부 동작을 변경하지 않으면서 내부 구조를 개선하는 것입니다.

언제 리팩토링하나요?

  1. 중복 코드 발견 시
  2. 함수가 너무 길어질 때 (20줄 이상)
  3. 클래스가 너무 많은 책임을 가질 때
  4. 이해하기 어려운 코드를 만날 때

5.2 일반적인 리팩토링 패턴​

함수 추출 (Extract Function)

// Before: 긴 함수
function printInvoice(invoice) {
  console.log(`Invoice for ${invoice.customer}`);
  console.log(`Amount: ${invoice.amount}`);

  // 세금 계산
  let tax = 0;
  if (invoice.amount > 1000) {
    tax = invoice.amount * 0.1;
  } else {
    tax = invoice.amount * 0.05;
  }

  console.log(`Tax: ${tax}`);
  console.log(`Total: ${invoice.amount + tax}`);
}

// After: 함수 추출
function printInvoice(invoice) {
  printHeader(invoice);
  const tax = calculateTax(invoice.amount);
  printTotal(invoice.amount, tax);
}

function printHeader(invoice) {
  console.log(`Invoice for ${invoice.customer}`);
  console.log(`Amount: ${invoice.amount}`);
}

function calculateTax(amount) {
  return amount > 1000 ? amount * 0.1 : amount * 0.05;
}

function printTotal(amount, tax) {
  console.log(`Tax: ${tax}`);
  console.log(`Total: ${amount + tax}`);
}

조건문 단순화 (Simplify Conditional)

# Before: 복잡한 조건문
def get_discount(customer):
    if customer.years > 5 and customer.total_purchases > 10000:
        if customer.is_premium:
            return 0.15
        else:
            return 0.10
    elif customer.years > 2:
        return 0.05
    else:
        return 0

# After: 조건문 단순화
def get_discount(customer):
    if is_loyal_premium_customer(customer):
        return 0.15
    if is_loyal_customer(customer):
        return 0.10
    if is_regular_customer(customer):
        return 0.05
    return 0

def is_loyal_premium_customer(customer):
    return customer.years > 5 and customer.total_purchases > 10000 and customer.is_premium

def is_loyal_customer(customer):
    return customer.years > 5 and customer.total_purchases > 10000

def is_regular_customer(customer):
    return customer.years > 2

매직 넘버 제거 (Remove Magic Numbers)

// Before: 매직 넘버
public class OrderProcessor {
    public double calculateShipping(double weight) {
        if (weight < 5) {
            return 10.0;
        } else if (weight < 20) {
            return 20.0;
        } else {
            return 30.0;
        }
    }
}

// After: 상수로 추출
public class OrderProcessor {
    private static final double LIGHT_WEIGHT_THRESHOLD = 5.0;
    private static final double MEDIUM_WEIGHT_THRESHOLD = 20.0;
    private static final double LIGHT_SHIPPING_COST = 10.0;
    private static final double MEDIUM_SHIPPING_COST = 20.0;
    private static final double HEAVY_SHIPPING_COST = 30.0;

    public double calculateShipping(double weight) {
        if (weight < LIGHT_WEIGHT_THRESHOLD) {
            return LIGHT_SHIPPING_COST;
        } else if (weight < MEDIUM_WEIGHT_THRESHOLD) {
            return MEDIUM_SHIPPING_COST;
        } else {
            return HEAVY_SHIPPING_COST;
        }
    }
}

리팩토링할 때는 테스트 없이 진행하지 마세요. 기존 테스트가 없다면, 리팩토링 전에 먼저 테스트를 작성하세요.

5.4 Boy Scout Rule​

"캠프장을 처음 왔을 때보다 더 깨끗하게 해놓고 떠나라."

// 코드를 수정할 때마다 조금씩 개선
// Before: 파일을 열었을 때
function calc(a, b, c) {
  return a * b + c;
}

// After: 작은 개선 추가
/**
 * 제품의 총 가격을 계산합니다.
 * @param {number} price - 단가
 * @param {number} quantity - 수량
 * @param {number} shippingCost - 배송비
 * @returns {number} 총 가격
 */
function calculateTotalPrice(price, quantity, shippingCost) {
  return price * quantity + shippingCost;
}

6. 기술 부채 관리 (Technical Debt Management)​

6.1 기술 부채 유형​

의도적 기술 부채

  • 빠른 출시를 위해 의식적으로 선택
  • 문서화하고 계획된 상환 필요

무의식적 기술 부채

  • 경험 부족, 설계 실수로 발생
  • 발견 즉시 해결 또는 기록
// 의도적 기술 부채 예시

const cache = new Map();

function getCachedData(key) {
  if (cache.has(key)) {
    return cache.get(key);
  }
  const data = expensiveOperation(key);
  cache.set(key, data);
  return data;
}

6.3 기술 부채 상환 전략​

20% 규칙: 스프린트 시간의 20%를 기술 부채 해결에 할당

# 기술 부채 우선순위 계산
def calculate_debt_priority(debt):
    """
    우선순위 = (영향도 × 발생 빈도) / 해결 비용
    """
    impact = debt.impact_score  # 1-10
    frequency = debt.occurrence_rate  # 1-10
    cost = debt.resolution_days  # 일수

    priority = (impact * frequency) / cost
    return priority

# 사용 예
debts = [
    {'name': 'Legacy auth', 'impact_score': 9, 'occurrence_rate': 8, 'resolution_days': 10},
    {'name': 'Duplicate code', 'impact_score': 5, 'occurrence_rate': 9, 'resolution_days': 3},
    {'name': 'Old dependencies', 'impact_score': 6, 'occurrence_rate': 4, 'resolution_days': 5},
]

sorted_debts = sorted(debts, key=calculate_debt_priority, reverse=True)

7.1 회고 (Retrospective)​

회고 형식 - Start, Stop, Continue:

## Sprint 12 회고

### 🟢 Start (시작할 것)
- 페어 프로그래밍을 복잡한 기능 개발 시 활용
- 주간 기술 공유 세션 시작
- 자동화된 성능 테스트 도입

### 🔴 Stop (중단할 것)
- 불필요하게 긴 회의
- 테스트 없이 배포하는 관행
- 코드 리뷰 없이 main 브랜치에 직접 푸시

### 🟡 Continue (계속할 것)
- 데일리 스탠드업 유지
- 코드 리뷰 문화
- 문서화 작업

7.2 메트릭 추적​

핵심 개발 메트릭:

  • 배포 빈도: 얼마나 자주 배포하는가?
  • 변경 리드 타임: 커밋부터 배포까지 걸리는 시간
  • 변경 실패율: 배포 중 실패 비율
  • 복구 시간: 장애 발생 시 복구까지 걸리는 시간
  • 코드 리뷰 시간: PR 생성부터 머지까지 시간
// GitHub API를 사용한 메트릭 수집 예시
async function calculateLeadTime(org, repo) {
  const prs = await github.pullRequests.list({
    owner: org,
    repo: repo,
    state: 'closed',
    per_page: 100
  });

  const leadTimes = prs.data
    .filter(pr => pr.merged_at)
    .map(pr => {
      const created = new Date(pr.created_at);
      const merged = new Date(pr.merged_at);
      return (merged - created) / (1000 * 60 * 60); // 시간 단위
    });

  const avgLeadTime = leadTimes.reduce((a, b) => a + b, 0) / leadTimes.length;
  return avgLeadTime.toFixed(2);
}
