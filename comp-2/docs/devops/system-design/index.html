시스템 설계 (System Design)

대규모 시스템을 설계하고 구축하기 위한 핵심 개념과 실전 가이드입니다.

1. 설계 원칙​

1.1 SOLID 원칙​

시스템 설계의 기본이 되는 5가지 객체지향 설계 원칙:

  • Single Responsibility: 단일 책임 원칙 - 하나의 컴포넌트는 하나의 책임만
  • Open/Closed: 개방-폐쇄 원칙 - 확장에는 열려있고 수정에는 닫혀있어야 함
  • Liskov Substitution: 리스코프 치환 원칙 - 하위 타입은 상위 타입을 대체 가능해야 함
  • Interface Segregation: 인터페이스 분리 원칙 - 특정 클라이언트를 위한 인터페이스 여러 개가 범용 인터페이스보다 낫다
  • Dependency Inversion: 의존관계 역전 원칙 - 구체화가 아닌 추상화에 의존해야 함

1.2 확장성 원칙​

Scale Up (수직 확장)
└── CPU, RAM, Disk 증설
└── 한계: 물리적 제약, 단일 장애점

Scale Out (수평 확장)
└── 서버 인스턴스 추가
└── 장점: 무한 확장 가능, 고가용성
팁

대부분의 현대 시스템은 Scale Out 방식을 선호합니다. AWS, GCP 등 클라우드 환경에서 Auto Scaling을 통해 자동으로 수평 확장이 가능합니다.

1.3 CAP 정리​

분산 시스템에서 세 가지 속성 중 두 가지만 동시에 만족 가능:

  • C (Consistency): 일관성 - 모든 노드가 동일한 데이터 조회
  • A (Availability): 가용성 - 모든 요청이 응답을 받음
  • P (Partition Tolerance): 분할 내성 - 네트워크 장애에도 시스템 동작
CP 시스템: MongoDB, HBase, Redis
└── 일관성 우선, 가용성 희생

AP 시스템: Cassandra, DynamoDB, CouchDB
└── 가용성 우선, 일관성 희생 (최종 일관성)

CA 시스템: RDBMS (단일 노드)
└── 분산 환경에서는 현실적으로 불가능

2. HLD vs LLD​

2.1 High-Level Design (상위 수준 설계)​

시스템의 전체 구조와 주요 컴포넌트 간 관계를 정의:

[Client] → [Load Balancer] → [Web Servers]
                                    ↓
                              [App Servers] ⇄ [Cache]
                                    ↓
                              [Database Cluster]
                                    ↓
                              [Storage (S3)]

포함 요소:

  • 시스템 아키텍처 다이어그램
  • 데이터 흐름 (Data Flow)
  • 기술 스택 선정
  • 확장성 전략
  • 트레이드오프 분석

2.2 Low-Level Design (하위 수준 설계)​

각 컴포넌트의 상세 구현을 정의:

# API 설계 예시
class UserService:
    def __init__(self, db_client, cache_client):
        self.db = db_client
        self.cache = cache_client

    def get_user(self, user_id: str) -> User:
        # Cache-Aside 패턴
        cached = self.cache.get(f"user:{user_id}")
        if cached:
            return User.from_json(cached)

        user = self.db.query("SELECT * FROM users WHERE id = ?", user_id)
        self.cache.set(f"user:{user_id}", user.to_json(), ttl=3600)
        return user

포함 요소:

  • 클래스 다이어그램
  • 데이터베이스 스키마
  • API 명세
  • 알고리즘 세부사항

3. 데이터베이스 설계​

3.1 SQL vs NoSQL 선택​

기준SQL (RDBMS)NoSQL
데이터 구조정형화된 스키마유연한 스키마
확장성수직 확장 (주로)수평 확장 (주로)
트랜잭션ACID 보장BASE (최종 일관성)
사용 사례금융, 회계, ERP로그, 세션, 캐시, SNS
예시PostgreSQL, MySQLMongoDB, Cassandra, Redis

3.2 정규화 vs 비정규화​

정규화 (Normalization)

중복을 제거하여 데이터 무결성 보장:

-- 3NF (Third Normal Form) 예시
CREATE TABLE users (
    id BIGINT PRIMARY KEY,
    email VARCHAR(255) UNIQUE,
    created_at TIMESTAMP
);

CREATE TABLE orders (
    id BIGINT PRIMARY KEY,
    user_id BIGINT REFERENCES users(id),
    total_amount DECIMAL(10,2),
    status VARCHAR(50)
);

비정규화 (Denormalization)

읽기 성능 최적화를 위해 의도적 중복:

-- 비정규화: JOIN 비용 제거
CREATE TABLE order_details (
    order_id BIGINT,
    user_email VARCHAR(255),  -- 중복 저장
    product_name VARCHAR(255), -- 중복 저장
    quantity INT,
    price DECIMAL(10,2)
);
경고

비정규화는 읽기 >> 쓰기 비율이 높을 때만 사용하세요. 데이터 일관성 유지가 복잡해집니다.

3.3 인덱싱 전략​

-- B-Tree 인덱스 (기본)
CREATE INDEX idx_users_email ON users(email);

-- 복합 인덱스 (Composite Index)
CREATE INDEX idx_orders_user_status ON orders(user_id, status);

-- 커버링 인덱스 (Covering Index)
CREATE INDEX idx_orders_covering ON orders(user_id, status) INCLUDE (total_amount);

인덱스 설계 원칙:

  • WHERE, JOIN, ORDER BY 절에 자주 사용되는 컬럼
  • Cardinality가 높은 컬럼 우선
  • 쓰기 성능 영향 고려 (인덱스 과다 금지)

3.4 샤딩 (Sharding)​

수평 분할을 통한 대용량 데이터 처리:

Hash-based Sharding:
user_id % shard_count → Shard #

Range-based Sharding:
user_id 1-1000000   → Shard 1
user_id 1000001-2000000 → Shard 2

Geographic Sharding:
region = 'us-west' → Shard A
region = 'eu-central' → Shard B
팁

샤딩 키는 변경이 거의 없고 균등 분산되는 컬럼을 선택하세요. user_id, tenant_id 등이 좋은 후보입니다.

4.3 Rate Limiting​

API 과부하 방지를 위한 속도 제한:

# Token Bucket 알고리즘
class RateLimiter:
    def __init__(self, capacity: int, refill_rate: float):
        self.capacity = capacity
        self.tokens = capacity
        self.refill_rate = refill_rate  # tokens/second
        self.last_refill = time.time()

    def allow_request(self) -> bool:
        self._refill()
        if self.tokens >= 1:
            self.tokens -= 1
            return True
        return False

    def _refill(self):
        now = time.time()
        elapsed = now - self.last_refill
        self.tokens = min(
            self.capacity,
            self.tokens + elapsed * self.refill_rate
        )
        self.last_refill = now

응답 헤더:

HTTP/1.1 200 OK
X-RateLimit-Limit: 1000
X-RateLimit-Remaining: 999
X-RateLimit-Reset: 1635360000

5. 확장성 설계​

5.1 로드 밸런싱​

트래픽을 여러 서버에 분산:

Layer 4 (Transport Layer):
└── TCP/UDP 기반 분산
└── 예시: AWS NLB, HAProxy

Layer 7 (Application Layer):
└── HTTP/HTTPS 헤더 기반 라우팅
└── 예시: AWS ALB, Nginx

알고리즘:
├── Round Robin: 순차 분배
├── Least Connections: 연결 수 최소 서버
├── IP Hash: 클라이언트 IP 기반
└── Weighted: 가중치 기반 분배

5.2 캐싱 전략​

캐싱 레이어:

Browser Cache (클라이언트)
    ↓
CDN (Content Delivery Network)
    ↓
Application Cache (Redis, Memcached)
    ↓
Database Query Cache
    ↓
Disk Cache

캐시 패턴:

# Cache-Aside (Lazy Loading)
def get_data(key):
    cached = cache.get(key)
    if cached:
        return cached
    data = db.query(key)
    cache.set(key, data, ttl=3600)
    return data

# Write-Through
def update_data(key, value):
    db.update(key, value)
    cache.set(key, value)  # 동시 업데이트

# Write-Behind (Write-Back)
def update_data(key, value):
    cache.set(key, value)  # 즉시 캐시 업데이트
    queue.enqueue(lambda: db.update(key, value))  # 비동기 DB 업데이트

캐시 무효화:

TTL (Time To Live):
└── cache.set(key, value, ttl=3600)  # 1시간 후 자동 만료

수동 무효화:
└── cache.delete(key)

Cache Stampede 방지:
└── 동시 다발적 캐시 미스 시 DB 부하 폭증
└── 해결: Lock 또는 확률적 조기 만료
경고

Cache Stampede 현상: 인기 데이터의 캐시가 만료될 때 동시 다수 요청이 DB를 직접 조회하여 과부하 발생. Locking이나 stale-while-revalidate 패턴으로 방지하세요.

5.3 데이터베이스 복제​

Master-Slave Replication:
[Master] ← (쓰기)
    ↓
    ↓ (비동기 복제)
    ↓
[Slave 1] [Slave 2] [Slave 3] ← (읽기)

Master-Master Replication:
[Master A] ⇄ [Master B]
└── 양방향 쓰기 가능
└── 충돌 해결 필요

5.4 마이크로서비스 아키텍처​

Monolithic:
[Single Application] → [Single Database]
└── 장점: 단순, 배포 용이
└── 단점: 확장 어려움, 장애 전파

Microservices:
[API Gateway]
    ├── [User Service] → [User DB]
    ├── [Order Service] → [Order DB]
    ├── [Payment Service] → [Payment DB]
    └── [Notification Service] → [Message Queue]

장점: 독립 배포, 기술 다양성, 장애 격리
단점: 복잡도 증가, 분산 트랜잭션, 네트워크 비용

6. 메시지 큐​

6.1 비동기 처리​

동기 처리:
Client → [API] → [Heavy Task] → Response
└── 문제: 긴 대기 시간

비동기 처리:
Client → [API] → [Queue] → Response (즉시)
              ↓
         [Worker] → [Heavy Task]

6.2 메시지 큐 패턴​

# Producer (메시지 발행)
from kafka import KafkaProducer

producer = KafkaProducer(
    bootstrap_servers=['localhost:9092'],
    value_serializer=lambda v: json.dumps(v).encode('utf-8')
)

producer.send('order-events', {
    'order_id': 12345,
    'user_id': 67890,
    'amount': 100.00
})

# Consumer (메시지 소비)
from kafka import KafkaConsumer

consumer = KafkaConsumer(
    'order-events',
    bootstrap_servers=['localhost:9092'],
    auto_offset_reset='earliest',
    group_id='order-processor'
)

for message in consumer:
    order = json.loads(message.value)
    process_order(order)

주요 메시지 큐:

  • RabbitMQ: AMQP 프로토콜, 복잡한 라우팅, 우선순위 큐
  • Apache Kafka: 대용량 스트리밍, 높은 처리량, 영구 저장
  • AWS SQS: 완전 관리형, 간단한 설정, FIFO 지원
  • Redis Pub/Sub: 실시간 메시징, 경량, 휘발성

6.3 이벤트 주도 아키텍처​

Event Sourcing:
모든 상태 변경을 이벤트로 저장

[Event Store]
├── OrderCreated {id: 1, amount: 100}
├── PaymentProcessed {id: 1, method: 'card'}
└── OrderShipped {id: 1, tracking: 'AB123'}

현재 상태 = 모든 이벤트의 합성
팁

이벤트 소싱은 감사 추적이 중요한 금융, 의료 시스템에 적합합니다. 모든 변경 이력이 보존되어 시간 여행 디버깅이 가능합니다.

7. 실전 시스템 설계 예시​

7.1 URL 단축 서비스 (예: bit.ly)​

요구사항:

  • 긴 URL을 짧은 URL로 변환
  • 짧은 URL 클릭 시 원본 URL로 리디렉션
  • 일일 1억 건 처리
  • 클릭 통계 추적

설계:

1. URL 인코딩:
   Base62 인코딩 (a-z, A-Z, 0-9)
   예: 12345 → dnh

2. 아키텍처:
   [Client]
      ↓
   [Load Balancer]
      ↓
   [API Servers] ⇄ [Redis Cache]
      ↓              (단축 URL → 원본 URL)
   [PostgreSQL]
      ↓
   [Analytics DB] (클릭 이벤트)

3. 데이터베이스:
   CREATE TABLE urls (
       id BIGSERIAL PRIMARY KEY,
       short_code VARCHAR(10) UNIQUE,
       original_url TEXT,
       created_at TIMESTAMP,
       expires_at TIMESTAMP
   );

   CREATE INDEX idx_short_code ON urls(short_code);

확장성:

  • Read-heavy: Redis 캐싱으로 99% 요청 처리
  • Sharding: short_code 첫 글자 기준
  • Analytics: Kafka → Clickhouse (시계열 분석)

7.2 채팅 시스템 (예: Slack)​

핵심 기능:

  • 실시간 메시징
  • 채널/DM 지원
  • 메시지 이력 조회
  • 파일 업로드

설계:

[Client]
   ↓
[WebSocket Gateway] ← (실시간 연결 유지)
   ↓
[Message Service] → [Kafka] → [DB Writer]
   ↓                            ↓
[Presence Service]         [Cassandra]
   ↓                       (메시지 저장)
[Redis]
(온라인 상태)

메시지 흐름:
1. User A → WebSocket → Message Service
2. Message Service → Kafka Topic (channel_id)
3. Kafka → User B의 WebSocket 연결
4. 비동기 DB 저장 (Cassandra)

데이터 모델:

-- Cassandra (Wide Column Store)
CREATE TABLE messages (
    channel_id UUID,
    message_id TIMEUUID,
    user_id UUID,
    content TEXT,
    created_at TIMESTAMP,
    PRIMARY KEY (channel_id, message_id)
) WITH CLUSTERING ORDER BY (message_id DESC);

3. 상세 설계 (15분)

8.1 규모 추정 예시​

DAU: 1,000만 명
평균 요청: 사용자당 10회/일
QPS = 10,000,000 × 10 / 86,400 ≈ 1,157 QPS
Peak QPS = 1,157 × 3 ≈ 3,500 QPS

데이터 크기:
메시지당 1KB
일일 1억 건 × 1KB = 100GB/일
연간 = 100GB × 365 ≈ 36TB

서버 수 추정:
서버당 1,000 QPS 처리 가능
필요 서버 = 3,500 / 1,000 ≈ 4대 (여유 2배 → 8대)
팁

시스템 설계 면접에서는 완벽한 답보다 사고 과정이 중요합니다. 가정을 명확히 하고, 트레이드오프를 설명하며, 질문에 열린 자세를 보이세요.
