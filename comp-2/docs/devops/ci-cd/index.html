CI/CD

CI/CD는 현대 소프트웨어 개발에서 필수적인 자동화 파이프라인으로, 코드 변경사항을 빠르고 안정적으로 프로덕션 환경에 배포하는 프로세스입니다.

1. CI/CD 개념​

1.1 CI/CD란?​

CI/CD는 Continuous Integration(지속적 통합)과 Continuous Delivery/Deployment(지속적 전달/배포)를 의미하며, 소프트웨어 개발 라이프사이클을 자동화하는 방법론입니다.

graph LR
    A[코드 커밋] --> B[빌드]
    B --> C[테스트]
    C --> D[패키징]
    D --> E[배포]
    E --> F[모니터링]

주요 목표:

  • 개발 주기 단축
  • 코드 품질 향상
  • 배포 위험 감소
  • 팀 협업 효율성 증대

1.2 CI/CD의 이점​

비즈니스 가치:

  • 빠른 시장 출시(Time to Market)
  • 고객 피드백 신속 반영
  • 운영 비용 절감
  • 경쟁력 강화

기술적 이점:

  • 자동화된 테스트로 버그 조기 발견
  • 일관된 빌드 및 배포 프로세스
  • 롤백 용이성
  • 인프라 코드화(Infrastructure as Code)
팁

CI/CD를 도입하면 초기에는 설정 시간이 소요되지만, 장기적으로 개발 생산성이 크게 향상됩니다.

2. CI vs CD vs CD​

2.1 Continuous Integration (CI)​

지속적 통합은 개발자들이 코드 변경사항을 자주 메인 브랜치에 통합하는 관행입니다.

핵심 프로세스:

1. 코드 커밋 및 푸시
   ↓
2. 자동 빌드 트리거
   ↓
3. 단위 테스트 실행
   ↓
4. 정적 코드 분석
   ↓
5. 빌드 결과 알림

CI의 주요 활동:

  • 자동화된 빌드
  • 단위 테스트 및 통합 테스트
  • 코드 품질 검사(Linting, SonarQube 등)
  • 빌드 아티팩트 생성

예시 - 간단한 CI 워크플로우:

// package.json - CI에서 실행할 스크립트
{
  "scripts": {
    "test": "jest --coverage",
    "lint": "eslint src/**/*.js",
    "build": "webpack --mode production"
  }
}

2.2 Continuous Delivery (CD)​

지속적 전달은 코드 변경사항이 자동으로 테스트되고 릴리스 준비 상태로 만들어지는 과정입니다.

특징:

  • 프로덕션 배포 준비 완료 상태 유지
  • 수동 승인 후 프로덕션 배포
  • 스테이징 환경에 자동 배포
  • 릴리스 의사결정은 비즈니스 판단

프로세스:

graph LR
    A[CI 완료] --> B[스테이징 배포]
    B --> C[통합 테스트]
    C --> D[승인 대기]
    D --> E[수동 승인]
    E --> F[프로덕션 배포]

2.3 Continuous Deployment (CD)​

지속적 배포는 모든 변경사항이 자동으로 프로덕션 환경에 배포되는 과정입니다.

특징:

  • 완전 자동화된 배포
  • 수동 개입 없음
  • 높은 신뢰도의 테스트 필요
  • 빠른 피드백 루프

Continuous Delivery vs Continuous Deployment:

구분Continuous DeliveryContinuous Deployment
배포 방식수동 승인 필요완전 자동화
프로덕션 배포수동 트리거자동 트리거
사용 사례규제 산업, 대규모 서비스스타트업, SaaS
위험도낮음높음
경고

Continuous Deployment를 도입하려면 매우 강력한 테스트 커버리지와 모니터링 시스템이 필수입니다.

3. 파이프라인 구성​

3.1 파이프라인 단계​

표준 CI/CD 파이프라인은 다음 단계로 구성됩니다:

1. Source Stage (소스 단계)

- Git 저장소에서 코드 체크아웃
- 의존성 캐싱
- 환경 변수 설정

2. Build Stage (빌드 단계)

- 소스 코드 컴파일
- 정적 자산 번들링
- Docker 이미지 빌드

3. Test Stage (테스트 단계)

- 단위 테스트 (Unit Tests)
- 통합 테스트 (Integration Tests)
- E2E 테스트 (End-to-End Tests)
- 보안 스캔 (Security Scanning)

4. Package Stage (패키징 단계)

- 아티팩트 생성
- 버전 태깅
- 레지스트리 업로드

5. Deploy Stage (배포 단계)

- 스테이징 환경 배포
- 프로덕션 환경 배포
- 헬스 체크

3.2 파이프라인 설계 원칙​

Fail Fast (빠른 실패):

stages:
  - lint        # 1분 - 빠른 피드백
  - unit-test   # 3분 - 기본 검증
  - build       # 5분 - 빌드 검증
  - e2e-test    # 15분 - 심층 검증

병렬 실행:

parallel:
  - job: test-node-14
  - job: test-node-16
  - job: test-node-18

환경 분리:

environments:
  - development   # 자동 배포
  - staging       # 자동 배포 + 승인
  - production    # 수동 승인 필수
팁

파이프라인 실행 시간은 10분 이내로 유지하는 것이 좋습니다. 더 긴 테스트는 야간 배치로 실행하세요.

3.3 환경 관리​

환경별 구성 파일:

# .env.development
NODE_ENV=development
API_URL=http://localhost:3000
DB_HOST=localhost
LOG_LEVEL=debug

# .env.staging
NODE_ENV=staging
API_URL=https://staging-api.example.com
DB_HOST=staging-db.example.com
LOG_LEVEL=info

# .env.production
NODE_ENV=production
API_URL=https://api.example.com
DB_HOST=prod-db.example.com
LOG_LEVEL=error

4. GitHub Actions​

4.1 GitHub Actions 개요​

GitHub Actions는 GitHub에 통합된 CI/CD 플랫폼으로, 워크플로우를 YAML 파일로 정의합니다.

기본 구조:

name: 워크플로우 이름
on: [트리거 이벤트]
jobs:
  job-name:
    runs-on: 운영체제
    steps:
      - name: 단계 이름
        uses: 액션

4.2 기본 워크플로우​

Node.js 애플리케이션 CI:

# .github/workflows/ci.yml
name: CI

on:
  push:
    branches: [ main, develop ]
  pull_request:
    branches: [ main ]

jobs:
  build-and-test:
    runs-on: ubuntu-latest

    strategy:
      matrix:
        node-version: [16.x, 18.x, 20.x]

    steps:
    - name: 코드 체크아웃
      uses: actions/checkout@v4

    - name: Node.js ${{ matrix.node-version }} 설정
      uses: actions/setup-node@v4
      with:
        node-version: ${{ matrix.node-version }}
        cache: 'npm'

    - name: 의존성 설치
      run: npm ci

    - name: 린트 검사
      run: npm run lint

    - name: 테스트 실행
      run: npm test -- --coverage

    - name: 빌드
      run: npm run build

    - name: 코드 커버리지 업로드
      uses: codecov/codecov-action@v3
      with:
        files: ./coverage/lcov.info

4.3 고급 워크플로우​

Docker 이미지 빌드 및 배포:

# .github/workflows/deploy.yml
name: Build and Deploy

on:
  push:
    branches: [ main ]
    tags:
      - 'v*'

env:
  REGISTRY: ghcr.io
  IMAGE_NAME: ${{ github.repository }}

jobs:
  build:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      packages: write

    steps:
    - name: 코드 체크아웃
      uses: actions/checkout@v4

    - name: Docker Buildx 설정
      uses: docker/setup-buildx-action@v3

    - name: 레지스트리 로그인
      uses: docker/login-action@v3
      with:
        registry: ${{ env.REGISTRY }}
        username: ${{ github.actor }}
        password: ${{ secrets.GITHUB_TOKEN }}

    - name: 메타데이터 추출
      id: meta
      uses: docker/metadata-action@v5
      with:
        images: ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}
        tags: |
          type=ref,event=branch
          type=ref,event=pr
          type=semver,pattern={{version}}
          type=semver,pattern={{major}}.{{minor}}

    - name: 이미지 빌드 및 푸시
      uses: docker/build-push-action@v5
      with:
        context: .
        push: true
        tags: ${{ steps.meta.outputs.tags }}
        labels: ${{ steps.meta.outputs.labels }}
        cache-from: type=gha
        cache-to: type=gha,mode=max

  deploy:
    needs: build
    runs-on: ubuntu-latest
    if: github.ref == 'refs/heads/main'

    steps:
    - name: 서버 배포
      uses: appleboy/ssh-action@v1.0.0
      with:
        host: ${{ secrets.DEPLOY_HOST }}
        username: ${{ secrets.DEPLOY_USER }}
        key: ${{ secrets.DEPLOY_KEY }}
        script: |
          cd /app
          docker-compose pull
          docker-compose up -d
          docker system prune -f

4.4 재사용 가능한 워크플로우​

공통 액션 정의:

# .github/workflows/reusable-test.yml
name: Reusable Test Workflow

on:
  workflow_call:
    inputs:
      node-version:
        required: true
        type: string
      test-command:
        required: false
        type: string
        default: 'npm test'

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
    - uses: actions/checkout@v4

    - name: Setup Node.js
      uses: actions/setup-node@v4
      with:
        node-version: ${{ inputs.node-version }}

    - run: npm ci
    - run: ${{ inputs.test-command }}

워크플로우 호출:

# .github/workflows/main.yml
name: Main CI

on: [push, pull_request]

jobs:
  test-node-16:
    uses: ./.github/workflows/reusable-test.yml
    with:
      node-version: '16'

  test-node-18:
    uses: ./.github/workflows/reusable-test.yml
    with:
      node-version: '18'
      test-command: 'npm run test:coverage'
팁

재사용 가능한 워크플로우를 사용하면 중복을 줄이고 유지보수성을 높일 수 있습니다.

4.5 Secrets 및 환경 변수 관리​

jobs:
  deploy:
    runs-on: ubuntu-latest
    environment: production

    steps:
    - name: 배포 스크립트 실행
      env:
        API_KEY: ${{ secrets.API_KEY }}
        DATABASE_URL: ${{ secrets.DATABASE_URL }}
        NODE_ENV: production
      run: |

  • Repository secrets: Settings > Secrets and variables > Actions
  • Environment secrets: Settings > Environments > Environment secrets

5. Jenkins​

5.1 Jenkins 개요​

Jenkins는 오픈소스 자동화 서버로, 플러그인 기반의 확장 가능한 CI/CD 플랫폼입니다.

주요 특징:

  • 1,800개 이상의 플러그인
  • 분산 빌드 지원
  • Pipeline as Code (Jenkinsfile)
  • 다양한 VCS 통합

5.2 Declarative Pipeline​

기본 Jenkinsfile:

// Jenkinsfile
pipeline {
    agent any

    environment {
        NODE_VERSION = '18'
        NPM_CONFIG_CACHE = "${WORKSPACE}/.npm"
    }

    stages {
        stage('Checkout') {
            steps {
                checkout scm
            }
        }

        stage('Install Dependencies') {
            steps {
                sh '''
                    node --version
                    npm --version
                    npm ci
                '''
            }
        }

        stage('Lint') {
            steps {
                sh 'npm run lint'
            }
        }

        stage('Test') {
            steps {
                sh 'npm test -- --coverage'
            }
            post {
                always {
                    junit 'test-results/**/*.xml'
                    publishHTML([
                        reportDir: 'coverage',
                        reportFiles: 'index.html',
                        reportName: 'Coverage Report'
                    ])
                }
            }
        }

        stage('Build') {
            steps {
                sh 'npm run build'
            }
        }

        stage('Archive') {
            steps {
                archiveArtifacts artifacts: 'dist/**/*', fingerprint: true
            }
        }
    }

    post {
        success {
            echo 'Pipeline succeeded!'
            // 슬랙 알림, 이메일 등
        }
        failure {
            echo 'Pipeline failed!'
            // 실패 알림
        }
        always {
            cleanWs()
        }
    }
}

5.3 Scripted Pipeline​

고급 Pipeline 예제:

// Jenkinsfile (Scripted)
node {
    def app
    def dockerImage

    stage('Checkout') {
        checkout scm
        env.GIT_COMMIT_SHORT = sh(
            script: "git rev-parse --short HEAD",
            returnStdout: true
        ).trim()
    }

    stage('Build Docker Image') {
        dockerImage = docker.build(
            "myapp:${env.GIT_COMMIT_SHORT}",
            "--build-arg NODE_ENV=production ."
        )
    }

    stage('Test') {
        dockerImage.inside {
            sh 'npm test'
        }
    }

    stage('Push Image') {
        docker.withRegistry('https://registry.example.com', 'docker-credentials') {
            dockerImage.push("${env.GIT_COMMIT_SHORT}")
            dockerImage.push('latest')
        }
    }

    stage('Deploy to Staging') {
        when {
            branch 'develop'
        }
        steps {
            sh '''
                kubectl set image deployment/myapp \
                    myapp=registry.example.com/myapp:${GIT_COMMIT_SHORT} \
                    -n staging
                kubectl rollout status deployment/myapp -n staging
            '''
        }
    }

    stage('Deploy to Production') {
        when {
            branch 'main'
        }
        input {
            message "Deploy to production?"
            ok "Deploy"
        }
        steps {
            sh '''
                kubectl set image deployment/myapp \
                    myapp=registry.example.com/myapp:${GIT_COMMIT_SHORT} \
                    -n production
                kubectl rollout status deployment/myapp -n production
            '''
        }
    }
}

5.4 멀티 브랜치 파이프라인​

// Jenkinsfile - Multi-branch
pipeline {
    agent any

    parameters {
        choice(
            name: 'DEPLOY_ENV',
            choices: ['dev', 'staging', 'production'],
            description: 'Deployment environment'
        )
    }

    stages {
        stage('Build') {
            steps {
                script {
                    if (env.BRANCH_NAME == 'main') {
                        sh 'npm run build:prod'
                    } else if (env.BRANCH_NAME == 'develop') {
                        sh 'npm run build:staging'
                    } else {
                        sh 'npm run build:dev'
                    }
                }
            }
        }

        stage('Deploy') {
            when {
                anyOf {
                    branch 'main'
                    branch 'develop'
                }
            }
            steps {
                script {
                    def env = (env.BRANCH_NAME == 'main') ? 'production' : 'staging'
                    sh "./deploy.sh ${env}"
                }
            }
        }
    }
}
경고

Jenkins의 Scripted Pipeline은 강력하지만 복잡할 수 있습니다. 가능하면 Declarative Pipeline을 사용하세요.

5.5 Jenkins 플러그인​

필수 플러그인:

플러그인 사용 예제:

pipeline {
    agent any

    stages {
        stage('SonarQube Analysis') {
            steps {
                script {
                    def scannerHome = tool 'SonarScanner'
                    withSonarQubeEnv('SonarQube') {
                        sh """
                            ${scannerHome}/bin/sonar-scanner \
                                -Dsonar.projectKey=myapp \
                                -Dsonar.sources=src \
                                -Dsonar.javascript.lcov.reportPaths=coverage/lcov.info
                        """
                    }
                }
            }
        }

        stage('Quality Gate') {
            steps {
                timeout(time: 1, unit: 'HOURS') {
                    waitForQualityGate abortPipeline: true
                }
            }
        }
    }
}

6. GitLab CI​

6.1 GitLab CI 개요​

GitLab CI는 GitLab에 내장된 CI/CD 도구로, .gitlab-ci.yml 파일로 파이프라인을 정의합니다.

기본 구조:

# .gitlab-ci.yml
stages:
  - build
  - test
  - deploy

variables:
  NODE_VERSION: "18"

before_script:
  - node --version
  - npm --version

build:
  stage: build
  script:
    - npm ci
    - npm run build
  artifacts:
    paths:
      - dist/
    expire_in: 1 week

6.2 실용 예제​

Node.js 애플리케이션 파이프라인:

# .gitlab-ci.yml
image: node:18-alpine

stages:
  - install
  - lint
  - test
  - build
  - deploy

cache:
  key: ${CI_COMMIT_REF_SLUG}
  paths:
    - node_modules/
    - .npm/

variables:
  npm_config_cache: "$CI_PROJECT_DIR/.npm"

install:
  stage: install
  script:
    - npm ci
  artifacts:
    paths:
      - node_modules/
    expire_in: 1 day

lint:
  stage: lint
  needs: [install]
  script:
    - npm run lint

test:unit:
  stage: test
  needs: [install]
  script:
    - npm run test:unit
  coverage: '/All files[^|]*\|[^|]*\s+([\d\.]+)/'
  artifacts:
    reports:
      junit: junit.xml
      coverage_report:
        coverage_format: cobertura
        path: coverage/cobertura-coverage.xml

test:e2e:
  stage: test
  needs: [install]
  services:
    - postgres:14
  variables:
    POSTGRES_DB: testdb
    POSTGRES_USER: testuser
    POSTGRES_PASSWORD: testpass
  script:
    - npm run test:e2e

build:
  stage: build
  needs: [test:unit, test:e2e]
  script:
    - npm run build
  artifacts:
    paths:
      - dist/
    expire_in: 1 week

deploy:staging:
  stage: deploy
  environment:
    name: staging
    url: https://staging.example.com
  needs: [build]
  only:
    - develop
  script:
    - echo "Deploying to staging..."
    - ./deploy.sh staging

deploy:production:
  stage: deploy
  environment:
    name: production
    url: https://example.com
  needs: [build]
  only:
    - main
  when: manual
  script:
    - echo "Deploying to production..."
    - ./deploy.sh production

6.3 Docker 통합​

Docker 이미지 빌드:

# .gitlab-ci.yml
variables:
  DOCKER_DRIVER: overlay2
  DOCKER_TLS_CERTDIR: "/certs"

build:docker:
  stage: build
  image: docker:24
  services:
    - docker:24-dind
  before_script:
    - docker login -u $CI_REGISTRY_USER -p $CI_REGISTRY_PASSWORD $CI_REGISTRY
  script:
    - docker build -t $CI_REGISTRY_IMAGE:$CI_COMMIT_SHORT_SHA .
    - docker build -t $CI_REGISTRY_IMAGE:latest .
    - docker push $CI_REGISTRY_IMAGE:$CI_COMMIT_SHORT_SHA
    - docker push $CI_REGISTRY_IMAGE:latest

7. 아티팩트 관리​

7.1 아티팩트란?​

아티팩트는 빌드 프로세스의 결과물로, 배포 가능한 패키지나 파일을 의미합니다.

아티팩트 유형:

  • 컴파일된 바이너리
  • Docker 이미지
  • NPM/Maven 패키지
  • ZIP/TAR 아카이브
  • 테스트 리포트

7.2 아티팩트 저장소​

주요 저장소 유형:

저장소용도도구 예시
Docker Registry컨테이너 이미지Docker Hub, Harbor, ECR
NPM RegistryNode.js 패키지npmjs.com, Verdaccio
Maven RepositoryJava 아티팩트Nexus, Artifactory
Generic Repository일반 파일S3, MinIO

7.3 아티팩트 버저닝​

Semantic Versioning (SemVer):

# 버전 형식: MAJOR.MINOR.PATCH
1.0.0   # 초기 릴리스
1.0.1   # 패치 (버그 수정)
1.1.0   # 마이너 (기능 추가)
2.0.0   # 메이저 (호환성 깨지는 변경)

자동 버저닝 예제:

# .github/workflows/release.yml
name: Release

on:
  push:
    branches: [ main ]

jobs:
  release:
    runs-on: ubuntu-latest
    steps:
    - uses: actions/checkout@v4
      with:
        fetch-depth: 0

    - name: Semantic Release
      uses: cycjimmy/semantic-release-action@v4
      env:
        GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
        NPM_TOKEN: ${{ secrets.NPM_TOKEN }}

jobs:

8. 배포 자동화​

8.1 배포 전략​

Blue-Green Deployment:

# blue-green-deploy.yml
- name: Blue 환경 배포
  hosts: blue_servers
  tasks:
    - name: 새 버전 배포
      docker_container:
        name: myapp
        image: "myapp:{{ version }}"
        state: started

- name: 헬스 체크
  hosts: blue_servers
  tasks:
    - name: 서비스 상태 확인
      uri:
        url: "http://{{ inventory_hostname }}/health"
        status_code: 200
      register: health_check
      until: health_check.status == 200
      retries: 10
      delay: 5

- name: 트래픽 전환
  hosts: loadbalancer
  tasks:
    - name: Green에서 Blue로 전환
      shell: |
        ./switch-traffic.sh blue

Canary Deployment:

# canary-deploy.yml
stages:
  - name: Canary 10%
    script:
      - kubectl set image deployment/myapp myapp=myapp:${VERSION}
      - kubectl scale deployment/myapp-canary --replicas=1
      - kubectl scale deployment/myapp-stable --replicas=9

  - name: 모니터링
    script:
      - ./monitor-metrics.sh --duration=300 --threshold=0.01

  - name: Canary 50%
    when: manual
    script:
      - kubectl scale deployment/myapp-canary --replicas=5
      - kubectl scale deployment/myapp-stable --replicas=5

  - name: Full Rollout
    when: manual
    script:
      - kubectl scale deployment/myapp-canary --replicas=10
      - kubectl scale deployment/myapp-stable --replicas=0

Rolling Update:

# kubernetes-rolling.yml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp
spec:
  replicas: 10
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 3        # 최대 추가 Pod 수
      maxUnavailable: 1  # 최대 불가용 Pod 수
  template:
    spec:
      containers:
      - name: myapp
        image: myapp:v2.0.0
        readinessProbe:
          httpGet:
            path: /health
            port: 8080
          initialDelaySeconds: 5
          periodSeconds: 5

  CLUSTER_NAME: prod-cluster

# auto-rollback.yml
deploy:

수동 롤백 스크립트:

#!/bin/bash
# rollback.sh

# 현재 배포 히스토리 확인
kubectl rollout history deployment/myapp

# 이전 버전으로 롤백
kubectl rollout undo deployment/myapp

# 특정 리비전으로 롤백
kubectl rollout undo deployment/myapp --to-revision=3

# 롤백 상태 확인
kubectl rollout status deployment/myapp

# 이벤트 확인
kubectl get events --sort-by=.metadata.creationTimestamp

배포 전 확인사항:

# pre-deployment-checks.yml
pre_deploy:

    - ./validate-env.sh

    # 4. 데이터베이스 마이그레이션 테스트

배포 후 검증:

#!/bin/bash
# post-deployment-validation.sh

set -e

echo "🔍 Starting post-deployment validation..."

# 1. 헬스 체크
echo "1️⃣ Health check..."
curl -f https://api.example.com/health || exit 1

# 2. API 기능 테스트
echo "2️⃣ API functional tests..."
./smoke-tests.sh

# 3. 성능 확인
echo "3️⃣ Performance check..."
RESPONSE_TIME=$(curl -o /dev/null -s -w '%{time_total}' https://api.example.com)
if (( $(echo "$RESPONSE_TIME > 2.0" | bc -l) )); then
    echo "❌ Response time too slow: ${RESPONSE_TIME}s"
    exit 1
fi

# 4. 로그 확인
echo "4️⃣ Checking logs for errors..."
kubectl logs -l app=myapp --tail=100 | grep -i error && exit 1 || true

# 5. 메트릭 확인
echo "5️⃣ Checking metrics..."
ERROR_RATE=$(curl -s 'http://prometheus:9090/api/v1/query?query=rate(http_requests_total{status=~"5.."}[5m])' | jq -r '.data.result[0].value[1]')
if (( $(echo "$ERROR_RATE > 0.01" | bc -l) )); then
    echo "❌ Error rate too high: ${ERROR_RATE}"
    exit 1
fi

echo "✅ All validations passed!"

요약​

CI/CD는 현대 소프트웨어 개발의 핵심 요소로, 다음과 같은 가치를 제공합니다:

  1. 자동화: 반복 작업 제거 및 일관성 확보
  2. 빠른 피드백: 문제 조기 발견 및 해결
  3. 안정적인 배포: 검증된 프로세스로 위험 감소
  4. 협업 향상: 팀 전체의 생산성 증대

성공적인 CI/CD 도입을 위해서는:

  • 작은 단위부터 시작하여 점진적으로 확장
  • 팀원 교육 및 문화 정착
  • 지속적인 개선과 최적화
  • 적절한 모니터링과 알림 체계 구축

이 가이드가 여러분의 CI/CD 여정에 도움이 되기를 바랍니다.
