아키텍처 패턴 (Architecture Patterns)

소프트웨어 아키텍처 패턴은 시스템 설계의 청사진을 제공합니다. 각 패턴은 특정 요구사항과 제약사항에 맞는 구조적 솔루션을 제시합니다.

1. Monolithic Architecture​

1.1 개요​

단일 애플리케이션으로 모든 기능이 하나의 코드베이스와 배포 단위로 구성됩니다.

// Monolithic 구조 예시
const express = require('express');
const app = express();

// 모든 기능이 하나의 애플리케이션에 포함
app.use('/api/users', require('./routes/users'));
app.use('/api/products', require('./routes/products'));
app.use('/api/orders', require('./routes/orders'));
app.use('/api/payments', require('./routes/payments'));

app.listen(3000);

1.2 주요 특징​

장점:

  • 간단한 개발 및 배포
  • 쉬운 테스트 환경 구축
  • 낮은 초기 복잡도
  • 단일 코드베이스 관리

단점:

  • 확장성 제한
  • 긴 빌드/배포 시간
  • 기술 스택 변경 어려움
  • 장애 격리 불가
팁

작은 팀이나 초기 스타트업에 적합합니다. 요구사항이 명확하고 빠른 출시가 중요한 경우 효과적입니다.

2. Microservices Architecture​

2.1 개요​

독립적으로 배포 가능한 작은 서비스들로 애플리케이션을 구성하는 패턴입니다.

// User Service (포트 3001)
const express = require('express');
const app = express();

app.get('/users/:id', async (req, res) => {
  const user = await UserModel.findById(req.params.id);
  res.json(user);
});

app.listen(3001);
// Order Service (포트 3002)
const express = require('express');
const axios = require('axios');
const app = express();

app.post('/orders', async (req, res) => {
  // User Service 호출
  const user = await axios.get(`http://user-service:3001/users/${req.body.userId}`);

  const order = await OrderModel.create({
    userId: user.data.id,
    items: req.body.items
  });

  res.json(order);
});

app.listen(3002);

2.2 주요 특징​

장점:

  • 독립적인 확장
  • 기술 스택 다양성
  • 빠른 배포 주기
  • 장애 격리
  • 팀 자율성

단점:

  • 높은 운영 복잡도
  • 분산 시스템 문제
  • 데이터 일관성 관리
  • 네트워크 오버헤드
경고

서비스 간 통신 실패, 데이터 일관성, 분산 트랜잭션 등을 반드시 고려해야 합니다. API Gateway와 서비스 메시 도입을 검토하세요.

3. SOA (Service-Oriented Architecture)​

3.1 개요​

재사용 가능한 서비스들을 통해 비즈니스 기능을 제공하는 아키텍처입니다.

<!-- SOAP 기반 서비스 정의 (WSDL) -->
<definitions name="UserService"
  targetNamespace="http://example.com/user"
  xmlns:soap="http://schemas.xmlsoap.org/wsdl/soap/">

  <message name="GetUserRequest">
    <part name="userId" type="xsd:string"/>
  </message>

  <message name="GetUserResponse">
    <part name="user" type="tns:User"/>
  </message>

  <portType name="UserPortType">
    <operation name="getUser">
      <input message="tns:GetUserRequest"/>
      <output message="tns:GetUserResponse"/>
    </operation>
  </portType>
</definitions>
// ESB (Enterprise Service Bus) 패턴
class ServiceBus {
  constructor() {
    this.services = new Map();
  }

  register(serviceName, endpoint) {
    this.services.set(serviceName, endpoint);
  }

  async route(serviceName, operation, data) {
    const endpoint = this.services.get(serviceName);
    return await endpoint[operation](data);
  }
}

// 사용 예시
const esb = new ServiceBus();
esb.register('userService', userServiceEndpoint);
esb.register('orderService', orderServiceEndpoint);

const user = await esb.route('userService', 'getUser', { id: '123' });

3.2 주요 특징​

장점:

  • 서비스 재사용성
  • 표준화된 통신
  • 느슨한 결합
  • 레거시 통합 용이

단점:

  • ESB 복잡도
  • 중앙 집중식 거버넌스
  • 무거운 프로토콜 (SOAP)
  • 성능 오버헤드
팁

대규모 엔터프라이즈 환경에서 여러 레거시 시스템을 통합할 때 유용합니다. 마이크로서비스보다 무겁지만 표준화가 중요한 경우 적합합니다.

4. Event-Driven Architecture​

4.1 개요​

이벤트 생성, 감지, 소비를 중심으로 시스템을 구성하는 패턴입니다.

// Event Emitter 패턴
const EventEmitter = require('events');

class OrderEventBus extends EventEmitter {}
const eventBus = new OrderEventBus();

// Event Publisher
class OrderService {
  async createOrder(orderData) {
    const order = await Order.create(orderData);

    // 이벤트 발행
    eventBus.emit('order.created', {
      orderId: order.id,
      userId: order.userId,
      total: order.total,
      timestamp: new Date()
    });

    return order;
  }
}

// Event Subscribers
eventBus.on('order.created', async (event) => {
  console.log('[Inventory] 재고 감소 처리:', event.orderId);
  await InventoryService.decreaseStock(event.items);
});

eventBus.on('order.created', async (event) => {
  console.log('[Notification] 주문 확인 이메일 발송:', event.userId);
  await EmailService.sendOrderConfirmation(event.userId, event.orderId);
});

eventBus.on('order.created', async (event) => {
  console.log('[Analytics] 주문 데이터 수집:', event.orderId);
  await AnalyticsService.trackOrder(event);
});
// Message Queue 기반 (RabbitMQ/Kafka)
const amqp = require('amqplib');

// Producer
async function publishEvent(eventType, data) {
  const connection = await amqp.connect('amqp://localhost');
  const channel = await connection.createChannel();

  await channel.assertExchange('events', 'topic', { durable: true });

  channel.publish(
    'events',
    eventType, // routing key: 'order.created', 'user.registered'
    Buffer.from(JSON.stringify(data))
  );

  console.log(`[Published] ${eventType}`);
}

// Consumer
async function subscribeToEvents(pattern) {
  const connection = await amqp.connect('amqp://localhost');
  const channel = await connection.createChannel();

  await channel.assertExchange('events', 'topic', { durable: true });
  const q = await channel.assertQueue('', { exclusive: true });

  await channel.bindQueue(q.queue, 'events', pattern);

  channel.consume(q.queue, (msg) => {
    const event = JSON.parse(msg.content.toString());
    console.log(`[Received] ${msg.fields.routingKey}:`, event);
    channel.ack(msg);
  });
}

// 사용
subscribeToEvents('order.*'); // order.created, order.cancelled 등

4.2 주요 특징​

장점:

  • 느슨한 결합
  • 비동기 처리
  • 확장성 우수
  • 실시간 반응

단점:

  • 디버깅 어려움
  • 이벤트 순서 보장 복잡
  • 메시지 중복 처리
  • 모니터링 복잡도
경고

이벤트 손실, 중복 처리, 순서 보장 문제를 해결해야 합니다. Idempotency(멱등성)를 보장하는 설계가 필수입니다.

5. Layered Architecture​

5.1 개요​

계층별로 책임을 분리하여 시스템을 구성하는 패턴입니다.

// 1. Presentation Layer (Controller)
class UserController {
  constructor(userService) {
    this.userService = userService;
  }

  async getUser(req, res) {
    try {
      const user = await this.userService.getUserById(req.params.id);
      res.json(user);
    } catch (error) {
      res.status(500).json({ error: error.message });
    }
  }

  async createUser(req, res) {
    const user = await this.userService.createUser(req.body);
    res.status(201).json(user);
  }
}

// 2. Business Logic Layer (Service)
class UserService {
  constructor(userRepository, emailService) {
    this.userRepository = userRepository;
    this.emailService = emailService;
  }

  async getUserById(id) {
    const user = await this.userRepository.findById(id);
    if (!user) {
      throw new Error('User not found');
    }
    return user;
  }

  async createUser(userData) {
    // 비즈니스 로직
    if (!this.validateEmail(userData.email)) {
      throw new Error('Invalid email');
    }

    const user = await this.userRepository.create(userData);

    // 이메일 발송
    await this.emailService.sendWelcome(user.email);

    return user;
  }

  validateEmail(email) {
    return /^[^\s@]+@[^\s@]+\.[^\s@]+$/.test(email);
  }
}

// 3. Data Access Layer (Repository)
class UserRepository {
  constructor(database) {
    this.db = database;
  }

  async findById(id) {
    return await this.db.query('SELECT * FROM users WHERE id = ?', [id]);
  }

  async create(userData) {
    const result = await this.db.query(
      'INSERT INTO users (name, email) VALUES (?, ?)',
      [userData.name, userData.email]
    );
    return { id: result.insertId, ...userData };
  }
}

// (실제 데이터베이스 연결 및 쿼리 실행)

6. Hexagonal Architecture (Ports and Adapters)​

6.1 개요​

비즈니스 로직을 외부 의존성으로부터 격리시키는 패턴입니다.

// Domain Layer (Core Business Logic)
class Order {
  constructor(id, userId, items) {
    this.id = id;
    this.userId = userId;
    this.items = items;
    this.status = 'pending';
  }

  calculateTotal() {
    return this.items.reduce((sum, item) => sum + item.price * item.quantity, 0);
  }

  confirm() {
    if (this.status !== 'pending') {
      throw new Error('Order already confirmed');
    }
    this.status = 'confirmed';
  }
}

// Port (Interface) - 비즈니스 로직이 필요로 하는 계약
class OrderRepository {
  async save(order) { throw new Error('Not implemented'); }
  async findById(id) { throw new Error('Not implemented'); }
}

class PaymentGateway {
  async processPayment(amount, userId) { throw new Error('Not implemented'); }
}

// Application Service (Use Case)
class OrderService {
  constructor(orderRepository, paymentGateway, notificationService) {
    this.orderRepository = orderRepository;
    this.paymentGateway = paymentGateway;
    this.notificationService = notificationService;
  }

  async createOrder(userId, items) {
    const order = new Order(null, userId, items);
    const total = order.calculateTotal();

    // 결제 처리
    await this.paymentGateway.processPayment(total, userId);

    // 주문 확정
    order.confirm();

    // 저장
    const savedOrder = await this.orderRepository.save(order);

    // 알림
    await this.notificationService.notify(userId, 'Order confirmed');

    return savedOrder;
  }
}

// Adapter - MongoDB Implementation
class MongoOrderRepository extends OrderRepository {
  constructor(mongoClient) {
    super();
    this.collection = mongoClient.db('shop').collection('orders');
  }

  async save(order) {
    const result = await this.collection.insertOne({
      userId: order.userId,
      items: order.items,
      status: order.status,
      total: order.calculateTotal()
    });
    order.id = result.insertedId;
    return order;
  }

  async findById(id) {
    const doc = await this.collection.findOne({ _id: id });
    return doc ? new Order(doc._id, doc.userId, doc.items) : null;
  }
}

// Adapter - Stripe Payment Gateway
class StripePaymentGateway extends PaymentGateway {
  constructor(stripeClient) {
    super();
    this.stripe = stripeClient;
  }

  async processPayment(amount, userId) {
    return await this.stripe.charges.create({
      amount: amount * 100,
      currency: 'usd',
      customer: userId
    });
  }
}

// Adapter - Email Notification
class EmailNotificationService {
  async notify(userId, message) {
    console.log(`Email sent to user ${userId}: ${message}`);
    // 실제 이메일 전송 로직
  }
}

// Dependency Injection (구성)
const orderRepository = new MongoOrderRepository(mongoClient);
const paymentGateway = new StripePaymentGateway(stripeClient);
const notificationService = new EmailNotificationService();

const orderService = new OrderService(
  orderRepository,
  paymentGateway,
  notificationService
);

7. CQRS (Command Query Responsibility Segregation)​

7.1 개요​

읽기(Query)와 쓰기(Command) 작업을 분리하는 패턴입니다.

// Command Model (Write)
class CreateUserCommand {
  constructor(name, email, password) {
    this.name = name;
    this.email = email;
    this.password = password;
  }
}

class UserCommandHandler {
  constructor(eventStore, writeDb) {
    this.eventStore = eventStore;
    this.writeDb = writeDb;
  }

  async handle(command) {
    // 쓰기 작업
    const userId = await this.writeDb.users.insert({
      name: command.name,
      email: command.email,
      password: await this.hashPassword(command.password),
      createdAt: new Date()
    });

    // Event Sourcing (선택적)
    await this.eventStore.append('UserCreated', {
      userId,
      name: command.name,
      email: command.email,
      timestamp: new Date()
    });

    return userId;
  }

  async hashPassword(password) {
    // 비밀번호 해싱 로직
    return password; // 예시
  }
}

// Query Model (Read)
class UserQueryService {
  constructor(readDb) {
    this.readDb = readDb;
  }

  async getUserProfile(userId) {
    // 읽기 최적화된 뷰
    return await this.readDb.query(`
      SELECT
        u.id,
        u.name,
        u.email,
        COUNT(o.id) as order_count,
        SUM(o.total) as total_spent
      FROM users_view u
      LEFT JOIN orders o ON u.id = o.user_id
      WHERE u.id = ?
      GROUP BY u.id
    `, [userId]);
  }

  async searchUsers(criteria) {
    // ElasticSearch 같은 검색 엔진 사용
    return await this.readDb.search('users', criteria);
  }
}

// API Layer
class UserController {
  constructor(commandHandler, queryService) {
    this.commandHandler = commandHandler;
    this.queryService = queryService;
  }

  // Command (POST)
  async createUser(req, res) {
    const command = new CreateUserCommand(
      req.body.name,
      req.body.email,
      req.body.password
    );

    const userId = await this.commandHandler.handle(command);
    res.status(201).json({ id: userId });
  }

  // Query (GET)
  async getUser(req, res) {
    const user = await this.queryService.getUserProfile(req.params.id);
    res.json(user);
  }

  // Query (GET)
  async searchUsers(req, res) {
    const users = await this.queryService.searchUsers(req.query);
    res.json(users);
  }
}
// Event Sourcing과 결합한 CQRS
class EventStore {
  constructor(db) {
    this.db = db;
  }

  async append(eventType, data) {
    await this.db.events.insert({
      type: eventType,
      data: data,
      timestamp: new Date()
    });
  }

  async getEvents(aggregateId) {
    return await this.db.events.find({ 'data.userId': aggregateId });
  }
}

// Read Model Projection (이벤트 기반 뷰 업데이트)
class UserViewProjection {
  constructor(readDb, eventStore) {
    this.readDb = readDb;
    this.eventStore = eventStore;

    // 이벤트 구독
    this.eventStore.on('UserCreated', this.onUserCreated.bind(this));
    this.eventStore.on('UserUpdated', this.onUserUpdated.bind(this));
  }

  async onUserCreated(event) {
    await this.readDb.users_view.insert({
      id: event.data.userId,
      name: event.data.name,
      email: event.data.email,
      created_at: event.timestamp
    });
  }

  async onUserUpdated(event) {
    await this.readDb.users_view.update(
      { id: event.data.userId },
      { $set: event.data.changes }
    );
  }
}

7.2 주요 특징​

장점:

  • 읽기/쓰기 독립 확장
  • 성능 최적화
  • 복잡한 쿼리 지원
  • 감사 추적 (Event Sourcing)

단점:

  • 시스템 복잡도 증가
  • 데이터 동기화 지연
  • 일관성 관리
  • 높은 학습 곡선
경고

읽기와 쓰기 모델 간 동기화 지연이 발생합니다. Eventual Consistency(최종 일관성)를 허용할 수 있는 시스템에만 적용하세요.

8. Serverless Architecture​

8.1 개요​

서버 관리 없이 함수 단위로 코드를 실행하는 패턴입니다.

// AWS Lambda Function (Node.js)
exports.handler = async (event) => {
  const { httpMethod, path, body } = event;

  if (httpMethod === 'POST' && path === '/users') {
    const userData = JSON.parse(body);

    // DynamoDB에 저장
    const AWS = require('aws-sdk');
    const dynamodb = new AWS.DynamoDB.DocumentClient();

    const params = {
      TableName: 'Users',
      Item: {
        id: AWS.util.uuid.v4(),
        name: userData.name,
        email: userData.email,
        createdAt: new Date().toISOString()
      }
    };

    await dynamodb.put(params).promise();

    return {
      statusCode: 201,
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify(params.Item)
    };
  }

  return {
    statusCode: 404,
    body: JSON.stringify({ error: 'Not found' })
  };
};

exports.imageProcessor = async (event) => {
  const AWS = require('aws-sdk');
  const sharp = require('sharp');
  const s3 = new AWS.S3();

  for (const record of event.Records) {
    const bucket = record.s3.bucket.name;
    const key = decodeURIComponent(record.s3.object.key.replace(/\+/g, ' '));

    // 원본 이미지 다운로드
    const image = await s3.getObject({ Bucket: bucket, Key: key }).promise();

    // 썸네일 생성
    const thumbnail = await sharp(image.Body)
      .resize(200, 200)
      .toBuffer();

    // 썸네일 업로드
    await s3.putObject({
      Bucket: bucket,
      Key: `thumbnails/${key}`,
      Body: thumbnail,
      ContentType: 'image/jpeg'
    }).promise();

    console.log(`Thumbnail created: thumbnails/${key}`);
  }
};

const AWS = require('aws-sdk');

// RESTful API 예시
const handlers = {
  // GET /users/{id}
  getUser: async (event) => {
    const { id } = event.pathParameters;

    const result = await dynamodb.get({
      TableName: process.env.USERS_TABLE,
      Key: { id }
    }).promise();

    return {
      statusCode: 200,
      body: JSON.stringify(result.Item)
    };
  },

  // GET /users
  listUsers: async (event) => {
    const result = await dynamodb.scan({
      TableName: process.env.USERS_TABLE,
      Limit: 100
    }).promise();

    return {
      statusCode: 200,
      body: JSON.stringify(result.Items)
    };
  }
};

exports.handler = async (event) => {
  const handler = handlers[event.requestContext.operationName];
  return handler ? await handler(event) : { statusCode: 404 };

8.2 주요 특징​

장점:

  • 자동 확장
  • 사용한 만큼 비용 지불
  • 서버 관리 불필요
  • 빠른 배포

단점:

  • Cold Start 지연
  • 실행 시간 제한
  • 벤더 종속성
  • 로컬 테스트 어려움
  • 디버깅 복잡
팁

트래픽이 불규칙하거나, 이벤트 기반 처리, 빠른 프로토타입 개발에 적합합니다. AWS Lambda는 최대 15분, Azure Functions는 최대 10분 실행 시간 제한이 있습니다.

경고

Cold Start로 인한 초기 응답 지연이 발생할 수 있습니다. 프로비저닝된 동시성(Provisioned Concurrency)을 사용하거나 keep-alive 패턴을 고려하세요.

10. 선택 가이드​

10.1 프로젝트 규모별​

소규모 프로젝트:

  • Monolithic + Layered Architecture
  • 간단한 배포, 빠른 개발

중규모 프로젝트:

  • Layered Architecture + Event-Driven
  • Hexagonal Architecture
  • 적절한 확장성과 유지보수성

대규모 프로젝트:

  • Microservices + Event-Driven
  • CQRS + Event Sourcing
  • 높은 확장성과 장애 격리

10.2 요구사항별​

높은 트래픽:

  • Microservices
  • CQRS
  • Serverless

복잡한 비즈니스 로직:

  • Hexagonal Architecture
  • Domain-Driven Design

실시간 처리:

  • Event-Driven Architecture
  • Serverless

레거시 통합:

  • SOA
  • API Gateway 패턴
하이브리드 접근

실무에서는 여러 패턴을 혼합하여 사용합니다. 예: Microservices + Event-Driven + CQRS를 함께 적용하거나, Monolithic 내부에서 Hexagonal Architecture를 사용할 수 있습니다.
