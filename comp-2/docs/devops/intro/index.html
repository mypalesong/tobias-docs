DevOps 가이드

DevOps는 현대 소프트웨어 개발과 운영의 핵심 방법론입니다. 이 가이드는 DevOps의 기초부터 실무까지 체계적으로 다룹니다.

1. DevOps란?​

1.1 정의​

DevOps는 Development(개발)와 Operations(운영)의 합성어로, 소프트웨어 개발팀과 IT 운영팀 간의 협업을 강화하여 소프트웨어 제품과 서비스를 빠르고 안정적으로 개발, 배포, 운영하는 문화와 방법론입니다.

DevOps는 단순한 도구나 프로세스가 아니라, 문화적 철학, 실천 방법, 도구의 조합입니다.

1.2 DevOps의 탄생 배경​

전통적인 소프트웨어 개발 환경에서는:

  • 개발팀: 새로운 기능을 빠르게 출시하고자 함
  • 운영팀: 시스템의 안정성과 신뢰성을 유지하고자 함

이러한 목표의 차이로 인해 두 팀 간의 갈등과 비효율이 발생했습니다.

전통적인 문제점:
┌─────────────┐        ┌─────────────┐
│  개발팀      │  벽   │   운영팀     │
│             │  ════  │             │
│ "빠른 변화" │        │ "안정성"    │
└─────────────┘        └─────────────┘
     ↓                        ↓
   혁신 추구              변경 최소화

DevOps는 이러한 **사일로(Silo)**를 허물고 협업을 통해 두 목표를 동시에 달성합니다.

1.3 DevOps의 핵심 가치​

  ┌────────────────────────────────────┐
  │         DevOps 핵심 가치            │
  └────────────────────────────────────┘
           │
           ├─── 문화(Culture): 협업과 신뢰
           │
           ├─── 자동화(Automation): 반복 작업 제거
           │
           ├─── 측정(Measurement): 데이터 기반 의사결정
           │
           ├─── 공유(Sharing): 지식과 책임 공유
           │
           └─── 지속적 개선(Continuous Improvement)
DevOps의 본질

DevOps는 도구가 아니라 문화입니다. 도구는 DevOps 문화를 실현하는 수단일 뿐입니다.

2. SDLC (Software Development Life Cycle) 개요​

2.1 SDLC란?​

SDLC는 소프트웨어를 계획, 개발, 테스트, 배포하는 전체 생명주기를 의미합니다.

2.2 전통적인 SDLC 단계​

1. 계획(Planning)
   ↓
2. 요구사항 분석(Requirements Analysis)
   ↓
3. 설계(Design)
   ↓
4. 구현(Implementation)
   ↓
5. 테스트(Testing)
   ↓
6. 배포(Deployment)
   ↓
7. 유지보수(Maintenance)

2.3 DevOps와 SDLC​

DevOps는 SDLC를 순환적이고 지속적으로 만듭니다:

        ┌─────────────────────────────────────┐
        │                                     │
        ↓                                     │
   ┌─────────┐    ┌─────────┐    ┌─────────┐│
   │  Plan   │───→│  Code   │───→│  Build  ││
   └─────────┘    └─────────┘    └─────────┘│
        ↑                              ↓     │
   ┌─────────┐    ┌─────────┐    ┌─────────┐│
   │ Monitor │←───│ Deploy  │←───│  Test   ││
   └─────────┘    └─────────┘    └─────────┘│
        │                                     │
        └─────────────────────────────────────┘
           DevOps Infinite Loop

2.4 주요 SDLC 모델​

2.4.1 Waterfall Model​

계획 → 분석 → 설계 → 구현 → 테스트 → 배포
  • 특징: 순차적, 각 단계 완료 후 다음 단계 진행
  • 단점: 유연성 부족, 피드백 반영 어려움

2.4.2 Agile Model​

스프린트 1: 계획→개발→테스트→리뷰
    ↓
스프린트 2: 계획→개발→테스트→리뷰
    ↓
스프린트 3: 계획→개발→테스트→리뷰
  • 특징: 반복적, 점진적 개발
  • 장점: 빠른 피드백, 유연한 대응

2.4.3 DevOps Model​

Agile을 확장하여 배포와 운영까지 자동화:

지속적 통합(CI) + 지속적 배포(CD) + 지속적 모니터링

3.2 워크플로우 비교​

3.2.1 Traditional Development​

개발자가 코드 작성
    ↓
수동으로 빌드
    ↓
QA팀에게 전달
    ↓
수동 테스트 (1-2주)
    ↓
버그 발견 → 개발자에게 반환
    ↓
수정 후 재테스트
    ↓
운영팀에게 배포 요청
    ↓
운영팀이 수동 배포 (주말 작업)
    ↓
배포 후 문제 발생 시 롤백 어려움

소요 시간: 2-4주 이상

3.2.2 DevOps Workflow​

개발자가 코드 커밋 (Git)
    ↓
자동 빌드 시작 (CI)
    ↓
자동 테스트 실행 (Unit, Integration)
    ↓
테스트 통과 시 자동 배포 (CD)
    ↓
모니터링 및 로그 수집
    ↓
문제 발견 시 자동 알림 또는 롤백

소요 시간: 분 단위

배포 빈도와 안정성

연구에 따르면, 배포 빈도가 높을수록 시스템이 더 안정적입니다. DevOps를 잘 실천하는 조직은 하루에 수십~수백 번 배포하면서도 장애율이 낮습니다.

3.3 문화적 차이​

Traditional Development​

  • 책임 분리: "우리는 코드만 짜요. 배포는 운영팀 일이에요."
  • 비난 문화: "누가 이 버그를 만들었어?"
  • 문서 중심: 모든 것을 문서화
  • 변경 두려움: "작동하면 건드리지 마!"

DevOps​

  • 공동 책임: "우리는 함께 제품을 만들고 운영합니다."
  • 학습 문화: "이 장애에서 무엇을 배울 수 있을까?"
  • 코드 중심: 코드로 모든 것을 표현
  • 지속적 개선: "더 나은 방법이 있을까?"

4. DevOps 핵심 개념​

4.1 CI/CD (Continuous Integration / Continuous Deployment)​

4.1.1 Continuous Integration (지속적 통합)​

개발자들이 작성한 코드를 자주(하루에 여러 번) 메인 브랜치에 통합하고, 자동으로 빌드 및 테스트하는 프랙티스입니다.

CI 프로세스:

개발자 코드 커밋
    ↓
Git에 Push
    ↓
CI 서버가 감지 (예: Jenkins, GitHub Actions)
    ↓
소스 코드 체크아웃
    ↓
빌드 실행
    ↓
자동 테스트 실행
    - Unit Tests
    - Integration Tests
    - Code Quality Checks (Lint, Sonar)
    ↓
결과 통보 (성공/실패)
    ↓
실패 시 즉시 수정

CI 예제: GitHub Actions

name: CI Pipeline

on:
  push:
    branches: [ main, develop ]
  pull_request:
    branches: [ main ]

jobs:
  build-and-test:
    runs-on: ubuntu-latest

    steps:
    - name: Checkout code
      uses: actions/checkout@v3

    - name: Setup Node.js
      uses: actions/setup-node@v3
      with:
        node-version: '18'
        cache: 'npm'

    - name: Install dependencies
      run: npm ci

    - name: Run linter
      run: npm run lint

    - name: Run tests
      run: npm test -- --coverage

    - name: Build application
      run: npm run build

    - name: Upload coverage reports
      uses: codecov/codecov-action@v3
      with:
        token: ${{ secrets.CODECOV_TOKEN }}

4.1.2 Continuous Delivery (지속적 전달)​

CI를 확장하여 코드가 항상 배포 가능한 상태로 유지되도록 하는 프랙티스입니다. 배포는 수동 승인 후 진행됩니다.

4.1.3 Continuous Deployment (지속적 배포)​

테스트를 통과한 모든 변경사항이 자동으로 프로덕션에 배포되는 프랙티스입니다.

Continuous Delivery vs Continuous Deployment

CD (Delivery):
코드 → CI → 자동 테스트 → 스테이징 → [수동 승인] → 프로덕션

CD (Deployment):
코드 → CI → 자동 테스트 → 스테이징 → 프로덕션
                                      ↑
                                  자동 배포

CD 예제: GitHub Actions + AWS

name: CD Pipeline

on:
  push:
    branches: [ main ]

jobs:
  deploy:
    runs-on: ubuntu-latest

    steps:
    - name: Checkout code
      uses: actions/checkout@v3

    - name: Configure AWS credentials
      uses: aws-actions/configure-aws-credentials@v2
      with:
        aws-access-key-id: ${{ secrets.AWS_ACCESS_KEY_ID }}
        aws-secret-access-key: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
        aws-region: ap-northeast-2

    - name: Login to Amazon ECR
      id: login-ecr
      uses: aws-actions/amazon-ecr-login@v1

    - name: Build, tag, and push image to Amazon ECR
      env:
        ECR_REGISTRY: ${{ steps.login-ecr.outputs.registry }}
        ECR_REPOSITORY: my-app
        IMAGE_TAG: ${{ github.sha }}
      run: |
        docker build -t $ECR_REGISTRY/$ECR_REPOSITORY:$IMAGE_TAG .
        docker push $ECR_REGISTRY/$ECR_REPOSITORY:$IMAGE_TAG

    - name: Deploy to ECS
      run: |
        aws ecs update-service \
          --cluster my-cluster \
          --service my-service \
          --force-new-deployment
CI/CD 실패 시 대응

CI/CD 파이프라인이 실패하면 즉시 수정하는 것이 최우선입니다. "나중에 고치지"는 기술 부채를 쌓는 지름길입니다.

4.2 Infrastructure as Code (IaC)​

인프라를 코드로 관리하는 방법론입니다. 서버, 네트워크, 스토리지 등을 코드로 정의하고 버전 관리합니다.

4.2.1 IaC의 장점​

  • 재현성: 동일한 환경을 반복적으로 생성 가능
  • 버전 관리: Git으로 인프라 변경 이력 추적
  • 자동화: 수동 설정 오류 제거
  • 문서화: 코드 자체가 인프라 문서
  • 테스트 가능: 인프라도 테스트 가능

4.2.2 IaC 도구​

Terraform 예제:

# main.tf
provider "aws" {
  region = "ap-northeast-2"
}

# VPC 생성
resource "aws_vpc" "main" {
  cidr_block           = "10.0.0.0/16"
  enable_dns_hostnames = true
  enable_dns_support   = true

  tags = {
    Name        = "main-vpc"
    Environment = "production"
  }
}

# 서브넷 생성
resource "aws_subnet" "public" {
  count                   = 2
  vpc_id                  = aws_vpc.main.id
  cidr_block              = "10.0.${count.index + 1}.0/24"
  availability_zone       = data.aws_availability_zones.available.names[count.index]
  map_public_ip_on_launch = true

  tags = {
    Name = "public-subnet-${count.index + 1}"
    Type = "public"
  }
}

# EC2 인스턴스 생성
resource "aws_instance" "web" {
  ami           = "ami-0c76973fbe0ee100c"
  instance_type = "t3.micro"
  subnet_id     = aws_subnet.public[0].id

  user_data = <<-EOF
              #!/bin/bash
              yum update -y
              yum install -y docker
              systemctl start docker
              systemctl enable docker
              EOF

  tags = {
    Name = "web-server"
  }
}

# 출력
output "instance_public_ip" {
  value = aws_instance.web.public_ip
}

  become: yes

        name: "{{ app_user }}"
        shell: /bin/bash
        create_home: yes

    - name: Copy application files
      copy:
        src: ./app/
        dest: /var/www/app/
        owner: "{{ app_user }}"
        group: "{{ app_user }}"
        mode: '0755'

    - name: Install Node.js dependencies
      npm:
        path: /var/www/app
        state: present
      become_user: "{{ app_user }}"

    - name: Configure Nginx
      template:
        src: nginx.conf.j2
        dest: /etc/nginx/sites-available/app
      notify: restart nginx

    - name: Enable Nginx site
      file:
        src: /etc/nginx/sites-available/app
        dest: /etc/nginx/sites-enabled/app
        state: link
      notify: restart nginx

    - name: Start application service
      systemd:
        name: webapp
        state: started
        enabled: yes

  handlers:
    - name: restart nginx
      service:
        name: nginx
        state: restarted
IaC 모범 사례
  • 상태 파일(State)을 원격 스토리지에 보관 (예: S3, Terraform Cloud)
  • 민감한 정보는 변수로 분리하고 암호화
  • 모듈화하여 재사용성 높이기
  • 변경 전 plan 명령으로 미리 확인

4.3 Automation (자동화)​

DevOps의 핵심은 반복 가능한 모든 것을 자동화하는 것입니다.

4.3.1 자동화 영역​

┌─────────────────────────────────────────┐
│         자동화 영역                      │
└─────────────────────────────────────────┘
    │
    ├─ 코드 통합 (CI)
    ├─ 테스트 (Unit, Integration, E2E)
    ├─ 보안 스캔 (SAST, DAST, SCA)
    ├─ 빌드 및 패키징
    ├─ 배포 (CD)
    ├─ 인프라 프로비저닝 (IaC)
    ├─ 구성 관리
    ├─ 모니터링 및 알림
    ├─ 백업 및 복구
    └─ 스케일링

4.3.2 자동화 예제: 테스트 자동화​

Jest를 사용한 자동화된 테스트:

// __tests__/userService.test.js
const UserService = require('../src/services/userService');
const db = require('../src/database');

describe('UserService', () => {
  let userService;

  beforeEach(() => {
    userService = new UserService(db);
  });

  afterEach(async () => {
    await db.cleanup();
  });

  describe('createUser', () => {
    it('should create a new user with valid data', async () => {
      const userData = {
        username: 'testuser',
        email: 'test@example.com',
        password: 'SecurePass123!'
      };

      const user = await userService.createUser(userData);

      expect(user).toBeDefined();
      expect(user.id).toBeTruthy();
      expect(user.username).toBe(userData.username);
      expect(user.email).toBe(userData.email);
      expect(user.password).not.toBe(userData.password); // 해시되어야 함
    });

    it('should throw error for duplicate username', async () => {
      const userData = {
        username: 'duplicate',
        email: 'test1@example.com',
        password: 'Pass123!'
      };

      await userService.createUser(userData);

      await expect(
        userService.createUser({
          ...userData,
          email: 'test2@example.com'
        })
      ).rejects.toThrow('Username already exists');
    });

    it('should validate email format', async () => {
      const userData = {
        username: 'testuser',
        email: 'invalid-email',
        password: 'Pass123!'
      };

      await expect(
        userService.createUser(userData)
      ).rejects.toThrow('Invalid email format');
    });
  });

  describe('getUserById', () => {
    it('should return user when found', async () => {
      const created = await userService.createUser({
        username: 'findme',
        email: 'findme@example.com',
        password: 'Pass123!'
      });

      const found = await userService.getUserById(created.id);

      expect(found).toEqual(expect.objectContaining({
        id: created.id,
        username: 'findme'
      }));
    });

    it('should return null when user not found', async () => {
      const user = await userService.getUserById('nonexistent-id');
      expect(user).toBeNull();
    });
  });
});

4.4 Monitoring and Logging (모니터링과 로깅)​

// metrics.js
const client = require('prom-client');

// 기본 메트릭 수집
const collectDefaultMetrics = client.collectDefaultMetrics;
collectDefaultMetrics({ timeout: 5000 });

// 커스텀 메트릭 정의
const httpRequestDuration = new client.Histogram({
  name: 'http_request_duration_seconds',
  help: 'Duration of HTTP requests in seconds',
  labelNames: ['method', 'route', 'status_code']
});

const httpRequestTotal = new client.Counter({
  name: 'http_requests_total',
  help: 'Total number of HTTP requests',
  labelNames: ['method', 'route', 'status_code']
});

const activeConnections = new client.Gauge({
  name: 'active_connections',
  help: 'Number of active connections'
});

// Express 미들웨어
function metricsMiddleware(req, res, next) {
  const start = Date.now();

  activeConnections.inc();

  res.on('finish', () => {
    const duration = (Date.now() - start) / 1000;
    const route = req.route ? req.route.path : req.path;

    httpRequestDuration
      .labels(req.method, route, res.statusCode)
      .observe(duration);

    httpRequestTotal
      .labels(req.method, route, res.statusCode)
      .inc();

    activeConnections.dec();
  });

  next();
}

// 메트릭 엔드포인트
function metricsEndpoint(req, res) {
  res.set('Content-Type', client.register.contentType);
  res.end(client.register.metrics());
}

module.exports = {
  metricsMiddleware,
  metricsEndpoint
};

5. DevOps 문화와 철학​

5.1 CALMS 프레임워크​

DevOps 문화의 핵심 요소:

┌────────────────────────────────────────┐
│      CALMS Framework                   │
└────────────────────────────────────────┘

C - Culture (문화)
    협업, 신뢰, 투명성

A - Automation (자동화)
    반복 작업 제거, 효율성 향상

L - Lean (린)
    낭비 제거, 가치 흐름 최적화

M - Measurement (측정)
    데이터 기반 의사결정

S - Sharing (공유)
    지식, 책임, 성과 공유

5.2 Three Ways (세 가지 길)​

The Phoenix Project에서 제시한 DevOps 원칙:

5.2.1 First Way: 시스템 사고 (Systems Thinking)​

개발 → 테스트 → 운영 → 고객
  ↓      ↓      ↓      ↓
최적화  자동화  안정화  만족
  • 전체 시스템의 흐름 최적화
  • 부분 최적화가 아닌 전체 최적화
  • 병목 지점 발견 및 개선

5.2.2 Second Way: 피드백 루프 증폭 (Amplify Feedback Loops)​

      문제 발견
         ↑
고객 ← 운영 ← 테스트 ← 개발
    피드백  피드백  피드백
  • 빠른 피드백으로 문제 조기 발견
  • 오른쪽에서 왼쪽으로의 정보 흐름 강화
  • 실패로부터 빠르게 학습

5.2.3 Third Way: 지속적 실험과 학습​

실험 → 측정 → 학습 → 개선
  ↑                    ↓
  └────────────────────┘
  • 위험을 감수하고 실험
  • 실패를 학습 기회로 활용
  • 지속적인 개선 문화

5.3 Blameless Postmortem (비난 없는 사후 분석)​

장애 발생 시, 개인을 비난하지 않고 시스템 개선에 집중합니다.

Postmortem 문서 예제:

# Postmortem: 2024-01-15 데이터베이스 장애

## 요약
- 날짜: 2024-01-15 14:30 - 15:45 KST
- 영향: 75분간 서비스 불가
- 영향받은 사용자: 약 50,000명
- 근본 원인: 데이터베이스 연결 풀 고갈

## 타임라인
- 14:30 - 모니터링 알람 발생 (응답 시간 증가)
- 14:32 - 온콜 엔지니어 확인 시작
- 14:35 - 데이터베이스 연결 문제 식별
- 14:40 - 긴급 스케일업 시도 (실패)
- 14:50 - 연결 풀 설정 조정
- 15:15 - 점진적 트래픽 복구 시작
- 15:45 - 완전 복구

## 근본 원인
최근 배포된 코드에서 데이터베이스 연결을 제대로 닫지 않아
연결 풀이 고갈됨. 특히 에러 처리 경로에서 연결이 누수됨.

## 해결 방법
1. 즉시: 문제가 있는 배포 롤백
2. 단기: 연결 풀 크기 증가
3. 장기: 코드 리뷰 프로세스 개선

## 액션 아이템
- [ ] 데이터베이스 연결 관리 라이브러리 도입 (담당: @dev-team, 기한: 2024-01-22)
- [ ] 연결 누수 감지 테스트 추가 (담당: @qa-team, 기한: 2024-01-25)
- [ ] 데이터베이스 모니터링 강화 (담당: @ops-team, 기한: 2024-01-20)
- [ ] 장애 대응 플레이북 업데이트 (담당: @sre-team, 기한: 2024-01-18)

## 배운 점
- 에러 처리 경로도 철저히 테스트해야 함
- 연결 관리는 자동화되어야 함 (try-finally, context manager)
- 프로덕션 배포 전 부하 테스트 필요
실패는 학습의 기회

"Fail fast, learn faster" - 빨리 실패하고 더 빠르게 배우세요.

5.4 You Build It, You Run It​

Amazon의 Werner Vogels가 제시한 원칙: 만든 사람이 운영한다.

전통적 방식:
개발팀: "코드는 완성했어요. 이제 운영팀이 돌려주세요."
운영팀: "이게 왜 안 돌아가죠? 개발팀에 물어봐야겠어요."

DevOps 방식:
팀: "우리가 만들었고, 우리가 운영하며, 우리가 책임집니다."

장점:

  • 개발자가 운영 관점에서 코드 작성
  • 빠른 피드백과 개선
  • 책임감 향상

pipeline {
    agent any

    environment {

                    url: 'https://github.com/user/repo.git'
            }
        }

        stage('Build') {
            steps {
                script {
                    sh 'npm install'
                    sh 'npm run build'
                }
            }
        }

        stage('Test') {
            parallel {
                stage('Unit Tests') {
                    steps {
                        sh 'npm run test:unit'
                    }
                }
                stage('Integration Tests') {
                    steps {
                        sh 'npm run test:integration'
                    }
                }
                stage('Lint') {
                    steps {
                        sh 'npm run lint'
                    }
                }
            }
        }

        stage('Security Scan') {
            steps {
                sh 'npm audit'
                sh 'trivy fs --security-checks vuln .'
            }
        }

        stage('Build Docker Image') {
            steps {
                script {
                    def imageTag = "${env.BUILD_NUMBER}"
                    sh """
                        docker build -t ${DOCKER_REGISTRY}/${APP_NAME}:${imageTag} .
                        docker tag ${DOCKER_REGISTRY}/${APP_NAME}:${imageTag} \
                                   ${DOCKER_REGISTRY}/${APP_NAME}:latest
                    """
                }
            }
        }

        stage('Push to Registry') {
            steps {
                script {
                    sh """
                        echo ${DOCKER_CREDENTIALS_PSW} | docker login ${DOCKER_REGISTRY} \
                            -u ${DOCKER_CREDENTIALS_USR} --password-stdin
                        docker push ${DOCKER_REGISTRY}/${APP_NAME}:${env.BUILD_NUMBER}
                        docker push ${DOCKER_REGISTRY}/${APP_NAME}:latest
                    """
                }
            }
        }

        stage('Deploy to Staging') {
            steps {
                sh """
                    kubectl set image deployment/${APP_NAME} \
                        ${APP_NAME}=${DOCKER_REGISTRY}/${APP_NAME}:${env.BUILD_NUMBER} \
                        -n staging
                """
            }
        }

        stage('Smoke Test') {
            steps {
                sh 'npm run test:smoke -- --env=staging'
            }
        }

        stage('Deploy to Production') {
            when {
                branch 'main'
            }
            steps {
                input message: 'Deploy to production?', ok: 'Deploy'
                sh """
                    kubectl set image deployment/${APP_NAME} \
                        ${APP_NAME}=${DOCKER_REGISTRY}/${APP_NAME}:${env.BUILD_NUMBER} \
                        -n production
                """
            }
        }
    }

    post {
        success {
            slackSend(
                color: 'good',
                message: "Build Successful: ${env.JOB_NAME} #${env.BUILD_NUMBER}"
            )
        }
        failure {
            slackSend(
                color: 'danger',
                message: "Build Failed: ${env.JOB_NAME} #${env.BUILD_NUMBER}"
            )
        }
        always {
            cleanWs()
        }
    }

9. DevOps 채택 이점​

9.1 비즈니스 이점​

9.1.1 출시 시간 단축 (Time to Market)​

전통적 방식:
요구사항 → 개발 (2주) → QA (1주) → 배포 대기 (1주) → 배포
총 소요 시간: 4주+

DevOps 방식:
요구사항 → 개발 → 자동 테스트 → 자동 배포
총 소요 시간: 수 시간 ~ 1일

실제 사례:

  • Amazon: 하루 평균 수천 번 배포
  • Netflix: 하루 수백 번 배포
  • Etsy: 하루 평균 50회 이상 배포

9.1.2 비용 절감​

비용 절감 영역:

1. 자동화로 인한 인건비 절감
   수동 배포: 4시간 × 4명 = 16 시간
   자동 배포: 10분 × 검증만 필요

2. 클라우드 리소스 최적화
   - 오토스케일링으로 과도한 프로비저닝 방지
   - 미사용 리소스 자동 정리
   - 예약 인스턴스 활용

3. 장애 비용 감소
   - 빠른 감지 및 복구
   - 자동 롤백
   - 장애 시간 최소화

9.1.3 품질 향상​

품질 향상 메커니즘:

1. 지속적 테스트
   ├─ 모든 커밋마다 자동 테스트
   ├─ 버그 조기 발견 (비용 ↓)
   └─ 테스트 커버리지 가시화

2. 빠른 피드백
   ├─ 실시간 모니터링
   ├─ 사용자 피드백 즉시 반영
   └─ A/B 테스트 용이

3. 작은 변경
   ├─ 변경 범위 축소
   ├─ 리스크 감소
   └─ 문제 추적 용이

9.2.1 안정성 향상​

9.2.2 확장성​

수평적 확장 (Horizontal Scaling):

전통적 방식:
1. 트래픽 증가 감지
2. 서버 구매 요청
3. 승인 대기 (수일)
4. 서버 도착 (수주)
5. 설정 및 배포 (수일)
총 소요: 수주 ~ 수개월

DevOps + 클라우드:
1. 메트릭이 임계값 초과
2. 오토스케일러가 자동으로 인스턴스 추가
3. 로드 밸런서가 트래픽 분산
총 소요: 수분

Kubernetes HPA 자동 스케일링:

apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: app-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: myapp
  minReplicas: 2
  maxReplicas: 50
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70
  - type: Pods
    pods:
      metric:
        name: http_requests_per_second
      target:
        type: AverageValue
        averageValue: "1000"
  behavior:
    scaleUp:
      stabilizationWindowSeconds: 0
      policies:
      - type: Percent
        value: 100
        periodSeconds: 15
    scaleDown:
      stabilizationWindowSeconds: 300
      policies:
      - type: Percent
        value: 50
        periodSeconds: 60

# .github/workflows/ci.yml
name: Continuous Integration

on:
  pull_request:
    branches: [ main, develop ]
  push:
    branches: [ main, develop ]

env:
  NODE_VERSION: '18'
  DOCKER_REGISTRY: ghcr.io
  IMAGE_NAME: ${{ github.repository }}

jobs:
  lint:
    name: Code Linting
    runs-on: ubuntu-latest

    steps:
    - name: Checkout code
      uses: actions/checkout@v3

    - name: Setup Node.js
      uses: actions/setup-node@v3
      with:
        node-version: ${{ env.NODE_VERSION }}
        cache: 'npm'

    - name: Install dependencies
      run: npm ci

    - name: Run ESLint
      run: npm run lint

    - name: Run Prettier check
      run: npm run format:check

  test:
    name: Unit & Integration Tests
    runs-on: ubuntu-latest

    services:
      postgres:
        image: postgres:15
        env:
          POSTGRES_DB: test_db
          POSTGRES_USER: test_user
          POSTGRES_PASSWORD: test_password
        options: >-
          --health-cmd pg_isready
          --health-interval 10s
          --health-timeout 5s
          --health-retries 5
        ports:
          - 5432:5432

      redis:
        image: redis:7-alpine
        options: >-
          --health-cmd "redis-cli ping"
          --health-interval 10s
          --health-timeout 5s
          --health-retries 5
        ports:
          - 6379:6379

    steps:
    - name: Checkout code
      uses: actions/checkout@v3

    - name: Setup Node.js
      uses: actions/setup-node@v3
      with:
        node-version: ${{ env.NODE_VERSION }}
        cache: 'npm'

    - name: Install dependencies
      run: npm ci

    - name: Run unit tests
      run: npm run test:unit -- --coverage
      env:
        NODE_ENV: test

    - name: Run integration tests
      run: npm run test:integration
      env:
        NODE_ENV: test
        DATABASE_URL: postgresql://test_user:test_password@localhost:5432/test_db
        REDIS_URL: redis://localhost:6379

    - name: Upload coverage to Codecov
      uses: codecov/codecov-action@v3
      with:
        files: ./coverage/coverage-final.json
        flags: unittests
        name: codecov-umbrella

  security:
    name: Security Scanning
    runs-on: ubuntu-latest

    steps:
    - name: Checkout code
      uses: actions/checkout@v3

    - name: Run npm audit
      run: npm audit --audit-level=moderate

    - name: Run Snyk security scan
      uses: snyk/actions/node@master
      env:
        SNYK_TOKEN: ${{ secrets.SNYK_TOKEN }}
      with:
        args: --severity-threshold=high

    - name: Run Trivy vulnerability scanner
      uses: aquasecurity/trivy-action@master
      with:
        scan-type: 'fs'
        scan-ref: '.'
        format: 'sarif'
        output: 'trivy-results.sarif'

    - name: Upload Trivy results to GitHub Security
      uses: github/codeql-action/upload-sarif@v2
      with:
        sarif_file: 'trivy-results.sarif'

  build:
    name: Build Application
    runs-on: ubuntu-latest
    needs: [lint, test, security]

    steps:
    - name: Checkout code
      uses: actions/checkout@v3

    - name: Setup Node.js
      uses: actions/setup-node@v3
      with:
        node-version: ${{ env.NODE_VERSION }}
        cache: 'npm'

    - name: Install dependencies
      run: npm ci

    - name: Build application
      run: npm run build
      env:
        NODE_ENV: production

    - name: Upload build artifacts
      uses: actions/upload-artifact@v3
      with:
        name: build-artifacts
        path: dist/
        retention-days: 7

  docker:
    name: Build Docker Image
    runs-on: ubuntu-latest
    needs: [build]
    if: github.event_name == 'push'

    permissions:
      contents: read
      packages: write

    steps:
    - name: Checkout code
      uses: actions/checkout@v3

    - name: Set up Docker Buildx
      uses: docker/setup-buildx-action@v2

    - name: Log in to Container Registry
      uses: docker/login-action@v2
      with:
        registry: ${{ env.DOCKER_REGISTRY }}
        username: ${{ github.actor }}
        password: ${{ secrets.GITHUB_TOKEN }}

    - name: Extract metadata
      id: meta
      uses: docker/metadata-action@v4
      with:
        images: ${{ env.DOCKER_REGISTRY }}/${{ env.IMAGE_NAME }}
        tags: |
          type=ref,event=branch
          type=sha,prefix={{branch}}-
          type=semver,pattern={{version}}

    - name: Build and push Docker image
      uses: docker/build-push-action@v4
      with:
        context: .
        push: true
        tags: ${{ steps.meta.outputs.tags }}
        labels: ${{ steps.meta.outputs.labels }}
        cache-from: type=gha
        cache-to: type=gha,mode=max
        build-args: |
          NODE_ENV=production
          BUILD_DATE=${{ github.event.head_commit.timestamp }}
          VCS_REF=${{ github.sha }}

10.2.3 CD 파이프라인 (Production)​

# .github/workflows/cd-production.yml
name: Deploy to Production

on:
  workflow_run:
    workflows: ["Continuous Integration"]
    branches: [main]
    types: [completed]

env:
  AWS_REGION: ap-northeast-2
  EKS_CLUSTER_NAME: production-cluster
  K8S_NAMESPACE: production

jobs:
  deploy:
    name: Deploy to Production
    runs-on: ubuntu-latest
    if: ${{ github.event.workflow_run.conclusion == 'success' }}

    environment:
      name: production
      url: https://example.com

    steps:
    - name: Checkout code
      uses: actions/checkout@v3

    - name: Configure AWS credentials
      uses: aws-actions/configure-aws-credentials@v2
      with:
        aws-access-key-id: ${{ secrets.AWS_ACCESS_KEY_ID }}
        aws-secret-access-key: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
        aws-region: ${{ env.AWS_REGION }}

    - name: Login to Amazon ECR
      id: login-ecr
      uses: aws-actions/amazon-ecr-login@v1

    - name: Update kubeconfig
      run: |
        aws eks update-kubeconfig \
          --region ${{ env.AWS_REGION }} \
          --name ${{ env.EKS_CLUSTER_NAME }}

    - name: Deploy to Kubernetes (Canary)
      run: |
        # Canary 배포: 10% 트래픽으로 시작
        kubectl set image deployment/myapp-canary \
          myapp=${{ steps.login-ecr.outputs.registry }}/myapp:${{ github.sha }} \
          -n ${{ env.K8S_NAMESPACE }}

        kubectl rollout status deployment/myapp-canary \
          -n ${{ env.K8S_NAMESPACE }} \
          --timeout=5m

    - name: Run smoke tests
      run: |
        npm run test:smoke -- --env=production-canary

    - name: Monitor canary metrics
      run: |
        # 5분간 메트릭 모니터링
        sleep 300

        # 에러율 체크
        ERROR_RATE=$(kubectl exec -n monitoring deploy/prometheus-server -- \
          promtool query instant \
          'http://localhost:9090' \
          'rate(http_requests_total{status=~"5..",deployment="myapp-canary"}[5m])')

        if (( $(echo "$ERROR_RATE > 0.01" | bc -l) )); then
          echo "Error rate too high: $ERROR_RATE"
          exit 1
        fi

    - name: Promote to full production
      run: |
        # 전체 배포
        kubectl set image deployment/myapp \
          myapp=${{ steps.login-ecr.outputs.registry }}/myapp:${{ github.sha }} \
          -n ${{ env.K8S_NAMESPACE }}

        kubectl rollout status deployment/myapp \
          -n ${{ env.K8S_NAMESPACE }} \
          --timeout=10m

    - name: Verify deployment
      run: |
        kubectl get pods -n ${{ env.K8S_NAMESPACE }} -l app=myapp
        kubectl get service -n ${{ env.K8S_NAMESPACE }} myapp

    - name: Run post-deployment tests
      run: |
        npm run test:e2e -- --env=production

    - name: Notify Slack
      if: always()
      uses: slackapi/slack-github-action@v1
      with:
        webhook-url: ${{ secrets.SLACK_WEBHOOK_URL }}
        payload: |
          {
            "text": "Production Deployment ${{ job.status }}",
            "blocks": [
              {
                "type": "section",
                "text": {
                  "type": "mrkdwn",
                  "text": "*Production Deployment*\nStatus: ${{ job.status }}\nCommit: ${{ github.sha }}\nActor: ${{ github.actor }}"
                }
              }
            ]
          }

  rollback:
    name: Rollback on Failure
    runs-on: ubuntu-latest
    needs: [deploy]
    if: failure()

    steps:
    - name: Rollback deployment
      run: |
        aws eks update-kubeconfig \
          --region ${{ env.AWS_REGION }} \
          --name ${{ env.EKS_CLUSTER_NAME }}

        kubectl rollout undo deployment/myapp \
          -n ${{ env.K8S_NAMESPACE }}

        kubectl rollout status deployment/myapp \
          -n ${{ env.K8S_NAMESPACE }}

    - name: Notify team
      uses: slackapi/slack-github-action@v1
      with:
        webhook-url: ${{ secrets.SLACK_WEBHOOK_URL }}
        payload: |
          {
            "text": ":warning: Production deployment failed and was rolled back",
            "blocks": [
              {
                "type": "section",
                "text": {
                  "type": "mrkdwn",
                  "text": "*ALERT: Rollback Executed*\n Deployment failed: ${{ github.sha }}\nRolled back to previous version"
                }
              }
            ]
          }

# infrastructure/kubernetes/base/prometheus-rules.yml
apiVersion: v1
kind: ConfigMap
metadata:
  name: prometheus-rules
  namespace: monitoring
data:
  alerts.yml: |
    groups:
    - name: application_alerts
      interval: 30s
      rules:
      - alert: HighErrorRate
        expr: |
          (
            rate(http_requests_total{status=~"5.."}[5m])
            /
            rate(http_requests_total[5m])
          ) > 0.05
        for: 5m
        labels:
          severity: critical
        annotations:
          summary: "High error rate detected"
          description: "Error rate is {{ $value | humanizePercentage }} for {{ $labels.instance }}"

      - alert: HighLatency
        expr: |
          histogram_quantile(0.95,
            rate(http_request_duration_seconds_bucket[5m])
          ) > 1
        for: 10m
        labels:
          severity: warning
        annotations:
          summary: "High latency detected"
          description: "95th percentile latency is {{ $value }}s for {{ $labels.instance }}"

      - alert: PodCrashLooping
        expr: |
          rate(kube_pod_container_status_restarts_total[15m]) > 0
        for: 5m
        labels:
          severity: critical
        annotations:
          summary: "Pod is crash looping"
          description: "Pod {{ $labels.pod }} in namespace {{ $labels.namespace }} is crash looping"

      - alert: HighMemoryUsage
        expr: |
          (
            container_memory_usage_bytes{pod=~"myapp-.*"}
            /
            container_spec_memory_limit_bytes{pod=~"myapp-.*"}
          ) > 0.9
        for: 10m
        labels:
          severity: warning
        annotations:
          summary: "High memory usage"
          description: "Memory usage is {{ $value | humanizePercentage }} for {{ $labels.pod }}"

마무리​

DevOps는 단순한 도구나 기술이 아니라 문화이자 철학입니다. 성공적인 DevOps 도입을 위해서는:

  1. 작게 시작하기: 한 번에 모든 것을 바꾸려 하지 마세요
  2. 측정하기: 개선을 측정할 수 있어야 합니다
  3. 실패로부터 배우기: 실패를 두려워하지 말고 학습 기회로 활용하세요
  4. 지속적 개선: DevOps는 여정이지 목적지가 아닙니다
  5. 팀워크: 혼자가 아닌 함께 성장하세요
DevOps의 핵심:
┌────────────────────────────────────┐

│   자동화를 구축하고,                │
│   측정을 통해 개선하고,             │
│   지속적으로 학습한다"              │
