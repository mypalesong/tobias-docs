배포 전략 (Deployment Strategies)

현대적인 배포 전략과 기법을 통해 안전하고 효율적인 서비스 배포를 수행하는 방법을 알아봅니다.

1. 배포 전략 개요​

1.1 배포 전략의 중요성​

배포 전략은 새로운 버전의 애플리케이션을 프로덕션 환경에 안전하게 출시하기 위한 방법론입니다.

주요 목표:

  • 서비스 다운타임 최소화
  • 배포 리스크 감소
  • 빠른 롤백 가능성
  • 사용자 영향 최소화
팁

적절한 배포 전략을 선택하면 서비스 안정성을 유지하면서도 빠른 기능 출시가 가능합니다.

1.2 배포 전략 비교​

전략다운타임리스크복잡도비용
Rolling없음중간낮음낮음
Blue-Green없음낮음중간높음
Canary없음매우 낮음높음중간
A/B Testing없음낮음높음중간

2. Blue-Green 배포​

2.1 개념​

두 개의 동일한 프로덕션 환경(Blue와 Green)을 유지하며, 트래픽을 즉시 전환하는 방식입니다.

2.2.2 배포 스크립트​

#!/bin/bash

# 1. Green 환경에 새 버전 배포
kubectl apply -f green-deployment.yaml

# 2. Green 환경이 준비될 때까지 대기
kubectl rollout status deployment/myapp-green

# 3. Green 환경 헬스체크
for i in {1..10}; do
  STATUS=$(kubectl get pods -l version=green -o jsonpath='{.items[*].status.phase}')
  if [[ "$STATUS" == *"Running"* ]]; then
    echo "Green environment is ready"
    break
  fi
  sleep 10
done

# 4. 서비스를 Green으로 전환
kubectl patch service myapp-service -p '{"spec":{"selector":{"version":"green"}}}'

echo "Traffic switched to Green environment"

# 5. Blue 환경 모니터링 후 제거 (선택사항)
# kubectl delete deployment myapp-blue

2.3 장단점​

장점:

  • 즉시 롤백 가능 (서비스 selector만 변경)
  • 배포 전 충분한 테스트 가능
  • 다운타임 없음

단점:

  • 두 배의 인프라 비용
  • 데이터베이스 스키마 변경 시 복잡도 증가
  • 상태 유지(Stateful) 애플리케이션 처리 어려움
경고

데이터베이스 마이그레이션이 필요한 경우, 하위 호환성을 유지하도록 설계해야 합니다.

3.2.2 점진적 트래픽 증가 스크립트​

#!/bin/bash

# Canary 배포 점진적 증가 스크립트

CANARY_WEIGHTS=(5 10 25 50 75 100)
STABLE_VERSION="v1"
CANARY_VERSION="v2"

for WEIGHT in "${CANARY_WEIGHTS[@]}"; do
  STABLE_WEIGHT=$((100 - WEIGHT))

  echo "Shifting traffic: Stable $STABLE_WEIGHT%, Canary $WEIGHT%"

  kubectl apply -f - <<EOF
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: myapp-virtualservice
spec:
  hosts:
  - myapp-service
  http:
  - route:
    - destination:
        host: myapp-service
        subset: $STABLE_VERSION
      weight: $STABLE_WEIGHT
    - destination:
        host: myapp-service
        subset: $CANARY_VERSION
      weight: $WEIGHT
EOF

  # 메트릭 모니터링 및 검증
  echo "Monitoring metrics for 5 minutes..."
  sleep 300

  # 에러율 체크 (예시)
  ERROR_RATE=$(curl -s "http://prometheus:9090/api/v1/query?query=rate(http_requests_total{status=~\"5..\",version=\"$CANARY_VERSION\"}[5m])" | jq -r '.data.result[0].value[1]')

  if (( $(echo "$ERROR_RATE > 0.01" | bc -l) )); then
    echo "Error rate too high! Rolling back..."
    kubectl apply -f rollback-virtualservice.yaml
    exit 1
  fi

  echo "Metrics look good. Proceeding..."
done

echo "Canary deployment completed successfully!"

kind: PrometheusRule
metadata:
  name: canary-monitoring
spec:
  groups:
  - name: canary

        severity: critical
      annotations:
        summary: "Canary version has high error rate"

        severity: warning
      annotations:
        summary: "Canary version has high latency"

3.4 장단점​

장점:

  • 리스크 최소화 (일부 사용자만 영향)
  • 실제 프로덕션 환경에서 검증
  • 점진적 확대로 안정성 확보

단점:

  • 복잡한 라우팅 로직 필요
  • 모니터링 인프라 필수
  • 배포 시간이 길어질 수 있음
팁

Canary 배포 시 주요 메트릭(에러율, 응답시간, CPU/메모리 사용률)을 실시간으로 모니터링하세요.

// ab-testing-middleware.js
const abTestingMiddleware = (req, res, next) => {
  let variant = req.cookies['ab_test_variant'];

  if (!variant) {
    // 새 사용자에게 그룹 할당
    const randomValue = Math.random();
    variant = randomValue < 0.5 ? 'A' : 'B';

    // Cookie 설정 (30일 유효)
    res.cookie('ab_test_variant', variant, {
      maxAge: 30 * 24 * 60 * 60 * 1000,
      httpOnly: true,
      secure: true
    });
  }

  // 요청 헤더에 추가
  req.headers['x-ab-variant'] = variant;

  // 분석 이벤트 기록
  analytics.track('ab_test_assigned', {
    userId: req.user?.id,
    variant: variant,
    timestamp: new Date().toISOString()
  });

  next();
};

module.exports = abTestingMiddleware;

}

5.4 통계적 유의성 검증​

# ab_test_analysis.py
import numpy as np
from scipy import stats

def analyze_ab_test(variant_a_data, variant_b_data, alpha=0.05):
    """
    A/B 테스트 결과의 통계적 유의성 검증

    Args:
        variant_a_data: Variant A의 전환율 데이터
        variant_b_data: Variant B의 전환율 데이터
        alpha: 유의수준 (기본값: 0.05)

    Returns:
        dict: 분석 결과
    """

    # 기본 통계
    n_a = len(variant_a_data)
    n_b = len(variant_b_data)
    conversions_a = sum(variant_a_data)
    conversions_b = sum(variant_b_data)

    rate_a = conversions_a / n_a
    rate_b = conversions_b / n_b

    # Z-test 수행
    pooled_rate = (conversions_a + conversions_b) / (n_a + n_b)
    se = np.sqrt(pooled_rate * (1 - pooled_rate) * (1/n_a + 1/n_b))
    z_score = (rate_b - rate_a) / se
    p_value = 2 * (1 - stats.norm.cdf(abs(z_score)))

    # 신뢰구간 계산
    ci_a = stats.norm.interval(1-alpha, loc=rate_a,
                                scale=np.sqrt(rate_a*(1-rate_a)/n_a))
    ci_b = stats.norm.interval(1-alpha, loc=rate_b,
                                scale=np.sqrt(rate_b*(1-rate_b)/n_b))

    # 결과 정리
    result = {
        'variant_a': {
            'sample_size': n_a,
            'conversions': conversions_a,
            'rate': rate_a,
            'confidence_interval': ci_a
        },
        'variant_b': {
            'sample_size': n_b,
            'conversions': conversions_b,
            'rate': rate_b,
            'confidence_interval': ci_b
        },
        'z_score': z_score,
        'p_value': p_value,
        'is_significant': p_value < alpha,
        'winner': 'B' if rate_b > rate_a and p_value < alpha else 'A' if rate_a > rate_b and p_value < alpha else 'Inconclusive'
    }

    return result

# 사용 예시
if __name__ == '__main__':
    # 예시 데이터
    variant_a = np.random.binomial(1, 0.12, 1000)  # 12% 전환율
    variant_b = np.random.binomial(1, 0.15, 1000)  # 15% 전환율

    results = analyze_ab_test(variant_a, variant_b)

    print(f"Variant A: {results['variant_a']['rate']:.2%}")
    print(f"Variant B: {results['variant_b']['rate']:.2%}")
    print(f"P-value: {results['p_value']:.4f}")
    print(f"Winner: {results['winner']}")
팁

A/B 테스트는 충분한 샘플 크기와 통계적 유의성이 확보될 때까지 진행해야 합니다.

6. Feature Flags​

6.1 개념​

코드는 배포하되 기능 활성화를 런타임에 제어하는 방식입니다.

6.2 구현 예제​

6.2.1 LaunchDarkly 스타일 구현​

// feature-flags-service.js
class FeatureFlagService {
  constructor(config) {
    this.flags = new Map();
    this.config = config;
    this.initializeFlags();
  }

  async initializeFlags() {
    // 원격 설정 서버에서 플래그 가져오기
    const response = await fetch(`${this.config.apiUrl}/flags`);
    const flags = await response.json();

    flags.forEach(flag => {
      this.flags.set(flag.key, flag);
    });

    // 실시간 업데이트 구독
    this.subscribeToUpdates();
  }

  isEnabled(flagKey, context = {}) {
    const flag = this.flags.get(flagKey);

    if (!flag) {
      return false;
    }

    // 전역 활성화 상태
    if (flag.enabled === false) {
      return false;
    }

    // 타겟팅 규칙 평가
    if (flag.targets && flag.targets.length > 0) {
      return this.evaluateTargets(flag.targets, context);
    }

    // 롤아웃 비율 평가
    if (flag.rollout) {
      return this.evaluateRollout(flag.rollout, context);
    }

    return flag.enabled;
  }

  evaluateTargets(targets, context) {
    for (const target of targets) {
      if (this.matchTarget(target, context)) {
        return target.enabled;
      }
    }
    return false;
  }

  matchTarget(target, context) {
    const { attribute, operator, values } = target;
    const userValue = context[attribute];

    switch (operator) {
      case 'in':
        return values.includes(userValue);
      case 'not_in':
        return !values.includes(userValue);
      case 'contains':
        return values.some(v => userValue?.includes(v));
      case 'matches':
        return values.some(v => new RegExp(v).test(userValue));
      default:
        return false;
    }
  }

  evaluateRollout(rollout, context) {
    // 일관된 해싱으로 사용자별 결정
    const userId = context.userId || context.sessionId;
    const hash = this.hashCode(userId);
    const percentage = (hash % 100) / 100;

    return percentage < rollout.percentage;
  }

  hashCode(str) {
    let hash = 0;
    for (let i = 0; i < str.length; i++) {
      const char = str.charCodeAt(i);
      hash = ((hash << 5) - hash) + char;
      hash = hash & hash;
    }
    return Math.abs(hash);
  }

  subscribeToUpdates() {
    const ws = new WebSocket(`${this.config.wsUrl}/flags-stream`);

    ws.onmessage = (event) => {
      const update = JSON.parse(event.data);
      this.flags.set(update.key, update);
    };
  }
}

module.exports = FeatureFlagService;

# 롤백 스크립트

NAMESPACE="production"

# 1. 현재 배포 상태 확인
echo "Current deployment status:"
kubectl get deployment $DEPLOYMENT -n $NAMESPACE

# 2. 배포 히스토리 조회
echo -e "\nDeployment history:"
kubectl rollout history deployment/$DEPLOYMENT -n $NAMESPACE

# 3. 사용자에게 리비전 선택 요청
read -p "Enter revision number to rollback to (or 'previous' for last): " REVISION

# 4. 롤백 실행
if [ "$REVISION" = "previous" ]; then
  echo "Rolling back to previous revision..."
  kubectl rollout undo deployment/$DEPLOYMENT -n $NAMESPACE
else
  echo "Rolling back to revision $REVISION..."
  kubectl rollout undo deployment/$DEPLOYMENT -n $NAMESPACE --to-revision=$REVISION
fi

# 5. 롤백 진행 상태 모니터링
echo -e "\nMonitoring rollback progress..."
kubectl rollout status deployment/$DEPLOYMENT -n $NAMESPACE

# 6. 롤백 후 상태 확인
echo -e "\nVerifying rollback..."
kubectl get pods -n $NAMESPACE -l app=$DEPLOYMENT

# 7. 헬스체크
echo -e "\nRunning health checks..."
for i in {1..5}; do
  HEALTHY=$(kubectl get pods -n $NAMESPACE -l app=$DEPLOYMENT -o jsonpath='{.items[*].status.conditions[?(@.type=="Ready")].status}' | grep -o "True" | wc -l)
  TOTAL=$(kubectl get pods -n $NAMESPACE -l app=$DEPLOYMENT --no-headers | wc -l)

  echo "Healthy pods: $HEALTHY/$TOTAL"

  if [ $HEALTHY -eq $TOTAL ]; then
    echo "All pods are healthy!"
    break
  fi

  sleep 10
done

# 8. 롤백 완료 알림
echo -e "\n=== Rollback completed ==="
kubectl describe deployment $DEPLOYMENT -n $NAMESPACE | grep -A 5 "Events"

// database-migration-rollback.js
const { Migration } = require('./migration-framework');

class DatabaseRollbackManager {
  constructor(db) {
    this.db = db;
  }

  async createBackup(version) {
    const timestamp = new Date().toISOString().replace(/[:.]/g, '-');
    const backupName = `backup_v${version}_${timestamp}`;

    console.log(`Creating backup: ${backupName}`);

    await this.db.query(`
      CREATE DATABASE ${backupName}
      TEMPLATE current_database;
    `);

    return backupName;
  }

  async rollbackMigration(targetVersion) {
    const currentVersion = await this.getCurrentVersion();

    console.log(`Rolling back from v${currentVersion} to v${targetVersion}`);

    // 백업 생성
    const backup = await this.createBackup(currentVersion);

    try {
      // 롤백할 마이그레이션 목록 가져오기
      const migrations = await this.getMigrationsToRollback(
        currentVersion,
        targetVersion
      );

      // 역순으로 롤백 실행
      for (const migration of migrations.reverse()) {
        console.log(`Rolling back migration: ${migration.name}`);

        await this.db.transaction(async (tx) => {
          await migration.down(tx);
          await this.updateMigrationVersion(migration.version - 1, tx);
        });
      }

      console.log('Rollback completed successfully');
      return { success: true, backup };

    } catch (error) {
      console.error('Rollback failed:', error);

      // 백업에서 복구
      console.log('Restoring from backup...');
      await this.restoreFromBackup(backup);

      throw error;
    }
  }

  async getMigrationsToRollback(currentVersion, targetVersion) {
    const migrations = [];

    for (let v = currentVersion; v > targetVersion; v--) {
      const migration = await Migration.findByVersion(v);
      if (migration) {
        migrations.push(migration);
      }
    }

    return migrations;
  }

  async getCurrentVersion() {
    const result = await this.db.query(
      'SELECT version FROM schema_migrations ORDER BY version DESC LIMIT 1'
    );
    return result.rows[0]?.version || 0;
  }

  async updateMigrationVersion(version, tx) {
    await tx.query(
      'DELETE FROM schema_migrations WHERE version = $1',
      [version + 1]
    );
  }

  async restoreFromBackup(backupName) {
    await this.db.query(`
      SELECT pg_terminate_backend(pid)
      FROM pg_stat_activity
      WHERE datname = current_database() AND pid <> pg_backend_pid();
    `);

    await this.db.query(`
      DROP DATABASE current_database;
      ALTER DATABASE ${backupName} RENAME TO current_database;
    `);
  }
}

module.exports = DatabaseRollbackManager;

8. Zero-Downtime 배포​

8.1 설계 원칙​

Zero-Downtime 배포를 위한 핵심 원칙:

  1. 하위 호환성 유지
  2. Graceful Shutdown
  3. 헬스체크 활용
  4. Connection Draining

8.2 Graceful Shutdown 구현​

// graceful-shutdown.js
const express = require('express');
const app = express();

// 활성 연결 추적
let activeConnections = 0;
let isShuttingDown = false;

// 연결 추적 미들웨어
app.use((req, res, next) => {
  if (isShuttingDown) {
    res.set('Connection', 'close');
    return res.status(503).json({
      error: 'Server is shutting down'
    });
  }

  activeConnections++;

  res.on('finish', () => {
    activeConnections--;
  });

  next();
});

// 헬스체크 엔드포인트
app.get('/health/live', (req, res) => {
  res.status(200).json({ status: 'ok' });
});

app.get('/health/ready', (req, res) => {
  if (isShuttingDown) {
    return res.status(503).json({
      status: 'not ready',
      reason: 'shutting down'
    });
  }

  // 데이터베이스 연결 등 확인
  const isReady = checkDependencies();

  if (isReady) {
    res.status(200).json({ status: 'ready' });
  } else {
    res.status(503).json({ status: 'not ready' });
  }
});

const server = app.listen(3000, () => {
  console.log('Server started on port 3000');
});

// Graceful shutdown 핸들러
const gracefulShutdown = async (signal) => {
  console.log(`\n${signal} received, starting graceful shutdown...`);

  isShuttingDown = true;

  // 새로운 연결 거부
  server.close(() => {
    console.log('HTTP server closed');
  });

  // 활성 연결이 종료될 때까지 대기
  const maxWaitTime = 30000; // 30초
  const startTime = Date.now();

  while (activeConnections > 0 && Date.now() - startTime < maxWaitTime) {
    console.log(`Waiting for ${activeConnections} active connections...`);
    await new Promise(resolve => setTimeout(resolve, 1000));
  }

  if (activeConnections > 0) {
    console.log(`Force closing ${activeConnections} connections`);
  }

  // 데이터베이스 연결 정리
  await closeDatabase();

  // 기타 리소스 정리
  await cleanupResources();

  console.log('Graceful shutdown completed');
  process.exit(0);
};

// 시그널 핸들러 등록
process.on('SIGTERM', () => gracefulShutdown('SIGTERM'));
process.on('SIGINT', () => gracefulShutdown('SIGINT'));

// Unhandled rejection 처리
process.on('unhandledRejection', (reason, promise) => {
  console.error('Unhandled Rejection at:', promise, 'reason:', reason);
  gracefulShutdown('unhandledRejection');
});

// backward-compatible-migration.js

// ❌ 잘못된 방법: 즉시 컬럼 삭제
class BadMigration {
  async up(db) {
    await db.schema.table('users', (table) => {
      table.dropColumn('old_email');
    });
  }
}

// ✅ 올바른 방법: 3단계 마이그레이션

// Step 1: 새 컬럼 추가 (v1.1 배포)
class AddNewColumnMigration {
  async up(db) {
    await db.schema.table('users', (table) => {
      table.string('email_address').nullable();
    });

    // 기존 데이터 복사
    await db.raw(`
      UPDATE users
      SET email_address = old_email
      WHERE email_address IS NULL
    `);
  }

  async down(db) {
    await db.schema.table('users', (table) => {
      table.dropColumn('email_address');
    });
  }
}

// Step 2: 애플리케이션 코드 업데이트 (v1.2 배포)
// - 새 컬럼(email_address)을 읽기/쓰기
// - 호환성을 위해 old_email도 계속 업데이트

class UserModel {
  async save() {
    await db('users').update({
      email_address: this.email,  // 새 컬럼
      old_email: this.email        // 하위 호환성
    });
  }

  async load() {
    const user = await db('users').first();
    this.email = user.email_address || user.old_email;
  }
}

// Step 3: 오래된 컬럼 제거 (v1.3 배포)
// - 충분한 시간이 지난 후 (예: 2주)
class RemoveOldColumnMigration {
  async up(db) {
    await db.schema.table('users', (table) => {
      table.dropColumn('old_email');
    });
  }

  async down(db) {
    await db.schema.table('users', (table) => {
      table.string('old_email');
    });
  }
}

  only:
    - main

apiVersion: v1

  name: alertmanager-config

  alertmanager.yml: |

      slack_api_url: 'https://hooks.slack.com/services/YOUR/SLACK/WEBHOOK'

    route:
      group_by: ['alertname', 'cluster', 'service']
      group_wait: 10s
      group_interval: 10s
      repeat_interval: 12h
      receiver: 'slack-notifications'
      routes:
      - match:
          severity: critical
        receiver: 'pagerduty-critical'
      - match:
          severity: warning
        receiver: 'slack-notifications'

    receivers:
    - name: 'slack-notifications'
      slack_configs:
      - channel: '#deployments'
        title: 'Deployment Alert'
        text: '{{ range .Alerts }}{{ .Annotations.description }}{{ end }}'

    - name: 'pagerduty-critical'
      pagerduty_configs:
      - service_key: 'YOUR_PAGERDUTY_KEY'
        description: '{{ .CommonAnnotations.summary }}'

10. 모범 사례 및 권장사항​

10.1 배포 전략 선택 가이드​

Rolling 배포를 선택하는 경우:

  • 리소스가 제한적인 환경
  • 상태 비저장(Stateless) 애플리케이션
  • 빠른 배포가 필요한 경우

Blue-Green 배포를 선택하는 경우:

  • 즉시 롤백이 중요한 경우
  • 충분한 인프라 리소스가 있는 경우
  • 데이터베이스 변경이 없거나 최소한인 경우

Canary 배포를 선택하는 경우:

  • 고위험 변경사항
  • 대규모 사용자 기반
  • 점진적 검증이 필요한 경우

A/B 테스팅을 선택하는 경우:

  • UI/UX 변경
  • 비즈니스 로직 실험
  • 데이터 기반 의사결정이 필요한 경우

10.2 공통 권장사항​

# best-practices.yaml
deployment_best_practices:
  always_do:
    - "모든 배포에 대해 롤백 계획 수립"
    - "배포 전 충분한 테스트 수행"
    - "메트릭 및 로그 모니터링"
    - "Graceful shutdown 구현"
    - "헬스체크 엔드포인트 제공"
    - "버전 관리 및 태깅"
    - "배포 자동화"

  never_do:
    - "프로덕션에 직접 배포"
    - "테스트 없이 배포"
    - "피크 시간대 배포"
    - "백업 없는 데이터베이스 변경"
    - "모니터링 없는 배포"
    - "하드코딩된 설정값"

  monitoring_metrics:
    - name: "에러율"
      threshold: "< 1%"
      alert: "critical"
    - name: "응답시간 (p95)"
      threshold: "< 500ms"
      alert: "warning"
    - name: "CPU 사용률"
      threshold: "< 70%"
      alert: "warning"
    - name: "메모리 사용률"
      threshold: "< 80%"
      alert: "warning"
    - name: "요청 처리량"
      threshold: "정상 범위 ±20%"
      alert: "info"
팁

배포는 기술적인 프로세스이지만, 팀 커뮤니케이션과 협업이 성공의 핵심입니다.

요약​

현대적인 배포 전략은 서비스의 안정성과 사용자 경험을 보장하면서도 빠른 기능 출시를 가능하게 합니다:

  1. Rolling 배포: 점진적 인스턴스 교체, 낮은 비용
  2. Blue-Green 배포: 즉시 전환 및 롤백, 높은 안정성
  3. Canary 배포: 점진적 트래픽 증가, 리스크 최소화
  4. A/B 테스팅: 데이터 기반 의사결정, 실험적 배포
  5. Feature Flags: 런타임 기능 제어, 유연한 출시
  6. Zero-Downtime 배포: 무중단 서비스, 사용자 경험 보장

각 전략의 장단점을 이해하고, 서비스 특성에 맞는 적절한 전략을 선택하여 안전하고 효율적인 배포를 수행하세요.
