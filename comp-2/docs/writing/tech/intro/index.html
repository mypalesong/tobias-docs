1. 테크니컬 라이팅 소개

테크니컬 라이팅은 기술적인 정보를 명확하고 정확하게 전달하는 전문적인 글쓰기입니다. 개발자, 엔지니어, 기술 문서 작성자에게 필수적인 스킬입니다.

1.1 테크니컬 라이팅이란?​

복잡한 기술 정보를 사용자가 쉽게 이해하고 활용할 수 있도록 작성하는 것입니다.

주요 문서 유형​

  • API 문서: 개발자를 위한 API 사용 가이드
  • 사용자 가이드: 제품 사용 방법 설명서
  • 튜토리얼: 단계별 학습 가이드
  • README: 프로젝트 소개 및 시작 가이드
  • 기술 사양서: 시스템 구조 및 설계 문서

1.2 테크니컬 라이팅의 특징​

1. 명확성 (Clarity)​

모호한 표현을 피하고 정확하게 작성합니다.

❌ 나쁜 예: "이 함수는 빠릅니다."
✅ 좋은 예: "이 함수는 O(log n)의 시간 복잡도를 가집니다."

2. 정확성 (Accuracy)​

기술적으로 올바른 정보를 제공합니다.

❌ 나쁜 예: "JSON은 프로그래밍 언어입니다."
✅ 좋은 예: "JSON은 데이터 교환 형식입니다."

3. 간결성 (Conciseness)​

필요한 정보만 포함하고 불필요한 내용은 제거합니다.

❌ 나쁜 예: "이 함수는 매우 유용한 기능을 제공하며,
           여러 상황에서 사용할 수 있고..."
✅ 좋은 예: "사용자 인증을 처리합니다."

4. 일관성 (Consistency)​

용어, 형식, 스타일을 일관되게 유지합니다.

✅ 통일된 용어 사용:
   - "사용자" (전체적으로) ❌ "사용자" / "유저" 혼용
   - "데이터베이스" (전체적으로) ❌ "DB" / "데이터베이스" 혼용

1.3 좋은 기술 문서의 조건​

✅ 사용자 중심​

독자가 누구인지 파악하고 그들의 수준에 맞춰 작성합니다.

독자접근 방식
초보자기본 개념부터 설명, 많은 예제
중급자핵심만 설명, 고급 기능 소개
전문가레퍼런스 형식, 상세 사양

✅ 실용적​

실제로 사용할 수 있는 예제와 코드를 제공합니다.

# ✅ 좋은 예: 실제로 작동하는 코드
def calculate_total(items):
    """
    장바구니 항목의 총 금액을 계산합니다.

    Args:
        items (list): 가격이 포함된 항목 리스트

    Returns:
        float: 총 금액

    Example:
        >>> items = [{"price": 1000}, {"price": 2000}]
        >>> calculate_total(items)
        3000
    """
    return sum(item["price"] for item in items)

✅ 구조화​

논리적인 순서로 정보를 배치합니다.

1. 개요 (무엇을 하는가?)
2. 사전 요구사항 (무엇이 필요한가?)
3. 설치/설정 (어떻게 시작하는가?)
4. 사용 방법 (어떻게 사용하는가?)
5. 고급 기능 (더 많은 것을 할 수 있는가?)
6. 문제 해결 (문제가 생기면?)
7. 레퍼런스 (자세한 정보는?)

✅ 검색 가능​

사용자가 필요한 정보를 빠르게 찾을 수 있어야 합니다.

  • 명확한 제목
  • 목차 제공
  • 키워드 강조
  • 검색 기능

1.4 테크니컬 라이팅 프로세스​

1단계: 리서치​

- 대상 독자 파악
- 기술 스펙 확인
- 기존 문서 검토
- SME(Subject Matter Expert)와 논의

2단계: 기획​

- 문서 구조 설계
- 포함할 내용 결정
- 예제 준비
- 스타일 가이드 확인

3단계: 작성​

- 초안 작성
- 코드 예제 작성 및 테스트
- 다이어그램 생성
- 스크린샷 캡처

4단계: 검토​

- 기술적 정확성 확인
- 오타/문법 수정
- 가독성 개선
- 예제 테스트

5단계: 발행​

- 버전 관리
- 배포
- 피드백 수집
- 정기 업데이트

1.5 핵심 원칙​

능동태 사용​

❌ 수동태: "데이터가 데이터베이스에 저장됩니다."
✅ 능동태: "데이터베이스에 데이터를 저장합니다."

현재 시제 사용​

❌ 미래: "이 함수는 결과를 반환할 것입니다."
✅ 현재: "이 함수는 결과를 반환합니다."

2인칭 사용​

❌ 3인칭: "개발자는 다음 명령어를 실행해야 합니다."
✅ 2인칭: "다음 명령어를 실행하세요."

긍정적 표현​

❌ 부정: "이 방법을 사용하지 마세요."
✅ 긍정: "대신 이 방법을 사용하세요."

1.6 도구와 리소스​

문서 작성 도구​

  • Markdown: 간단한 문서
  • Docusaurus: 문서 사이트 생성
  • Sphinx: Python 문서
  • GitBook: 인터랙티브 문서
  • Notion: 협업 문서

다이어그램 도구​

  • Mermaid: 코드로 다이어그램 생성
  • Draw.io: 플로우차트, 아키텍처
  • PlantUML: UML 다이어그램
  • Excalidraw: 손그림 스타일

스타일 가이드​

  • Google Developer Documentation Style Guide
  • Microsoft Writing Style Guide
  • Apple Style Guide
  • Write the Docs

1.7 흔한 실수​

❌ 전문 용어 남용​

나쁜 예: "이 API는 RESTful하며 OAuth 2.0을 통한
         인증을 거쳐 JSON 페이로드를 반환합니다."

좋은 예: "이 API는 표준 REST 방식으로 작동합니다.
         로그인 후 JSON 형식으로 데이터를 받습니다."

❌ 설명 없는 코드​

나쁜 예:
```bash
npm install && npm run build

좋은 예: 패키지를 설치하고 프로젝트를 빌드합니다:

npm install  # 의존성 설치
npm run build  # 프로덕션 빌드

### ❌ 최신 정보 미반영
문서는 살아있는 것입니다. 정기적으로 업데이트하세요.

### ❌ 독자 가정
```markdown
나쁜 예: "당연히 Docker를 알고 계시죠?"
좋은 예: "Docker가 설치되어 있어야 합니다.
         [설치 가이드](#) 참고"

1.8 실전 팁​

실제로 테스트하세요

작성한 모든 명령어와 코드를 직접 실행해보세요.

피드백을 받으세요

실제 사용자에게 문서를 읽어보게 하고 어려운 부분을 찾으세요.

버전을 명시하세요

소프트웨어 버전에 따라 내용이 다를 수 있습니다.

이 문서는 Node.js v18.x 기준으로 작성되었습니다.

1.9 다음 단계​

테크니컬 라이팅의 기본을 이해했다면:

  • API 문서 작성: 개발자를 위한 API 문서 작성법
  • 튜토리얼 작성: 효과적인 학습 가이드 만들기
  • README 작성: 프로젝트 문서화 완벽 가이드

좋은 문서는 제품의 성공을 좌우합니다!
