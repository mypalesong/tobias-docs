Git Tags

태그를 사용하여 중요한 버전을 표시하고 관리하는 방법을 배웁니다.

Tag란?​

Tag는 특정 커밋에 붙이는 레이블입니다. 주로 릴리스 버전을 표시하는 데 사용됩니다.

Tag 종류​

Lightweight Tag​

단순히 커밋을 가리키는 포인터입니다.

# Lightweight tag 생성
git tag v1.0.0

# 특정 커밋에 태그
git tag v1.0.0 abc123

Annotated Tag (권장)​

추가 정보(작성자, 날짜, 메시지)를 포함하는 완전한 Git 객체입니다.

# Annotated tag 생성
git tag -a v1.0.0 -m "Release version 1.0.0"

# 특정 커밋에 annotated tag
git tag -a v1.0.0 abc123 -m "Release 1.0.0"

Tag 관리​

Tag 목록 보기​

# 모든 태그 목록
git tag

# 패턴으로 필터링
git tag -l "v1.*"
git tag -l "v2.0.*"

# 태그 상세 정보
git show v1.0.0

# Annotated tag 정보만
git tag -n

Tag 생성​

# 현재 커밋에 annotated tag
git tag -a v1.0.0 -m "First stable release"

# Lightweight tag
git tag v1.0.0

# 과거 커밋에 태그
git tag -a v0.9.0 abc123 -m "Beta release"

# 서명된 태그 (GPG)
git tag -s v1.0.0 -m "Signed release"

Tag 푸시​

# 특정 태그 푸시
git push origin v1.0.0

# 모든 태그 푸시
git push origin --tags

# Annotated tag만 푸시
git push --follow-tags
팁

git push는 기본적으로 태그를 푸시하지 않습니다. 명시적으로 푸시해야 합니다.

Tag 삭제​

# 로컬 태그 삭제
git tag -d v1.0.0

# 원격 태그 삭제
git push origin --delete v1.0.0

# 또는
git push origin :refs/tags/v1.0.0

Tag 체크아웃​

# 태그로 체크아웃 (detached HEAD)
git checkout v1.0.0

# 태그에서 새 브랜치 생성
git checkout -b hotfix/v1.0.1 v1.0.0

Semantic Versioning (SemVer)​

권장되는 버전 명명 규칙입니다: MAJOR.MINOR.PATCH

v1.2.3
│ │ │
│ │ └─ PATCH: 버그 수정
│ └─── MINOR: 기능 추가 (하위 호환)
└───── MAJOR: Breaking changes

예시​

# 초기 릴리스
git tag -a v1.0.0 -m "Initial release"

# 버그 수정
git tag -a v1.0.1 -m "Fix login bug"

# 새 기능 추가
git tag -a v1.1.0 -m "Add user profile feature"

# Breaking change
git tag -a v2.0.0 -m "Redesigned API (breaking)"

실습 예제​

예제 1: 첫 릴리스​

# 1. 릴리스 준비
git checkout main
git pull

# 2. 버전 업데이트
echo '{"version": "1.0.0"}' > package.json
git add package.json
git commit -m "chore: bump version to 1.0.0"

# 3. 태그 생성
git tag -a v1.0.0 -m "Release version 1.0.0

Features:
- User authentication
- Dashboard
- Profile management"

# 4. 푸시
git push origin main
git push origin v1.0.0

예제 2: Hotfix 릴리스​

# 1. Hotfix 브랜치 생성
git checkout -b hotfix/v1.0.1 v1.0.0

# 2. 버그 수정
# ... 수정 ...
git commit -am "fix: resolve critical security issue"

# 3. Main에 병합
git checkout main
git merge hotfix/v1.0.1

# 4. 태그 생성
git tag -a v1.0.1 -m "Hotfix: Security patch"

# 5. 푸시
git push origin main
git push origin v1.0.1

# 6. Hotfix 브랜치 삭제
git branch -d hotfix/v1.0.1

예제 3: Pre-release 태그​

# Alpha
git tag -a v2.0.0-alpha.1 -m "Alpha release"

# Beta
git tag -a v2.0.0-beta.1 -m "Beta release"

# Release Candidate
git tag -a v2.0.0-rc.1 -m "Release candidate 1"

# Final
git tag -a v2.0.0 -m "Final release"

GitHub Releases​

웹 UI에서​

  1. Releases 탭 클릭
  2. "Create a new release" 클릭
  3. 태그 선택 또는 생성
  4. 릴리스 노트 작성
  5. 파일 첨부 (선택)
  6. "Publish release" 클릭

CLI로​

# GitHub Release 생성
gh release create v1.0.0 \
  --title "Version 1.0.0" \
  --notes "## What's New
- Feature A
- Feature B

## Bug Fixes
- Fixed issue #123" \
  dist/*.zip

# Pre-release
gh release create v2.0.0-beta.1 --prerelease

# Draft release
gh release create v1.1.0 --draft

Release Notes 작성​

좋은 Release Notes​

# Version 1.2.0

## 🎉 New Features
- **Authentication**: Added OAuth2 support (#123)
- **Dashboard**: New analytics widget (#124)
- **API**: GraphQL endpoint (#125)

## 🐛 Bug Fixes
- Fixed login redirect issue (#130)
- Resolved memory leak in dashboard (#131)

## 🔄 Changes
- Updated dependencies
- Improved performance by 30%

## ⚠️ Breaking Changes
- `oldFunction()` is deprecated, use `newFunction()` instead

## 📚 Documentation
- Added API reference
- Updated installation guide

## 🙏 Contributors
Thanks to @user1, @user2, @user3

자동 생성​

# GitHub CLI로 자동 생성
gh release create v1.2.0 --generate-notes

# 또는 수동으로 커밋 로그 사용
git log v1.1.0..v1.2.0 --pretty=format:"- %s (%h)" --reverse

Tag 검색 및 비교​

태그 간 차이 확인​

# 두 태그 간 변경사항
git diff v1.0.0 v1.1.0

# 커밋 로그
git log v1.0.0..v1.1.0 --oneline

# 파일 변경 통계
git diff --stat v1.0.0 v1.1.0

# 특정 파일의 차이
git diff v1.0.0 v1.1.0 -- src/app.js

태그가 포함된 브랜치 찾기​

# 태그를 포함하는 브랜치
git branch --contains v1.0.0

# 태그가 속한 릴리스
git describe v1.0.0

고급 사용법​

Git Describe​

현재 위치를 가장 가까운 태그로 설명:

# 현재 커밋 설명
git describe
# 출력: v1.0.0-5-g1234abc
#       태그-커밋수-해시

# 항상 태그 표시
git describe --tags

# 정확한 매치만
git describe --exact-match

Signed Tags (GPG)​

# GPG 키 생성
gpg --gen-key

# Git에 키 설정
git config --global user.signingkey YOUR_KEY_ID

# Signed tag 생성
git tag -s v1.0.0 -m "Signed release"

# 서명 확인
git tag -v v1.0.0

Tag Hooks​

# .git/hooks/pre-push
#!/bin/sh

# 태그 푸시 시 검증
if [ -n "$(git rev-list --tags --max-count=1)" ]; then
    echo "Running tests before tag push..."
    npm test || exit 1
fi

자동화​

CI/CD에서 태그 사용​

# .github/workflows/release.yml
name: Release

on:
  push:
    tags:
      - 'v*'

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3

      - name: Get version
        run: echo "VERSION=${GITHUB_REF#refs/tags/}" >> $GITHUB_ENV

      - name: Build
        run: npm run build

      - name: Create Release
        uses: actions/create-release@v1
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
        with:
          tag_name: ${{ env.VERSION }}
          release_name: Release ${{ env.VERSION }}

자동 버전 범프​

# package.json 사용
npm version patch  # 1.0.0 -> 1.0.1
npm version minor  # 1.0.0 -> 1.1.0
npm version major  # 1.0.0 -> 2.0.0

# 자동으로 커밋 및 태그 생성됨
git push --follow-tags

Best Practices​

1. Annotated Tags 사용​

# ✅ 좋은 예
git tag -a v1.0.0 -m "Release 1.0.0"

# ❌ 나쁜 예
git tag v1.0.0

2. Semantic Versioning 준수​

# ✅ 좋은 예
v1.0.0, v1.1.0, v2.0.0

# ❌ 나쁜 예
release-1, final, production

3. 태그에 의미 있는 메시지​

# ✅ 좋은 예
git tag -a v1.0.0 -m "First stable release with user auth and dashboard"

# ❌ 나쁜 예
git tag -a v1.0.0 -m "version 1"

4. 태그 전에 테스트​

# 테스트 실행
npm test

# 빌드 확인
npm run build

# 그 다음 태그
git tag -a v1.0.0 -m "Release 1.0.0"

일반적인 실수​

1. ❌ 잘못된 태그 푸시​

# 문제
git push origin v1.0.0  # 잘못된 태그

# 해결
git tag -d v1.0.0  # 로컬 삭제
git push origin --delete v1.0.0  # 원격 삭제
git tag -a v1.0.0 -m "Correct tag"  # 재생성
git push origin v1.0.0

2. ❌ 태그 이동 불가​

# 태그는 이동할 수 없음
# 삭제 후 재생성 필요
git tag -d v1.0.0
git tag -a v1.0.0 abc123 -m "Moved tag"
git push origin :refs/tags/v1.0.0
git push origin v1.0.0
