시나리오 9: 레거시 프로젝트 리팩토링

준호가 5년된 오래된 코드베이스를 현대적인 구조로 개선합니다.

📌 상황 설명​

회사의 핵심 서비스가 5년 전에 만들어진 레거시 코드로 돌아가고 있습니다. 기술 부채가 쌓여 새 기능 추가가 어렵고, 버그도 자주 발생합니다. 준호가 점진적 리팩토링 프로젝트를 시작합니다.

🎯 목표​

  • 안전한 점진적 리팩토링 전략
  • Git Bisect로 버그 도입 시점 찾기
  • Interactive Rebase로 커밋 정리
  • 대규모 변경 시 리스크 관리
  • 테스트 주도 리팩토링

💬 대화​

준호: "이 코드는 5년 전 제가 신입일 때 작성한 건데... 지금 보니 부끄럽네요 😅"

민지: "그래도 5년 동안 잘 작동했잖아요!"

준호: "맞아요. 하지만 이제는 개선할 때가 됐어요. 단, 안전하게 해야 합니다."

⚡ 실습​

Week 1: 현황 파악 및 계획​

1단계: 코드베이스 분석​

# 코드 통계
cloc src/
# JavaScript: 15,000 lines
# 평균 파일 크기: 500 lines (너무 큼!)

# 복잡도 분석
npx complexity-report src/

# 가장 많이 변경된 파일 찾기
git log --all -M -C --name-only --format='format:' | sort | uniq -c | sort -n -r | head -20

# 특정 파일의 변경 히스토리
git log --follow --oneline -- src/legacy-service.js

2단계: 리팩토링 계획​

# Legacy Refactoring Roadmap

## Phase 1: 테스트 커버리지 (Week 1-2)
- [ ] E2E 테스트 추가
- [ ] 핵심 비즈니스 로직 단위 테스트
- [ ] 목표: 80% 커버리지

## Phase 2: 작은 개선 (Week 3-4)
- [ ] ESLint/Prettier 적용
- [ ] 함수 분리 (500줄 → 50줄 이하)
- [ ] 매직 넘버를 상수로

## Phase 3: 구조 개선 (Week 5-8)
- [ ] 레이어 분리 (Controller, Service, Repository)
- [ ] 의존성 주입 패턴 적용
- [ ] 타입스크립트 마이그레이션

## Phase 4: 성능 최적화 (Week 9-10)
- [ ] 느린 쿼리 최적화
- [ ] 캐싱 추가
- [ ] 불필요한 API 호출 제거

Week 2: Git Bisect로 버그 찾기​

문제: 언제부턴가 특정 기능이 느려졌는데, 어느 커밋부터인지 모름

3단계: Git Bisect 사용​

# Bisect 시작
git bisect start

# 현재 버전은 느림 (bad)
git bisect bad

# 6개월 전 버전은 빠름 (good)
git checkout v1.5.0
npm test  # 확인
git bisect good v1.5.0

# Git이 중간 커밋으로 이동
# Bisecting: 150 revisions left to test
git checkout <commit>

# 테스트
npm run perf-test

# 느리면
git bisect bad

# 빠르면
git bisect good

# 반복... 결국 문제 커밋 찾음!
# abc1234 is the first bad commit
# commit abc1234
# feat: add user activity logging

자동화된 Bisect:

# 테스트 스크립트 작성
cat > test-performance.sh << 'EOF'
#!/bin/bash
npm run perf-test
if [ $? -eq 0 ]; then
  exit 0  # good
else
  exit 1  # bad
fi
EOF

chmod +x test-performance.sh

# 자동 bisect
git bisect start HEAD v1.5.0
git bisect run ./test-performance.sh

# Git이 자동으로 문제 커밋을 찾음!

Week 3-4: 점진적 리팩토링​

4단계: Strangler Fig 패턴​

// legacy-service.js (기존 코드 - 500줄)
// 한 번에 다 바꾸지 않고, 새 코드와 공존

// 1단계: 새 서비스 생성
// services/user-service-v2.js
export class UserServiceV2 {
  async getUser(id) {
    // 새로운 깔끔한 구현
  }
}

// 2단계: Feature Flag로 전환
import { UserService } from './legacy-service';
import { UserServiceV2 } from './user-service-v2';

const useV2 = process.env.USE_V2_SERVICE === 'true';

export const userService = useV2
  ? new UserServiceV2()
  : new UserService();

// 3단계: 점진적 전환 (10% → 50% → 100%)
// 4단계: 레거시 코드 제거

5단계: 작은 단위로 커밋​

# ❌ 나쁜 예: 거대한 리팩토링 커밋
git commit -m "refactor: rewrite everything" (15,000 lines changed)

# ✅ 좋은 예: 작은 단위로
git commit -m "refactor: extract getUserById function"      (50 lines)
git commit -m "refactor: add type definitions for User"     (30 lines)
git commit -m "test: add unit tests for getUserById"        (100 lines)
git commit -m "refactor: replace magic numbers with const"  (20 lines)

Week 5: Interactive Rebase로 커밋 정리​

6단계: 지저분한 커밋 히스토리 정리​

# 현재 상태 (지저분함)
git log --oneline -10
# abc1234 refactor: user service
# def5678 fix: typo
# ghi9012 refactor: user service again
# jkl3456 wip: user service
# mno7890 fix: lint error
# pqr1234 test: add tests
# stu6543 fix: test
# vwx9876 refactor: more changes

Interactive Rebase로 정리:

git rebase -i HEAD~10

# 에디터에서:
pick abc1234 refactor: user service
fixup def5678 fix: typo                  # abc1234에 합침
squash ghi9012 refactor: user service again  # abc1234에 합치고 메시지 수정
fixup jkl3456 wip: user service
fixup mno7890 fix: lint error
pick pqr1234 test: add tests
fixup stu6543 fix: test
reword vwx9876 refactor: more changes    # 메시지만 수정

# 결과 (깔끔!)
git log --oneline -3
# xyz1111 refactor: modernize user service implementation
# abc2222 test: add comprehensive user service tests
# def3333 refactor: improve error handling

Week 6: 대규모 파일 이동​

7단계: Git이 추적할 수 있도록 이동​

# ❌ 나쁜 예: 삭제 후 새로 생성
rm src/old-structure.js
# ... 새 파일 생성 ...
# Git이 이를 다른 파일로 인식

# ✅ 좋은 예: git mv 사용
git mv src/old-structure.js src/new-structure/service.js

# Git이 rename으로 인식
git log --follow -- src/new-structure/service.js
# 이전 이력도 모두 보임!

# 내용 변경은 별도 커밋으로
git commit -m "refactor: move user service to new location"

# 그 다음 내용 수정
# ... 수정 ...
git commit -m "refactor: update user service implementation"

Week 7: 테스트 주도 리팩토링​

8단계: 안전한 리팩토링 프로세스​

# 1. 현재 동작을 테스트로 고정
# test/legacy-behavior.test.js
describe('Legacy User Service', () => {
  it('should return user with correct format', async () => {
    const user = await userService.getUser(1);
    expect(user).toEqual({
      id: 1,
      name: 'John',
      email: 'john@example.com'
    });
  });
});

npm test  # 통과 확인

# 2. 리팩토링
# ... 코드 개선 ...

# 3. 테스트 여전히 통과하는지 확인
npm test  # 여전히 통과해야 함

# 4. 커밋
git commit -m "refactor: improve user service readability

No behavior change. All tests pass."

Week 8: 백업 브랜치​

9단계: 실험적 변경 전 백업​

# 큰 변경 전 백업 브랜치 생성
git branch backup/before-big-refactor

# 또는 태그로
git tag backup-$(date +%Y%m%d)-before-typescript-migration

# 실험적 리팩토링
# ... 많은 변경 ...

# 잘 안되면 쉽게 복구
git reset --hard backup/before-big-refactor

# 잘 되면 계속 진행

🎓 배운 내용​

Git Bisect​

# 문제를 일으킨 커밋 찾기
git bisect start
git bisect bad HEAD
git bisect good v1.0.0

# 각 단계에서 테스트
git bisect good/bad

# 자동화
git bisect run npm test

안전한 리팩토링 전략​

1. 테스트 추가 (동작 고정)
   ↓
2. 작은 단위로 리팩토링
   ↓
3. 테스트 통과 확인
   ↓
4. 커밋
   ↓
5. 반복

커밋 정리 패턴​

# Interactive Rebase 명령어
pick    # 커밋 유지
reword  # 메시지만 수정
edit    # 커밋 수정
squash  # 이전 커밋과 합침 (메시지 편집)
fixup   # 이전 커밋과 합침 (메시지 버림)
drop    # 커밋 삭제

💡 실전 팁​

1. Branch by Abstraction​

// Phase 1: 인터페이스 추가
interface UserRepository {
  getUser(id: number): Promise<User>;
}

// Phase 2: 레거시 구현
class LegacyUserRepo implements UserRepository {
  // 기존 코드
}

// Phase 3: 새 구현
class ModernUserRepo implements UserRepository {
  // 새 코드
}

// Phase 4: Feature Flag로 전환
const repo = USE_NEW ? new ModernUserRepo() : new LegacyUserRepo();

// Phase 5: 레거시 제거

2. 리팩토링 체크리스트​

Before Refactoring:
- [ ] 기존 동작을 테스트로 커버
- [ ] 백업 브랜치 생성
- [ ] 변경 범위 최소화 계획
- [ ] PR 리뷰어 지정

During Refactoring:
- [ ] 작은 단위로 커밋
- [ ] 각 커밋마다 테스트 통과
- [ ] 의미 있는 커밋 메시지
- [ ] 정기적으로 main rebase

After Refactoring:
- [ ] 모든 테스트 통과
- [ ] 성능 저하 없는지 확인
- [ ] 문서 업데이트
- [ ] 팀원 리뷰

3. 성능 회귀 방지​

// benchmark.js
const Benchmark = require('benchmark');

const suite = new Benchmark.Suite;

suite
  .add('Legacy#getUser', function() {
    return legacyService.getUser(1);
  })
  .add('Modern#getUser', function() {
    return modernService.getUser(1);
  })
  .on('complete', function() {
    console.log('Fastest is ' + this.filter('fastest').map('name'));
  })
  .run();
