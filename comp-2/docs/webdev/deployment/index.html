배포 (Deployment)

웹 애플리케이션을 개발한 후 실제 사용자가 접근할 수 있도록 배포하는 방법을 알아봅니다.

1. 배포란?​

배포(Deployment)는 개발한 웹 애플리케이션을 실제 서버에 업로드하여 인터넷을 통해 누구나 접근할 수 있도록 만드는 과정입니다.

1.1 배포 프로세스​

graph LR
    A[로컬 개발] --> B[빌드]
    B --> C[테스트]
    C --> D[배포]
    D --> E[프로덕션 서버]
    E --> F[사용자 접근]

1.2 배포의 단계​

  1. 개발 (Development): 로컬 환경에서 코드 작성
  2. 빌드 (Build): 프로덕션용 최적화된 파일 생성
  3. 테스트 (Test): 배포 전 기능 및 성능 검증
  4. 배포 (Deploy): 서버에 파일 업로드
  5. 모니터링 (Monitoring): 배포 후 상태 확인
팁

배포 전에 항상 로컬에서 프로덕션 빌드를 테스트하세요. 개발 환경과 프로덕션 환경의 동작이 다를 수 있습니다.

2. 호스팅 vs 배포​

2.1 호스팅 (Hosting)​

호스팅은 웹 애플리케이션을 실행할 서버 공간을 제공하는 서비스입니다.

호스팅 유형:

  • 공유 호스팅 (Shared Hosting): 여러 사이트가 하나의 서버를 공유
  • VPS (Virtual Private Server): 가상 전용 서버
  • 클라우드 호스팅 (Cloud Hosting): AWS, Google Cloud, Azure 등
  • 전용 호스팅 (Dedicated Hosting): 물리적 서버 전체 사용

2.2 배포 (Deployment)​

배포는 코드를 호스팅 서버에 업로드하고 실행 가능한 상태로 만드는 과정입니다.

배포 방법:

  • 수동 배포: FTP, SFTP를 통한 파일 업로드
  • 자동 배포: Git 연동, CI/CD 파이프라인
  • 컨테이너 배포: Docker, Kubernetes 사용

2.3 비교표​

구분호스팅배포
정의서버 공간 제공코드를 서버에 업로드
역할인프라 제공애플리케이션 실행
예시Netlify, Vercel, AWSGit push, CI/CD, FTP
팁

현대적인 플랫폼(Netlify, Vercel)은 호스팅과 배포를 모두 제공하여 한 번에 해결할 수 있습니다.

3. 정적 사이트 호스팅​

정적 사이트는 서버 측 처리 없이 HTML, CSS, JavaScript 파일만으로 동작하는 웹사이트입니다.

3.1 정적 사이트의 장점​

  1. 빠른 속도: CDN을 통한 전 세계 배포
  2. 높은 보안: 서버 측 취약점 없음
  3. 낮은 비용: 무료 또는 저렴한 호스팅
  4. 확장성: 트래픽 증가에 자동 대응

3.2 주요 정적 사이트 호스팅 플랫폼​

3.2.1 Netlify​

특징:

  • Git 기반 자동 배포
  • 무료 SSL 인증서
  • 서버리스 함수 지원
  • 폼 처리 기능

무료 플랜:

  • 월 100GB 대역폭
  • 월 300분 빌드 시간
  • 무제한 사이트

3.2.2 Vercel​

특징:

  • Next.js 최적화
  • Edge Functions 지원
  • 실시간 협업 기능
  • 자동 HTTPS

무료 플랜:

  • 월 100GB 대역폭
  • 무제한 배포
  • 서버리스 함수 지원

3.2.3 GitHub Pages​

특징:

  • GitHub 저장소 직접 연동
  • Jekyll 지원
  • 완전 무료
  • 커스텀 도메인 지원

제약사항:

  • 1GB 저장소 크기 제한
  • 월 100GB 대역폭 제한
  • 공개 저장소 권장

3.3 플랫폼 비교​

4.4 netlify.toml 설정 파일​

프로젝트 루트에 netlify.toml 파일을 생성하여 배포 설정을 코드로 관리할 수 있습니다.

netlify.toml
# 빌드 설정
[build]
  command = "npm run build"
  publish = "build"

  # 환경 변수
  [build.environment]
    NODE_VERSION = "18"
    REACT_APP_API_URL = "https://api.example.com"

# 리다이렉트 설정 (SPA용)
[[redirects]]
  from = "/*"
  to = "/index.html"
  status = 200

# 헤더 설정
[[headers]]
  for = "/*"
  [headers.values]
    X-Frame-Options = "DENY"
    X-XSS-Protection = "1; mode=block"
    Content-Security-Policy = "default-src 'self'"

# 프리뷰 배포 설정
[context.deploy-preview]
  command = "npm run build:preview"

# 브랜치별 설정
[context.production]
  command = "npm run build"

[context.branch-deploy]
  command = "npm run build:staging"
팁

netlify.toml 파일을 사용하면 배포 설정을 버전 관리할 수 있어 팀원들과 공유하기 편리합니다.

4.5 자동 배포 설정​

Netlify는 기본적으로 연결된 Git 저장소에 푸시하면 자동으로 배포됩니다.

6. GitHub Pages 배포 가이드​

6.1 GitHub Pages란?​

GitHub Pages는 GitHub 저장소에서 직접 정적 웹사이트를 호스팅하는 무료 서비스입니다.

6.2 배포 방법​

6.2.1 방법 1: GitHub Actions 사용 (권장)​

Step 1: GitHub Actions 워크플로우 파일 생성

.github/workflows/deploy.yml
name: Deploy to GitHub Pages

on:
  push:
    branches:
      - main

permissions:
  contents: read
  pages: write
  id-token: write

jobs:
  build:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '18'
          cache: 'npm'

      - name: Install dependencies
        run: npm ci

      - name: Build
        run: npm run build

      - name: Upload artifact
        uses: actions/upload-pages-artifact@v3
        with:
          path: ./build

  deploy:
    needs: build
    runs-on: ubuntu-latest

    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}

    steps:
      - name: Deploy to GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4

Step 2: GitHub 저장소 설정

  1. 저장소 → Settings → Pages
  2. Source: "GitHub Actions" 선택
  3. 워크플로우 파일 커밋 및 푸시
git add .github/workflows/deploy.yml
git commit -m "Add GitHub Pages deployment workflow"
git push origin main

6.2.2 방법 2: gh-pages 브랜치 사용​

Step 1: gh-pages 패키지 설치

npm install --save-dev gh-pages

Step 2: package.json에 스크립트 추가

package.json
{
  "name": "my-website",
  "version": "1.0.0",
  "homepage": "https://username.github.io/repository-name",
  "scripts": {
    "predeploy": "npm run build",
    "deploy": "gh-pages -d build",
    "build": "react-scripts build",
    "start": "react-scripts start"
  },
  "devDependencies": {
    "gh-pages": "^6.0.0"
  }
}

Step 3: 배포 실행

npm run deploy

이 명령어는:

  1. predeploy 스크립트 실행 (빌드)
  2. build 폴더 내용을 gh-pages 브랜치로 푸시
  3. 자동으로 사이트 배포

Step 4: GitHub Pages 활성화

  1. 저장소 → Settings → Pages
  2. Source: "Deploy from a branch" 선택
  3. Branch: gh-pages / / (root) 선택
  4. Save 클릭

6.4 SPA 라우팅 문제 해결​

GitHub Pages는 클라이언트 사이드 라우팅을 지원하지 않으므로, 새로고침 시 404 에러가 발생합니다.

6.4.1 해결 방법: 404.html 리다이렉트​

public/404.html
<!DOCTYPE html>
<html>
  <head>
    <meta charset="utf-8">
    <title>Redirecting</title>
    <script>
      // GitHub Pages에서 SPA 라우팅 지원
      sessionStorage.redirect = location.href;
    </script>
    <meta http-equiv="refresh" content="0;URL='/'">
  </head>
  <body>
  </body>
</html>
public/index.html
<!DOCTYPE html>
<html lang="ko">
  <head>
    <meta charset="utf-8" />
    <title>My App</title>
    <script>
      // 404.html에서 리다이렉트된 경우 처리
      (function(){
        var redirect = sessionStorage.redirect;
        delete sessionStorage.redirect;
        if (redirect && redirect !== location.href) {
          history.replaceState(null, null, redirect);
        }
      })();
    </script>
  </head>
  <body>

8. SSL/HTTPS 설정​

8.1 SSL/HTTPS란?​

SSL (Secure Sockets Layer)은 웹사이트와 사용자 간의 데이터를 암호화하는 보안 프로토콜입니다.

HTTPS의 장점:

  1. 보안: 데이터 암호화로 중간자 공격 방지
  2. 신뢰: 브라우저 주소창에 자물쇠 표시
  3. SEO: Google 검색 순위 향상
  4. 성능: HTTP/2 지원으로 더 빠른 로딩

8.2 SSL 인증서 종류​

8.2.1 인증 수준별​

  1. DV (Domain Validation): 도메인 소유권만 확인
  2. OV (Organization Validation): 조직 정보 확인
  3. EV (Extended Validation): 엄격한 신원 확인

8.2.2 무료 SSL 인증서​

Let's Encrypt:

  • 완전 무료
  • 자동 갱신
  • 90일 유효기간
  • 대부분의 호스팅 플랫폼이 지원

8.3 Netlify SSL 설정​

Netlify는 자동으로 Let's Encrypt SSL 인증서를 제공합니다.

8.3.1 자동 HTTPS 활성화​

  1. 사이트 → "Domain settings"
  2. "HTTPS" 섹션에서 상태 확인

8.3.2 HTTPS 강제 적용​

  from = "http://*"
  to = "https://:splat"

  force = true

또는 Netlify UI에서:

  1. "Domain settings" → "HTTPS"
  2. "Force HTTPS" 토글 활성화

8.4 Vercel SSL 설정​

8.4.1 자동 HTTPS​

  • 모든 Vercel 도메인은 자동으로 HTTPS 적용
  • 커스텀 도메인도 자동 인증서 발급

9.1 환경 변수란?​

환경 변수는 애플리케이션의 설정 값을 코드 외부에서 관리하는 방법입니다.

사용 이유:

  1. 보안: API 키, 비밀번호를 코드에 직접 노출하지 않음
  2. 유연성: 환경별로 다른 설정 사용 (개발, 스테이징, 프로덕션)
  3. 편의성: 코드 변경 없이 설정 변경 가능

10. CI/CD 기초 (GitHub Actions)​

10.1 CI/CD란?​

CI (Continuous Integration): 코드 변경사항을 자동으로 빌드하고 테스트하는 프로세스

CD (Continuous Deployment/Delivery): 테스트를 통과한 코드를 자동으로 배포하는 프로세스

10.2 CI/CD의 장점​

  1. 자동화: 수동 작업 최소화
  2. 빠른 피드백: 문제를 조기에 발견
  3. 일관성: 매번 동일한 프로세스 실행
  4. 신뢰성: 사람의 실수 방지
  5. 생산성: 개발자가 코드 작성에 집중

// 요청 시 캐시 우선
self.addEventListener('fetch', event => {
  event.respondWith(
    caches.match(event.request)
      .then(response => {
        if (response) {
          return response;
        }
        return fetch(event.request);
      })
  );
});

11.7.1 CDN이란?​

CDN (Content Delivery Network)은 전 세계에 분산된 서버를 통해 콘텐츠를 제공하여 로딩 속도를 향상시킵니다.

CDN의 장점:

  • 빠른 콘텐츠 전송
  • 서버 부하 감소
  • 높은 가용성
  • DDoS 공격 방어

  • Netlify: 자동 CDN 배포
  • Vercel: Edge Network 제공
  • Cloudflare Pages: 글로벌 CDN

별도 CDN 서비스:

  • Cloudflare
  • Amazon CloudFront
  • Google Cloud CDN

}
