3. 고객 리서치

개요​

고객 리서치는 제품 개발의 핵심입니다. 고객의 진짜 문제를 이해하고, 니즈를 파악하며, 솔루션의 방향성을 검증하는 과정입니다. 체계적인 고객 리서치를 통해 제품-시장 적합성(Product-Market Fit)을 달성할 수 있습니다.

고객 인터뷰 방법론​

인터뷰 유형​

1. 문제 탐색 인터뷰 (Problem Discovery)​

목적: 고객의 문제점과 니즈 파악

시기: 제품 개발 전, 아이디어 검증 단계

구조

도입 (5분):
  - 자기소개
  - 인터뷰 목적 설명
  - 동의 획득

배경 파악 (10분):
  - "현재 어떤 일을 하시나요?"
  - "하루 일과를 설명해주세요"
  - "어떤 도구를 사용하시나요?"

문제 탐색 (20분):
  - "가장 큰 어려움은 무엇인가요?"
  - "왜 그것이 문제인가요?"
  - "현재 어떻게 해결하고 계신가요?"
  - "이상적인 해결책은 무엇일까요?"

우선순위 (10분):
  - "10점 만점에 얼마나 심각한 문제인가요?"
  - "해결을 위해 비용을 지불할 의향이 있나요?"
  - "얼마까지 지불할 수 있나요?"

마무리 (5분):
  - 추가 질문
  - 다른 인터뷰 대상 추천
  - 감사 인사

총 50분

질문 예시

배경:
- "현재 프로젝트를 어떻게 관리하고 계신가요?"
- "팀원은 몇 명인가요? 어떻게 협업하나요?"
- "어떤 도구를 사용하고 계신가요? (도구 스택)"

문제:
- "프로젝트 관리에서 가장 힘든 점은 무엇인가요?"
- "지난 주에 그 문제로 인해 어떤 일이 있었나요? (구체적 사례)"
- "그 문제가 해결되지 않으면 어떤 일이 생기나요?"

현재 해결책:
- "현재 그 문제를 어떻게 해결하고 있나요?"
- "그 방법의 장단점은 무엇인가요?"
- "왜 더 나은 해결책을 찾고 계신가요?"

이상적 해결책:
- "이상적인 솔루션이 있다면 어떤 모습일까요?"
- "꼭 필요한 기능은 무엇인가요?"
- "없어도 되는 기능은 무엇인가요?"

2. 솔루션 검증 인터뷰 (Solution Validation)​

목적: 제안하는 솔루션의 적합성 검증

시기: MVP 개발 전, 프로토타입 단계

구조

도입 (5분):
  - 재연결 (이전 인터뷰 참조)
  - 오늘 목적 설명

문제 재확인 (5분):
  - "지난번 말씀하신 [문제]는 여전히 문제인가요?"
  - "상황이 변했나요?"

솔루션 제시 (15분):
  - 프로토타입/목업 시연
  - "이것이 문제를 해결할 수 있을까요?"
  - "어떤 점이 좋고, 어떤 점이 부족한가요?"

사용 시나리오 (15분):
  - "실제로 어떻게 사용할 것 같나요?"
  - "하루 중 언제 사용하게 될까요?"
  - "누가 함께 사용할까요?"

가격 검증 (10분):
  - "이 솔루션에 비용을 지불할 의향이 있나요?"
  - "적정 가격은 얼마라고 생각하나요?"
  - "경쟁 제품 대비 어떤가요?"

마무리 (5분):
  - 베타 테스트 참여 의향
  - 추가 피드백

총 55분

주의사항

❌ 피해야 할 것:
- "이 기능 좋지 않나요?" (유도 질문)
- "이 가격이면 구매하시겠어요?" (가설적 질문)
- 너무 많은 기능 소개 (압도)

✅ 해야 할 것:
- "이것으로 문제를 해결할 수 있나요?" (중립적)
- "실제로 사용하는 모습을 보여주세요" (행동 관찰)
- 핵심 기능만 보여주기 (집중)

3. 사용성 테스트 (Usability Testing)​

목적: 제품 사용 편의성 검증

시기: 베타 버전 개발 후

구조

준비 (5분):
  - 환경 설정
  - Think-aloud 방법 설명
  - 긴장 완화

작업 수행 (30분):
  - 작업 1: 회원가입
  - 작업 2: 첫 프로젝트 생성
  - 작업 3: 팀원 초대
  - 작업 4: 작업 할당
  - 작업 5: 리포트 생성

피드백 (15분):
  - "어떤 부분이 어려웠나요?"
  - "놀라운 점은 무엇이었나요?"
  - "개선이 필요한 부분은?"

마무리 (5분):
  - 전반적 인상
  - 추천 의향

총 55분

측정 지표

정량적:
  - 작업 완료 시간
  - 작업 성공률
  - 클릭 수
  - 오류 횟수
  - 도움말 참조 횟수

정성적:
  - 좌절/만족 표현
  - 예상 외 행동
  - 피드백 코멘트
  - 전반적 만족도 (1-10)

인터뷰 모집 전략​

타겟 페르소나 정의​

페르소나 1: 스타트업 PM
  인구통계:
    - 나이: 28-35세
    - 직무: 프로덕트 매니저
    - 회사: 스타트업 (10-50명)
    - 경력: 3-7년

  행동:
    - 도구: Notion, Figma, Slack 사용
    - 구매 권한: 있음 (소액)
    - 예산: 팀당 월 $500 이하
    - 의사결정: 빠름 (2주 이내)

  니즈:
    - 빠른 프로토타이핑
    - 팀 협업
    - 고객 피드백 관리
    - ROI 측정

  모집 채널:
    - LinkedIn (Product Manager 그룹)
    - Product Hunt
    - Indie Hackers
    - Twitter (PM 인플루언서 팔로워)

페르소나 2: 중견기업 IT 디렉터
  인구통계:
    - 나이: 35-50세
    - 직무: IT 디렉터
    - 회사: 중견기업 (100-1000명)
    - 경력: 10-20년

  행동:
    - 도구: Jira, Confluence, Salesforce
    - 구매 권한: 있음 (승인 필요)
    - 예산: 부서당 월 $5,000-$50,000
    - 의사결정: 느림 (3-6개월)

  니즈:
    - 기존 시스템 통합
    - 보안 & 컴플라이언스
    - 확장성
    - 벤더 신뢰성

  모집 채널:
    - LinkedIn (IT 그룹)
    - Gartner Peer Insights
    - 산업 컨퍼런스
    - CIO 커뮤니티

4. 리퍼럴 (Referral)

방법:
  인터뷰 마지막에:
  "비슷한 경험을 가진 분을 아시나요? 소개해주시면 감사하겠습니다."

인센티브:
  - 추가 기프티콘 ($10-$20)
  - 무료 플랜 업그레이드 (3개월)
  - 얼리 액세스

효과:
  - 높은 품질 (유사 페르소나)
  - 빠른 모집 (네트워크 활용)
  - 비용 효율적

주의:
  - 과도한 리퍼럴 방지 (편향)
  - 다양성 확보 (여러 소스 혼합)

인터뷰 진행 기술​

질문 기법​

1. 5 Whys (다섯 번의 왜)

목적: 근본 원인 파악

예시:
고객: "프로젝트 관리 도구가 너무 복잡해요."

1차: "왜 복잡하다고 느끼시나요?"
→ "기능이 너무 많아서요."

2차: "왜 기능이 많은 것이 문제인가요?"
→ "어디서부터 시작해야 할지 모르겠어요."

3차: "왜 시작점을 찾기 어려우신가요?"
→ "튜토리얼이 너무 길고 지루해요."

4차: "왜 튜토리얼이 길다고 느끼시나요?"
→ "필요 없는 기능까지 모두 설명해요."

5차: "왜 필요 없는 기능이 포함되어 있나요?"
→ "제 팀은 간단한 작업 관리만 필요한데, 도구는 엔터프라이즈용이에요."

근본 원인 발견:
→ 타겟 고객과 제품 복잡도 불일치

주의사항

❌ "왜요? 왜요? 왜요?" (심문 느낌)
✅ "그게 왜 문제인가요?", "그 이유가 궁금한데요", "조금 더 설명해주시겠어요?"

❌ 5회 고정 (융통성 없음)
✅ 3-7회 (상황에 따라 유연하게)

2. JTBD (Jobs-to-be-Done)

프레임워크:
"When [상황], I want to [동기], so I can [결과]."

질문:
- "언제 이 도구를 사용하시나요?" (상황)
- "무엇을 하려고 하시나요?" (동기)
- "어떤 결과를 원하시나요?" (결과)

예시:
고객 답변:
"클라이언트 미팅 전에 (상황)
프로젝트 진행 상황을 정리하고 싶어요 (동기)
전문적으로 보이기 위해서요 (결과)"

JTBD:
"클라이언트 미팅 전, 프로젝트 상황을 정리해서, 전문성을 보여주고 싶다"

제품 시사점:
- 빠른 리포트 생성 기능
- 시각적 대시보드 (프레젠테이션 모드)
- 클라이언트 공유 기능

3. 행동 vs 의견

목적: 실제 행동 파악 (의견은 부정확)

❌ 의견 질문:
"프로젝트 관리 도구에서 어떤 기능이 중요하다고 생각하세요?"
→ 답변: "간트 차트요!"
(실제로는 사용 안 할 수도)

✅ 행동 질문:
"지난주에 프로젝트 관리 도구를 어떻게 사용하셨나요?"
→ 답변: "매일 아침 작업 리스트 확인하고, 완료 표시했어요."
(실제 사용 패턴 파악)

❌ 가설적 질문:
"이 기능이 있으면 사용하실 건가요?"
→ 답변: "네!" (실제로는 안 씀)

✅ 과거 행동 질문:
"비슷한 문제를 어떻게 해결하셨나요?"
→ 답변: "엑셀로 수동 관리했어요."
(실제 해결책과 동기)

4. 침묵의 힘

기법: 질문 후 3-5초 기다리기

효과:
  - 고객이 더 깊이 생각함
  - 추가 정보 제공
  - 진짜 의견 도출

예시:
면접관: "가장 큰 어려움은 무엇인가요?"
고객: "음... 시간 관리요."
면접관: [침묵, 고개 끄덕임]
고객: "사실 시간 관리뿐 아니라, 우선순위를 정하는 게 어려워요.

인사이트:
→ 시간 관리가 아니라 우선순위 문제

피해야 할 질문​

1. 유도 질문 (Leading Questions)

❌ "이 기능 좋지 않나요?"
✅ "이 기능에 대해 어떻게 생각하세요?"

❌ "빠른 온보딩이 중요하죠?"

✅ "경쟁사 제품과 비교하면 어떤가요?"

2. 다중 질문 (Multiple Questions)

❌ "이 기능을 사용하실 건가요? 어떻게 사용하실 건가요?
    팀원들도 좋아할까요? 가격은 적절한가요?"
(고객 혼란, 일부 질문 누락)

✅ "이 기능을 사용하실 건가요?"
[답변 듣기]
"어떻게 사용하실 계획인가요?"
[답변 듣기]
"팀원들 반응은 어떨 것 같나요?"
...

한 번에 하나씩!

니즈 유형​

1. 기능적 니즈 (Functional Needs)​

정의: 제품이 수행해야 할 작업

예시:

  - "팀원에게 작업을 할당하고 싶다"

측정:
  - 완료 가능 여부 (예/아니오)
  - 완료 시간 (빠를수록 좋음)
  - 정확도 (오류율 낮을수록 좋음)

우선순위:
  - 높음 (기본적 가치)
  - 충족되지 않으면 제품 사용 불가

2. 감정적 니즈 (Emotional Needs)​

정의: 제품 사용 시 느끼고 싶은 감정

예시:
  - "통제력을 느끼고 싶다" (혼란 → 정돈)
  - "자신감을 갖고 싶다" (불안 → 확신)
  - "성취감을 느끼고 싶다" (좌절 → 만족)

측정:
  - 감정 척도 (1-10)
  - 만족도 조사
  - NPS (추천 의향)

우선순위:
  - 중간 (차별화 요소)
  - 경쟁 제품과 차이 만듦

3. 사회적 니즈 (Social Needs)​

정의: 다른 사람에게 어떻게 보이고 싶은가

예시:
  - "전문적으로 보이고 싶다"
  - "혁신적인 팀으로 인정받고 싶다"
  - "효율적인 리더로 보이고 싶다"

측정:
  - 브랜드 인지도
  - 추천/공유율
  - 소셜 증거 (사례, 리뷰)

우선순위:
  - 낮음-중간 (부가 가치)
  - B2B에서는 중요 (의사결정 정당화)

  - 있으면: 당연함
  - 없으면: 매우 불만족
  - 예시: 작업 생성, 수정, 삭제

성능 기능 (One-Dimensional):
  - 많을수록: 만족도 증가
  - 적을수록: 만족도 감소
  - 예시: 속도, 저장 용량

매력 기능 (Attractive):
  - 있으면: 큰 만족
  - 없어도: 불만 없음
  - 예시: AI 추천, 자동화

무관심 기능 (Indifferent):
  - 있어도: 무관심
  - 없어도: 무관심
  - 예시: 불필요한 고급 기능

역기능 (Reverse):
  - 있으면: 불만족
  - 없으면: 만족
  - 예시: 복잡한 UI, 불필요한 알림

Kano 설문​

기능: AI 작업 우선순위 추천

질문 1 (Functional):
"이 기능이 있다면 어떻게 느끼시겠어요?"
  A. 매우 만족한다
  B. 당연하다
  C. 상관없다
  D. 어쩔 수 없이 받아들인다
  E. 매우 불만족한다

질문 2 (Dysfunctional):
"이 기능이 없다면 어떻게 느끼시겠어요?"
  A. 매우 만족한다
  B. 당연하다
  C. 상관없다
  D. 어쩔 수 없이 받아들인다
  E. 매우 불만족한다

분류 (응답 조합):
  A-E: Attractive (매력 기능)
  A-D: Attractive
  B-E: Must-Be (기본 기능)
  C-C: Indifferent (무관심)
  ...

결과 분석 (예시)​

| 기능 | Must-Be | Performance | Attractive | Indifferent | 분류 |
|------|---------|-------------|------------|-------------|------|
| 작업 생성 | 85% | 10% | 5% | 0% | Must-Be |
| 빠른 속도 | 10% | 70% | 15% | 5% | Performance |
| AI 추천 | 5% | 20% | 60% | 15% | Attractive |
| 간트 차트 | 15% | 30% | 25% | 30% | Indifferent |

우선순위:
1. Must-Be (작업 생성): MVP 필수
2. Performance (빠른 속도): 경쟁 우위
3. Attractive (AI 추천): 차별화
4. Indifferent (간트 차트): 후순위 또는 제외

MoSCoW 우선순위​

Must Have (필수):
  - 제품 동작에 필수
  - 없으면 제품 사용 불가
  - 예시: 사용자 인증, 작업 CRUD

Should Have (중요):
  - 중요하지만 필수는 아님
  - MVP에 포함 권장
  - 예시: 팀 협업, 파일 첨부

Could Have (선택):
  - 있으면 좋음
  - 시간/예산 여유 시 추가
  - 예시: 통합, 커스텀 필드

Won't Have (제외):
  - 현재 버전에서 제외
  - 나중에 고려
  - 예시: 고급 분석, 모바일 앱 (v1)

적용 예시:
Must: 60% 기능
Should: 20% 기능
Could: 15% 기능
Won't: 5% 기능
