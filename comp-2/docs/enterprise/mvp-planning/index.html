4. MVP 기획

개요​

MVP(Minimum Viable Product)는 최소한의 기능으로 고객 가치를 제공하고, 학습을 최대화하는 제품입니다. 완벽한 제품이 아니라, 가설을 검증하고 빠르게 배우기 위한 도구입니다.

MVP 철학​

Eric Ries의 정의​

"The minimum viable product is that version of a new product
which allows a team to collect the maximum amount of validated
learning about customers with the least effort."

MVP는 최소한의 노력으로 고객에 대한 최대한의 검증된 학습을 얻을 수 있는
신제품의 버전입니다.

MVP vs 완제품​

MVP:
  목적: 학습, 검증
  범위: 핵심 기능만
  품질: 동작하는 수준
  시간: 빠름 (2-3개월)
  비용: 낮음 ($50K-$200K)
  고객: 얼리어답터
  결과: 방향성 확인

완제품:
  목적: 시장 지배
  범위: 모든 기능
  품질: 완벽
  시간: 느림 (12-18개월)
  비용: 높음 ($500K-$2M)
  고객: 주류 시장
  결과: 수익 극대화

MVP의 3가지 타입​

1. 검증 MVP (Validation MVP)​

목적: 문제 존재 여부 확인
방법: 프로토타입, 랜딩 페이지, 설문
시간: 1-2주
비용: $1K-$10K
학습: 문제가 진짜 있는가?
예시: Dropbox 동영상 (제품 없이 수요 검증)

2. 학습 MVP (Learning MVP)​

목적: 솔루션 적합성 확인
방법: 기본 기능 제품
시간: 1-3개월
비용: $50K-$200K
학습: 우리 솔루션이 맞는가?
예시: Airbnb (창업자가 직접 사진 촬영)

3. 확장 MVP (Growth MVP)​

목적: 성장 채널 확인
방법: 완성도 높은 제품
시간: 3-6개월
비용: $200K-$500K
학습: 어떻게 성장할 것인가?
예시: Instagram (사진 공유만 집중)

핵심 기능 정의​

가치 제안 캔버스 (Value Proposition Canvas)​

고객 프로필​

고객 작업 (Customer Jobs):
  기능적:
    - 프로젝트 관리
    - 팀 협업
    - 진행 상황 추적

  사회적:
    - 전문적으로 보이기
    - 효율적인 리더로 인정받기

  감정적:
    - 통제력 느끼기
    - 스트레스 줄이기

고객 고통 (Pains):
  - 도구가 너무 복잡함 (8/10 심각도)
  - 팀원들이 업데이트 안 함 (7/10)
  - 여러 도구 사용의 피로 (7/10)
  - 보고서 작성에 시간 소요 (6/10)

고객 이득 (Gains):
  - 빠른 온보딩 (<30분)
  - 실시간 협업
  - 한눈에 보는 대시보드
  - 자동 리포트 생성

가치 제안​

제품 & 서비스:
  - 간단한 작업 관리 도구
  - 실시간 협업 기능
  - 시각적 대시보드
  - 원클릭 리포트

고통 완화제 (Pain Relievers):
  - 5분 온보딩 (vs 2주)
  - 직관적 UI (학습 불필요)
  - 올인원 플랫폼 (도구 전환 불필요)
  - 자동 리포트 (수동 작업 제거)

이득 창출자 (Gain Creators):
  - AI 우선순위 추천
  - 실시간 알림
  - 모바일 접근
  - 통합 (Slack, Google Drive)

핵심 기능 도출​

| 고객 Pain | Pain Reliever | 핵심 기능 | 우선순위 |
|-----------|---------------|-----------|----------|
| 복잡성 | 간단한 UI | 작업 CRUD, 리스트/보드 뷰 | P0 |
| 업데이트 누락 | 실시간 알림 | 알림 시스템, 활동 피드 | P0 |
| 도구 피로 | 올인원 | 파일 첨부, 댓글, 멘션 | P0 |
| 보고서 작성 | 자동 리포트 | 대시보드, 내보내기 | P1 |
| 우선순위 혼란 | AI 추천 | AI 우선순위 엔진 | P1 |

Kano 모델 적용​

Must-Be (기본):
  - 작업 생성/수정/삭제
  - 사용자 인증
  - 팀 관리
  → MVP 필수

Performance (성능):
  - 로딩 속도 (<3초)
  - 실시간 동기화
  - 모바일 반응형
  → MVP 포함 (경쟁 우위)

Attractive (매력):
  - AI 우선순위
  - 자동화
  - 커스텀 테마
  → MVP 선택적 포함 (차별화)

Indifferent (무관심):
  - 간트 차트
  - 타임 트래킹
  - 고급 리포팅
  → MVP 제외 (Phase 2)

MoSCoW 우선순위​

Must Have (필수 - 60%):
  사용자 관리:
    - 회원가입/로그인
    - 프로필 관리
    - 팀 초대

  작업 관리:
    - 작업 CRUD
    - 작업 할당
    - 마감일 설정
    - 상태 변경

  협업:
    - 댓글
    - 멘션 (@user)
    - 활동 피드

  보기:
    - 리스트 뷰
    - 보드 뷰 (칸반)

Should Have (중요 - 20%):
  - 파일 첨부 (최대 10MB)
  - 검색
  - 필터링
  - 알림 (인앱, 이메일)
  - 대시보드 (간단한)

Could Have (선택 - 15%):
  - 타임라인 뷰
  - 라벨/태그
  - 우선순위 표시
  - 통합 (Slack)
  - 모바일 앱 (반응형 웹)

Won't Have (제외 - 5%):
  - 간트 차트
  - 타임 트래킹
  - 고급 리포팅
  - 자동화
  - API (Phase 2)
  - SSO (Enterprise 기능)

기능 명세서​

Epic & User Story​

Epic 1: 사용자 관리​

Epic: 사용자 관리
Description: 사용자가 가입하고 로그인하여 프로필을 관리할 수 있다

User Stories:

1. 회원가입
   As a: 신규 사용자
   I want to: 이메일로 회원가입
   So that: 서비스를 사용할 수 있다

   Acceptance Criteria:
     - [ ] 이메일, 비밀번호 입력
     - [ ] 이메일 중복 검사
     - [ ] 비밀번호 강도 검증 (8자 이상, 영문+숫자)
     - [ ] 이메일 인증 (선택적)
     - [ ] 가입 완료 후 대시보드로 이동

   Priority: P0
   Story Points: 5
   Dependencies: 없음

2. 로그인
   As a: 등록된 사용자
   I want to: 이메일과 비밀번호로 로그인
   So that: 내 작업을 확인할 수 있다

   Acceptance Criteria:
     - [ ] 이메일, 비밀번호 입력
     - [ ] 인증 성공 시 토큰 발급
     - [ ] "Remember Me" 옵션 (선택적)
     - [ ] 실패 시 에러 메시지
     - [ ] 비밀번호 찾기 링크

   Priority: P0
   Story Points: 3
   Dependencies: 회원가입

3. 프로필 관리
   As a: 로그인한 사용자
   I want to: 프로필 정보 수정
   So that: 최신 정보를 유지할 수 있다

   Acceptance Criteria:
     - [ ] 이름, 프로필 사진 수정
     - [ ] 이메일 변경 (재인증 필요)
     - [ ] 비밀번호 변경
     - [ ] 알림 설정

   Priority: P1
   Story Points: 3
   Dependencies: 로그인

4. 팀 초대
   As a: 팀 관리자
   I want to: 팀원 초대
   So that: 함께 작업할 수 있다

   Acceptance Criteria:
     - [ ] 이메일로 초대 발송
     - [ ] 초대 수락/거절
     - [ ] 팀원 목록 조회
     - [ ] 팀원 제거 (관리자만)

   Priority: P0
   Story Points: 5
   Dependencies: 로그인

Epic 2: 작업 관리​

Epic: 작업 관리
Description: 사용자가 작업을 생성하고 관리할 수 있다

User Stories:

1. 작업 생성
   As a: 팀원
   I want to: 새 작업 생성
   So that: 해야 할 일을 기록할 수 있다

   Acceptance Criteria:
     - [ ] 제목 입력 (필수)
     - [ ] 설명 입력 (선택)
     - [ ] 담당자 지정 (선택)
     - [ ] 마감일 설정 (선택)
     - [ ] 상태 선택 (Todo, In Progress, Done)
     - [ ] 생성 후 작업 목록에 표시

   Priority: P0
   Story Points: 5
   Dependencies: 로그인

2. 작업 수정
   As a: 팀원
   I want to: 작업 정보 수정
   So that: 변경 사항을 반영할 수 있다

   Acceptance Criteria:
     - [ ] 모든 필드 수정 가능
     - [ ] 변경 이력 기록
     - [ ] 실시간 동기화
     - [ ] 저장 후 알림

   Priority: P0
   Story Points: 3
   Dependencies: 작업 생성

3. 작업 삭제
   As a: 작업 생성자 또는 관리자
   I want to: 작업 삭제
   So that: 불필요한 작업을 제거할 수 있다

   Acceptance Criteria:
     - [ ] 삭제 확인 다이얼로그
     - [ ] 소프트 삭제 (30일 보관)
     - [ ] 휴지통에서 복구 가능
     - [ ] 30일 후 영구 삭제

   Priority: P0
   Story Points: 3
   Dependencies: 작업 생성

4. 작업 할당
   As a: PM
   I want to: 작업을 팀원에게 할당
   So that: 책임을 명확히 할 수 있다

   Acceptance Criteria:
     - [ ] 드롭다운에서 팀원 선택
     - [ ] 할당 시 팀원에게 알림
     - [ ] 재할당 가능
     - [ ] 할당 해제 가능

   Priority: P0
   Story Points: 3
   Dependencies: 작업 생성, 팀 초대

5. 마감일 관리
   As a: 팀원
   I want to: 작업에 마감일 설정
   So that: 시간 관리를 할 수 있다

   Acceptance Criteria:
     - [ ] 날짜 선택 (캘린더 UI)
     - [ ] 마감일 임박 시 알림 (1일 전)
     - [ ] 지난 마감일 빨간색 표시
     - [ ] 마감일 수정/삭제 가능

   Priority: P0
   Story Points: 3
   Dependencies: 작업 생성

Epic 3: 협업​

Epic: 협업
Description: 팀원들이 작업에 대해 소통할 수 있다

User Stories:

1. 댓글 작성
   As a: 팀원
   I want to: 작업에 댓글 작성
   So that: 의견을 공유할 수 있다

   Acceptance Criteria:
     - [ ] 텍스트 입력
     - [ ] 마크다운 지원 (선택적)
     - [ ] 파일 첨부 (선택적)
     - [ ] 실시간 표시
     - [ ] 댓글 수정/삭제

   Priority: P0
   Story Points: 5
   Dependencies: 작업 생성

2. 멘션 (@)
   As a: 팀원
   I want to: 댓글에서 팀원 멘션
   So that: 특정 사람에게 알릴 수 있다

   Acceptance Criteria:
     - [ ] @ 입력 시 팀원 목록 표시
     - [ ] 멘션된 사람에게 알림
     - [ ] 멘션 클릭 시 프로필 보기
     - [ ] 여러 명 멘션 가능

   Priority: P0
   Story Points: 3
   Dependencies: 댓글 작성

3. 활동 피드
   As a: 팀원
   I want to: 작업의 모든 활동 보기
   So that: 변경 사항을 추적할 수 있다

   Acceptance Criteria:
     - [ ] 시간순 정렬 (최신 상단)
     - [ ] 활동 유형: 생성, 수정, 댓글, 상태 변경
     - [ ] 사용자 이름 및 시간 표시
     - [ ] 필터링 (댓글만, 변경만)

   Priority: P0
   Story Points: 5
   Dependencies: 작업 생성, 댓글

4. 파일 첨부
   As a: 팀원
   I want to: 작업에 파일 첨부
   So that: 관련 자료를 공유할 수 있다

   Acceptance Criteria:
     - [ ] 드래그 앤 드롭 업로드
     - [ ] 파일 크기 제한 (10MB)
     - [ ] 파일 형식: 이미지, PDF, DOC, XLS
     - [ ] 미리보기 (이미지, PDF)
     - [ ] 다운로드

   Priority: P1
   Story Points: 5
   Dependencies: 작업 생성

Epic: 보기

1. 리스트 뷰

   Acceptance Criteria:

     - [ ] 클릭하여 상세 보기
     - [ ] 인라인 편집 (제목, 담당자)
     - [ ] 드래그로 순서 변경

   Priority: P0
   Story Points: 5
   Dependencies: 작업 생성

2. 보드 뷰 (칸반)

   So that: 진행 상황을 시각적으로 파악할 수 있다

   Acceptance Criteria:
     - [ ] 열: Todo, In Progress, Done
     - [ ] 드래그 앤 드롭으로 상태 변경
     - [ ] 각 열의 작업 수 표시
     - [ ] 필터 (담당자, 마감일)
     - [ ] 커스텀 열 (선택적)

   Priority: P0
   Story Points: 8
   Dependencies: 작업 생성

3. 필터 & 검색
   As a: 팀원
   I want to: 작업 필터링 및 검색
   So that: 원하는 작업을 빠르게 찾을 수 있다

   Acceptance Criteria:
     - [ ] 검색 (제목, 설명)
     - [ ] 필터: 담당자, 상태, 마감일
     - [ ] 여러 필터 조합
     - [ ] 필터 저장 (선택적)
     - [ ] 검색 결과 하이라이트

   Priority: P1
   Story Points: 5
   Dependencies: 리스트 뷰

4. 대시보드
   As a: PM
   I want to: 프로젝트 개요 보기
   So that: 전체 진행 상황을 파악할 수 있다

   Acceptance Criteria:
     - [ ] 작업 통계 (총, 완료, 진행 중)
     - [ ] 팀원별 작업 수
     - [ ] 마감일 임박 작업
     - [ ] 완료율 그래프
     - [ ] 최근 활동

   Priority: P1
   Story Points: 8
   Dependencies: 작업 생성

기술 요구사항​

성능:
  - 페이지 로딩: <3초 (초기)
  - 작업 생성/수정: <1초
  - 검색: <2초
  - 실시간 동기화: <5초

확장성:
  - 팀 크기: 최대 100명
  - 작업 수: 팀당 10,000개
  - 파일 저장: 팀당 5GB
  - 동시 접속: 1,000명

보안:
  - HTTPS (필수)
  - 비밀번호 해싱 (bcrypt)
  - JWT 인증
  - CORS 설정
  - XSS, CSRF 방어

호환성:
  - 브라우저: Chrome, Firefox, Safari, Edge (최신 2버전)
  - 모바일: 반응형 웹 (iOS Safari, Android Chrome)
  - 해상도: 1024px 이상

가용성:
  - Uptime: 99.5% (MVP)
  - 백업: 일일
  - 복구 시간: <4시간

프로토타이핑​

프로토타입 단계​

1. 저충실도 (Low-Fidelity)​

목적: 빠른 아이디어 검증
도구: 종이, 화이트보드, Balsamiq
시간: 1-2일
비용: 거의 없음

성공 지표 (Metrics)​

MVP 지표​

1. Product-Market Fit 지표​

PMF Survey (Sean Ellis Test)

질문: "이 제품이 없어진다면 어떻게 느끼시겠어요?"

선택지:
  - 매우 실망할 것이다
  - 조금 실망할 것이다
  - 별로 실망하지 않을 것이다
  - 해당 없음 (더 이상 사용 안 함)

PMF 달성 기준:
  - "매우 실망" ≥ 40%

현실:
  - Superhuman: 58% (PMF 달성)
  - 평균 SaaS: 20-30% (PMF 미달성)

측정 시기:
  - 최소 100명 활성 사용자 이후
  - 2-4주 사용 후 설문

NPS (Net Promoter Score)

질문: "이 제품을 동료에게 추천할 가능성은?"

척도: 0-10
  - 9-10: Promoter (추천자)
  - 7-8: Passive (중립)
  - 0-6: Detractor (비추천자)

NPS = % Promoters - % Detractors

기준:
  - NPS > 50: 훌륭함 (Excellent)
  - NPS > 30: 좋음 (Good)
  - NPS > 0: 괜찮음 (Okay)
  - NPS < 0: 문제 있음 (Poor)

MVP 목표: NPS > 30

2. 사용 지표​

활성 사용자

DAU (Daily Active Users):
  정의: 하루 동안 로그인한 사용자 수
  목표: 100+ (MVP 출시 1개월)

MAU (Monthly Active Users):
  정의: 한 달 동안 로그인한 사용자 수
  목표: 500+ (MVP 출시 1개월)

DAU/MAU Ratio:
  정의: 사용자 참여도 (Stickiness)
  계산: DAU / MAU
  기준:
    - > 20%: 매우 높음 (Facebook 50%)
    - 10-20%: 높음
    - 5-10%: 보통
    - < 5%: 낮음
  목표: > 20%

참여 지표

작업 생성 수:
  측정: 사용자당 평균 작업 생성 수/주
  목표: > 10개/사용자/주

댓글 수:
  측정: 작업당 평균 댓글 수
  목표: > 3개/작업

세션 길이:
  측정: 평균 사용 시간/세션
  목표: > 10분

세션 빈도:
  측정: 사용자당 평균 세션 수/주
  목표: > 5회/주

3. 유지 지표 (Retention)​

코호트 분석

| 주 | 신규 | W1 | W2 | W3 | W4 | W8 |
|----|------|----|----|----|----|-----|
| Week 1 | 100 | 60% | 40% | 30% | 25% | 20% |
| Week 2 | 150 | 65% | 45% | 35% | 30% | - |
| Week 3 | 200 | 70% | 50% | 40% | - | - |

분석:
- Week 1 유지율: 60% (좋음)
- Week 4 유지율: 25-30% (개선 필요)
- 트렌드: 유지율 증가 중 (긍정적)

목표:
- Week 1: > 60%
- Week 4: > 30%
- Week 8: > 20%

이탈 분석 (Churn Analysis)

Monthly Churn Rate:
  정의: 이번 달에 이탈한 사용자 비율
  계산: (이탈 사용자 / 시작 사용자) × 100
  기준:
    - < 5%: 훌륭함
    - 5-10%: 좋음
    - 10-20%: 보통
    - > 20%: 문제 있음
  목표: < 10%

이탈 이유:
  1. 사용하지 않음 (40%)
     → 온보딩 개선, 참여 유도
  2. 가격 (25%)
     → 가치 전달 강화, 할인
  3. 기능 부족 (20%)
     → 우선순위 기능 개발
  4. 경쟁사 전환 (10%)
     → 차별화 강화
  5. 기타 (5%)

4. 성장 지표​

바이럴 계수 (Viral Coefficient)

K = 초대 수 × 전환율

예시:
  - 사용자 1명이 평균 3명 초대
  - 초대 수락률: 30%
  - K = 3 × 0.3 = 0.9

해석:
  - K > 1: 바이럴 성장 (exponential)
  - K = 1: 선형 성장
  - K < 1: 둔화 성장 (유료 마케팅 필요)

목표: K > 0.5 (MVP), K > 1 (PMF 후)

성장률 (Growth Rate)

월 성장률 = (이번 달 사용자 - 지난 달 사용자) / 지난 달 사용자 × 100

예시:
  - 1월: 100명
  - 2월: 150명
  - 성장률: (150-100)/100 = 50%

기준:
  - > 20%/월: 매우 빠름 (하이퍼 그로스)
  - 10-20%/월: 빠름
  - 5-10%/월: 보통
  - < 5%/월: 느림

목표: > 20%/월 (MVP), > 10%/월 (성숙기)

대시보드 예시​

MVP 대시보드 (실시간):

사용자:
  - 총 사용자: 532
  - DAU: 128 (24%)
  - MAU: 532 (100%)
  - DAU/MAU: 24%
  - 신규 (오늘): 12
  - 성장률 (월): 35%

참여:
  - 작업 생성 (오늘): 156
  - 작업 생성 (평균/사용자/주): 8.5
  - 댓글 (오늘): 89
  - 세션 길이 (평균): 12분
  - 세션 빈도 (평균/주): 5.2회

유지:
  - Week 1 유지율: 65%
  - Week 4 유지율: 32%
  - 월 이탈률: 8%

PMF:
  - Sean Ellis: 42% (PMF 달성!)
  - NPS: 38 (좋음)

수익 (프리미엄 출시 후):
  - MRR: $2,450

2. Measure (측정):
   - 지표: DAU/MAU, 작업 생성 수, NPS
   - 기간: 4주 (50-100명 베타)
   - 데이터 수집: Mixpanel, Google Analytics

3. Learn (학습):
   - 결과 분석:
     - DAU/MAU: 25% (목표 20% 초과) ✅
     - 작업 생성: 12개/사용자/주 (목표 10 초과) ✅
     - NPS: 42 (목표 30 초과) ✅
   - 인사이트:
     - 보드 뷰를 선호 (70% 사용)
     - 모바일 접근성 요구 (60% 요청)
     - 통합 필요 (Slack 50% 요청)
   - 결정:
     - Pivot: ❌ (가설 검증됨)
     - Persevere: ✅ (방향 유지)
     - 다음 우선순위: 모바일 최적화, Slack 통합

4. 다음 루프:
   - 새 가설: "모바일 최적화가 참여도를 높인다"
   - 실험: 반응형 개선 + 모바일 UX
   - 기간: 2주
   - 측정: 모바일 사용률, 세션 빈도
   ...

루프 반복 주기: 2-4주

Pivot vs Persevere​

Pivot 신호 (방향 전환)

정량적:
  - DAU/MAU < 10% (낮은 참여)
  - Week 4 유지율 < 15%
  - 월 이탈률 > 20%
  - NPS < 0
  - 성장률 < 5%/월

정성적:
  - 고객 피드백: "필요 없다", "사용 안 한다"
  - Sean Ellis < 20% "매우 실망"
  - 경쟁 우위 없음
  - 팀 사기 저하

Pivot 유형:
  1. Zoom-in: 기능 하나에 집중
  2. Zoom-out: 제품을 기능의 일부로
  3. Customer Segment: 다른 고객 타겟
  4. Customer Need: 다른 문제 해결
  5. Platform: 제품 → 플랫폼
  6. Business Model: 가격/수익 모델 변경
  7. Technology: 다른 기술 스택
  8. Channel: 판매 채널 변경

예시:
  - Instagram: 위치 기반 SNS → 사진 공유
  - Slack: 게임 → 팀 메신저
  - Twitter: 팟캐스트 플랫폼 → 마이크로블로그

Persevere 신호 (방향 유지)

정량적:
  - DAU/MAU > 20%
  - Week 4 유지율 > 30%
  - 월 이탈률 < 10%
  - NPS > 30
  - 성장률 > 20%/월

정성적:
  - 고객 피드백: "이게 없으면 힘들다"
  - Sean Ellis > 40% "매우 실망"
  - 명확한 차별화
  - 팀 사기 높음

액션:
  - 핵심 기능 강화
  - 새 기능 추가 (고객 요청)
  - 성장 가속 (마케팅, 영업)
  - 팀 확장

실전 사례​

사례 1: Dropbox MVP​

초기 가설​

"사람들은 파일을 여러 기기에서 동기화하고 싶어한다."

MVP 전략​

1단계: 동영상 MVP (2008년)

방법: 제품 시연 동영상 제작
  - 길이: 3분 30초
  - 내용: Dropbox 사용 방법, 유머
  - 배포: Hacker News, Digg

비용: $0 (자체 제작)
시간: 1일

결과:
  - 조회수: 수백만
  - 베타 신청: 5,000 → 75,000 (하룻밤)
  - 학습: 수요 있음! 제품 개발 정당화

교훈:
  - 제품 없이도 수요 검증 가능
  - 동영상이 복잡한 개념 설명에 효과적

2단계: 베타 MVP (2008년)

기능:
  - 파일 동기화 (Windows, Mac)
  - 폴더 공유
  - 웹 접근

비용: ~$200K (개발)
시간: 6개월

베타 전략:
  - 초대제 (희소성)
  - 추천 보상 (250MB → 500MB)
  - 얼리어답터 타겟 (기술 커뮤니티)

결과:
  - 베타 사용자: 100,000
  - 유지율: 매우 높음
  - 바이럴 계수: 1.2

학습:
  - 제품이 문제를 해결함
  - 바이럴 성장 가능
  - 스토리지 비용이 주요 과제

3단계: 프리미엄 모델 (2009년)

가격:
  - 무료: 2GB
  - Pro: $9.99/월, 50GB

결과:
  - 첫 달 수익: $100K+
  - 전환율: 4% (업계 평균 2%)
  - 학습: 프리미엄 모델 동작함

성과​

2010년: 4M 사용자
2013년: 100M 사용자
2021년: 700M 사용자
평가액: $10B+

사례 2: Airbnb MVP​

초기 가설​

"여행자들은 저렴한 숙소를 원하고,
집주인들은 빈 방으로 수익을 원한다."

MVP 여정​

1단계: 3명 MVP (2007년)

상황:
  - 샌프란시스코 디자인 컨퍼런스
  - 호텔이 만실
  - 창업자들이 월세 걱정

아이디어:
  - 자기 아파트에 에어매트 3개 깔기
  - 아침 식사 제공
  - 1박 $80

방법:
  - 간단한 웹사이트 (하룻밤에 제작)
  - 사진 (디지털 카메라)
  - 이메일 예약

결과:
  - 게스트: 3명 (모두 예약!)
  - 수익: $240
  - 학습: 수요 존재, 사람들이 낯선 이 집에 묵을 의향 있음

비용: $0
시간: 1일

2단계: 컨퍼런스 타겟 MVP (2008년)

전략:
  - 대형 컨퍼런스 타겟 (SXSW, DNC)
  - 호텔 만실 시기 공략

기능:
  - 리스팅 생성
  - 예약 시스템 (간단한)
  - 결제 (PayPal)

문제:
  - 사진 품질 낮음 → 예약 적음
  - 창업자들이 직접 방문해서 전문 사진 촬영
  - 수동 프로세스 (확장 불가)

결과:
  - 예약률: 2-3배 증가 (전문 사진 후)
  - 학습: 사진 품질이 핵심, 수동도 OK (초기)

비용: $5K (사진 장비, 여행)
시간: 2개월

3단계: 확장 MVP (2009년)

개선:
  - 프로 사진 서비스 (무료, 나중에 유료)
  - 리뷰 시스템
  - 호스트/게스트 프로필
  - 결제 에스크로 (신뢰 구축)

성장 해킹:
  - Craigslist 통합 (자동 포스팅)
  - 이메일 공유 (친구 초대)
  - 소셜 공유 (Facebook)

결과:
  - 월 예약: 100 → 1,000
  - 성장률: 100%/월
  - 학습: 신뢰가 핵심 (리뷰, 프로필, 사진)

성과​

2011년: $100M 펀딩 (Series B)
2015년: $25.5B 평가액
2021년: IPO, $100B+ 시가총액

교훈​

1. 수동으로 시작해도 OK:
   - 직접 사진 촬영 (확장 불가해도)
   - 학습이 목적, 효율성은 나중에

2. 하나의 이벤트에 집중:
   - 컨퍼런스 타겟
   - 호텔 만실 시기
   - 니치에서 시작 → 확장

3. 신뢰 구축이 핵심:
   - 리뷰, 프로필, 사진
   - 낯선 이의 집에 묵는 심리적 장벽 극복

4. 기존 플랫폼 활용:
   - Craigslist 통합
   - 초기 트래픽 확보

사례 3: Instagram MVP​

초기 가설 (Burbn, 2010년)​

"사람들은 체크인과 사진 공유를 원한다."
(Foursquare + 사진)

Pivot 과정​

1단계: Burbn MVP (2010년 3월)

기능:
  - 위치 체크인
  - 친구와 계획 공유
  - 사진 공유
  - 댓글, 좋아요

문제:
  - 너무 복잡 (기능 과다)
  - 사용자 혼란 ("어디서부터?")
  - Foursquare와 중복

사용 패턴 분석:
  - 체크인: 20% 사용
  - 사진 공유: 80% 사용 ← 핵심 발견!

학습: 사진만 집중해야 한다

2단계: Instagram MVP (2010년 10월)

Pivot: Burbn → Instagram
  - 모든 기능 제거
  - 사진 공유만 유지

핵심 기능 (MVP):
  - 사진 업로드
  - 필터 (11개)
  - 피드 (시간순)
  - 좋아요, 댓글
  - 팔로우

개발 기간: 8주 (Burbn 기반)

차별화:
  - 빠름 (업로드 <10초)
  - 아름다움 (필터)
  - 간단함 (3탭만: 피드, 카메라, 프로필)

출시 전략:
  - iOS만 (집중)
  - 인플루언서 시드 (사진작가, 디자이너)
  - Product Hunt, Hacker News

결과​

Day 1: 25,000 가입
Week 1: 100,000 가입
Month 1: 1M 가입

성장률: 300%/월

문제:
  - 서버 과부하 (AWS 긴급 증설)
  - 2명 팀 (Kevin, Mike)
  - 24/7 근무

2012년: Facebook이 $1B에 인수
2021년: 1B+ 사용자

교훈​

1. 덜어내는 용기:
   - 복잡한 Burbn → 간단한 Instagram
   - 하나의 핵심 가치에 집중

2. 데이터 기반 결정:
   - 사용 패턴 분석 (80% 사진)
   - 가설보다 현실

3. 단순함의 힘:
   - 3탭만
   - 학습 곡선 없음
   - 즉시 사용 가능

4. 차별화:
   - 필터 (아름다운 사진)
   - 속도 (빠른 업로드)
   - 경쟁사 대비 명확한 우위

마치며​

MVP는 완벽한 제품을 만드는 것이 아니라, 빠르게 학습하고 올바른 제품을 찾아가는 과정입니다.
