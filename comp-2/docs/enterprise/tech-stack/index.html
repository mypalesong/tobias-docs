5. 기술 스택

개요​

기술 스택 선택은 제품의 성능, 확장성, 개발 속도, 유지보수 비용에 직접적인 영향을 미칩니다. 엔터프라이즈 앱은 안정성, 보안, 확장성을 최우선으로 고려해야 합니다.

아키텍처 설계​

아키텍처 패턴​

1. 모놀리식 (Monolithic)​

구조:
  단일 코드베이스, 단일 배포 단위

장점:
  - 간단한 개발 (초기)
  - 쉬운 배포
  - 낮은 운영 복잡도
  - 빠른 로컬 개발

단점:
  - 확장성 제한
  - 기술 스택 고정
  - 큰 코드베이스 (유지보수 어려움)
  - 배포 위험 (전체 재시작)

적합한 경우:
  - MVP, 초기 스타트업
  - 작은 팀 (<10명)
  - 명확한 도메인
  - 빠른 출시 필요

예시:
  Ruby on Rails: Basecamp, GitHub (초기)
  Django: Instagram (초기)
  Laravel: Laravel Nova

- 공유 데이터베이스
- 직접 함수 호출

2. 마이크로서비스 (Microservices)​

구조:
  독립적인 서비스들의 조합

장점:
  - 독립 배포
  - 기술 스택 자유
  - 수평 확장 용이
  - 팀 독립성

단점:
  - 높은 운영 복잡도
  - 네트워크 오버헤드
  - 데이터 일관성 어려움
  - 디버깅 복잡

적합한 경우:
  - 대규모 애플리케이션
  - 큰 팀 (>50명)
  - 복잡한 도메인
  - 높은 확장성 필요

예시:
  Netflix: 수백 개 마이크로서비스
  Uber: 2,000+ 마이크로서비스
  Amazon: 서비스 지향 아키텍처 (SOA)

- 각자 데이터베이스

3. 모듈러 모놀리스 (Modular Monolith)​

구조:
  모놀리식 + 모듈화

장점:
  - 모놀리식의 단순함
  - 모듈 간 명확한 경계
  - 필요 시 마이크로서비스 전환 용이
  - 중간 복잡도

단점:
  - 규율 필요 (모듈 경계 유지)
  - 모놀리식 한계 여전히 존재
  - 팀 간 조정 필요

적합한 경우:
  - 성장하는 스타트업
  - 중간 규모 팀 (10-50명)
  - 미래 확장 고려
  - 점진적 전환 계획

예시:
  Shopify: 모듈러 모놀리스
  GitHub: Rails 모놀리스 (모듈화)
  Basecamp: Modular Rails

- 단일 프로세스
- 모듈 간 명확한 인터페이스
- 공유 또는 분리된 DB

추천 아키텍처​

  추천: 모놀리식
  이유:

  이유:
    - 확장성 준비
    - 팀 성장 대응
    - 점진적 전환 가능

  추천: 마이크로서비스 (필요 시)
  이유:
    - 높은 확장성
    - 팀 독립성
    - 복잡한 비즈니스 로직

주의:
  - 과도한 엔지니어링 방지
  - 비즈니스 니즈 우선
  - 필요할 때 전환

React 코드 예시

// 작업 리스트 컴포넌트
import React from 'react';
import { useQuery, useMutation } from 'react-query';
import { getTasks, createTask } from './api';

function TaskList() {
  const { data: tasks, isLoading } = useQuery('tasks', getTasks);
  const mutation = useMutation(createTask);

  const handleCreate = async (title: string) => {
    await mutation.mutateAsync({ title });
  };

  if (isLoading) return <div>로딩 중...</div>;

  return (
    <div className="task-list">
      {tasks.map(task => (
        <TaskItem key={task.id} task={task} />
      ))}
      <CreateTaskForm onSubmit={handleCreate} />
    </div>
  );
}

export default TaskList;

NestJS 코드 예시

// tasks.controller.ts
import { Controller, Get, Post, Body } from '@nestjs/common';
import { TasksService } from './tasks.service';
import { CreateTaskDto } from './dto/create-task.dto';

@Controller('tasks')
export class TasksController {
  constructor(private tasksService: TasksService) {}

  @Get()
  async findAll() {
    return this.tasksService.findAll();
  }

  @Post()
  async create(@Body() createTaskDto: CreateTaskDto) {
    return this.tasksService.create(createTaskDto);
  }
}

// tasks.service.ts
import { Injectable } from '@nestjs/common';
import { PrismaService } from '../prisma/prisma.service';

@Injectable()
export class TasksService {
  constructor(private prisma: PrismaService) {}

  async findAll() {
    return this.prisma.task.findMany({
      include: { assignee: true }
    });
  }

  async create(data: CreateTaskDto) {
    return this.prisma.task.create({ data });
  }
}

# main.py
from fastapi import FastAPI, Depends
from sqlalchemy.orm import Session
from . import models, schemas, database

app = FastAPI()

@app.get("/tasks", response_model=list[schemas.Task])
def get_tasks(db: Session = Depends(database.get_db)):
    return db.query(models.Task).all()

@app.post("/tasks", response_model=schemas.Task)
def create_task(
    task: schemas.TaskCreate,
    db: Session = Depends(database.get_db)
):
    db_task = models.Task(**task.dict())
    db.add(db_task)
    db.commit()
    db.refresh(db_task)
    return db_task

# schemas.py
from pydantic import BaseModel
from datetime import datetime

class TaskCreate(BaseModel):
    title: str
    description: str | None = None
    assignee_id: int | None = None

class Task(TaskCreate):
    id: int
    created_at: datetime
    updated_at: datetime

    class Config:
        orm_mode = True

JWT 구현 예시 (NestJS)

// auth.service.ts
import { Injectable } from '@nestjs/common';
import { JwtService } from '@nestjs/jwt';
import * as bcrypt from 'bcrypt';

@Injectable()
export class AuthService {
  constructor(
    private jwtService: JwtService,
    private usersService: UsersService,
  ) {}

  async validateUser(email: string, password: string) {
    const user = await this.usersService.findByEmail(email);
    if (user && await bcrypt.compare(password, user.passwordHash)) {
      const { passwordHash, ...result } = user;
      return result;
    }
    return null;
  }

  async login(user: any) {
    const payload = { email: user.email, sub: user.id };

    return {
      accessToken: this.jwtService.sign(payload, { expiresIn: '15m' }),
      refreshToken: this.jwtService.sign(payload, { expiresIn: '7d' }),
    };
  }

  async refreshToken(refreshToken: string) {
    const payload = this.jwtService.verify(refreshToken);
    const newPayload = { email: payload.email, sub: payload.sub };

    return {
      accessToken: this.jwtService.sign(newPayload, { expiresIn: '15m' }),
    };
  }
}

확장성​

수직 확장 (Vertical Scaling)​

방법: 서버 사양 업그레이드

장점:
  - 간단 (코드 변경 불필요)
  - 낮은 복잡도

단점:
  - 한계 있음 (최대 사양)
  - 비쌈
  - 단일 장애 지점 (SPOF)

예시:
  - CPU: 2 core → 8 core
  - RAM: 4GB → 32GB
  - 비용: 2배 증가

수평 확장 (Horizontal Scaling)​

방법: 서버 수 증가

장점:
  - 무한 확장 가능
  - 고가용성
  - 비용 효율적 (작은 인스턴스 여러 개)

단점:
  - 복잡도 증가
  - 상태 관리 어려움
  - 데이터 일관성

구현:
  - 로드 밸런서 (ALB, NLB)
  - 세션 스토어 (Redis)
  - 데이터베이스 복제
  - CDN (정적 자산)

예시:
  - 서버: 1대 → 10대
  - RPS: 1,000 → 10,000

캐싱 전략​

1. 브라우저 캐시:
   방법: Cache-Control 헤더
   대상: 정적 자산 (JS, CSS, 이미지)
   TTL: 1년

   Cache-Control: public, max-age=31536000, immutable

2. CDN 캐시:
   방법: CloudFront, Cloudflare
   대상: 정적 자산, API (일부)
   TTL: 1시간-1일

   이점:
     - 빠른 응답 (엣지 위치)
     - Origin 부하 감소
     - 글로벌 사용자 지원

3. 애플리케이션 캐시:
   방법: Redis, Memcached
   대상: DB 쿼리 결과, 세션
   TTL: 5분-1시간

   예시:
     - 사용자 정보: 15분
     - 작업 목록: 5분
     - 설정: 1시간

4. 데이터베이스 쿼리 캐시:
   방법: PostgreSQL 쿼리 캐시
   대상: 자주 실행되는 쿼리
   자동: DB가 관리

캐시 무효화:
  - 시간 기반 (TTL)
  - 이벤트 기반 (데이터 변경 시)
  - 수동 (관리자)

Redis 캐싱 예시

import { Injectable } from '@nestjs/common';
import { Redis } from 'ioredis';

@Injectable()
export class CacheService {
  private redis: Redis;

  constructor() {
    this.redis = new Redis(process.env.REDIS_URL);
  }

  async get<T>(key: string): Promise<T | null> {
    const value = await this.redis.get(key);
    return value ? JSON.parse(value) : null;
  }

  async set(key: string, value: any, ttl: number = 300): Promise<void> {
    await this.redis.setex(key, ttl, JSON.stringify(value));
  }

  async del(key: string): Promise<void> {
    await this.redis.del(key);
  }
}

// 사용 예시
@Injectable()
export class TasksService {
  constructor(
    private prisma: PrismaService,
    private cache: CacheService,
  ) {}

  async findAll(userId: number) {
    const cacheKey = `tasks:user:${userId}`;

    // 캐시 확인
    let tasks = await this.cache.get(cacheKey);

    if (!tasks) {
      // 캐시 미스: DB 조회
      tasks = await this.prisma.task.findMany({
        where: { assigneeId: userId }
      });

      // 캐시 저장 (5분)
      await this.cache.set(cacheKey, tasks, 300);
    }

    return tasks;
  }

  async create(data: CreateTaskDto) {
    const task = await this.prisma.task.create({ data });

    // 캐시 무효화
    await this.cache.del(`tasks:user:${data.assigneeId}`);

    return task;
  }
}

데이터베이스 최적화​

인덱싱:
  목적: 쿼리 속도 향상
  대상:
    - WHERE 절 컬럼
    - JOIN 컬럼
    - ORDER BY 컬럼

  주의:
    - 쓰기 성능 저하
    - 너무 많은 인덱스 방지

  예시:
    CREATE INDEX idx_tasks_assignee ON tasks(assignee_id);
    CREATE INDEX idx_tasks_status_due ON tasks(status, due_date);

읽기 복제 (Read Replica):
  목적: 읽기 분산
  구조:
    - Master: 쓰기
    - Replica 1-3: 읽기

  이점:
    - 읽기 처리량 증가
    - 고가용성

  주의:
    - 복제 지연 (Replication Lag)
    - 일관성 문제

연결 풀링 (Connection Pooling):
  목적: DB 연결 재사용
  설정:
    - 최소: 10
    - 최대: 100
    - 유휴 타임아웃: 10초

  이점:
    - 연결 오버헤드 감소
    - 리소스 효율적

쿼리 최적화:
  - N+1 문제 해결 (JOIN, include)
  - SELECT *  방지 (필요한 컬럼만)
  - LIMIT, OFFSET (페이지네이션)
  - 인덱스 사용 (EXPLAIN ANALYZE)

로그 레벨:
  - ERROR: 에러 발생
  - WARN: 경고 (잠재적 문제)
  - INFO: 중요 이벤트 (사용자 로그인)
  - DEBUG: 디버그 정보
  - TRACE: 상세 추적

로그 형식:
  - JSON (구조화)
  - 타임스탬프, 레벨, 메시지
  - 요청 ID (추적)
  - 사용자 ID, IP (보안)

로그 저장:
  개발: 콘솔
  프로덕션: CloudWatch, Elasticsearch
  장기 보관: S3 (압축, 저렴)

보관 기간:
  - ERROR: 90일
  - WARN: 30일
  - INFO: 7일
  - DEBUG: 1일 (프로덕션 비활성)

로깅 예시 (NestJS)

import { Logger } from '@nestjs/common';

@Injectable()
export class TasksService {
  private readonly logger = new Logger(TasksService.name);

  async create(data: CreateTaskDto, userId: number) {
    this.logger.log(`Creating task: "${data.title}" for user ${userId}`);

    try {
      const task = await this.prisma.task.create({ data });

      this.logger.log(`Task created: ${task.id}`);
      return task;
    } catch (error) {
      this.logger.error(
        `Failed to create task for user ${userId}`,
        error.stack,
      );
      throw error;
    }
  }
}
