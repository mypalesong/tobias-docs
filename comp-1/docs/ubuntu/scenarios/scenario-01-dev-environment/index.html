시나리오 1: Julia의 풀스택 개발 환경 구축

Julia가 새로운 스타트업에 풀스택 개발자로 입사한 첫날, WSL 2로 완벽한 개발 환경을 구축합니다.

📌 상황 설명​

Julia는 스타트업 '테크노바'에 풀스택 개발자로 입사했습니다. 회사에서 지급받은 Windows 노트북으로 React 프론트엔드와 Node.js 백엔드를 모두 개발해야 합니다. 선배 개발자 태훈이 WSL 2를 추천했고, Julia는 오늘 개발 환경을 설정하기로 했습니다.

🎯 목표​

  • ✅ WSL 2에 Node.js 개발 환경 구축
  • ✅ VS Code와 WSL 통합
  • ✅ Git 및 SSH 키 설정
  • ✅ 실제 프로젝트 클론 및 실행
  • ✅ 프론트엔드 + 백엔드 동시 실행

💬 대화​

태훈: "Julia님 입사 축하해요! 오늘은 개발 환경부터 세팅해볼까요?"

Julia: "감사합니다! WSL이라는 게 정말 좋나요? 저는 지금까지 Windows에서 직접 Node.js 설치했었는데..."

태훈: "WSL 2가 훨씬 빠르고 Linux와 똑같은 환경이에요. Docker도 잘 돌고, 파일 감시 기능도 문제없어요. 한번 써보면 못 돌아갑니다!"

Julia: "좋아요! 바로 시작해봐요."

⚡ 실습​

1단계: WSL 2 및 Ubuntu 설치 확인​

태훈: "먼저 WSL이 설치되어 있는지 확인해볼게요."

# PowerShell에서 확인
wsl --list --verbose

Julia: "아, WSL 2와 Ubuntu가 이미 설치되어 있네요! IT팀에서 미리 설정해둔 것 같아요."

태훈: "완벽하네요! 바로 시작할 수 있겠어요."

2단계: Ubuntu 업데이트​

태훈: "먼저 시스템을 최신 상태로 업데이트합시다."

# Ubuntu 터미널 실행 (시작 메뉴에서 "Ubuntu")
# 또는 Windows Terminal에서

# 패키지 목록 업데이트
sudo apt update

# 설치된 패키지 업그레이드
sudo apt upgrade -y

Julia: "업데이트 중이에요... 완료!"

3단계: 필수 개발 도구 설치​

태훈: "개발에 필요한 기본 도구들을 설치할게요."

# 빌드 도구 설치
sudo apt install -y build-essential

# curl 설치 (이미 있을 수 있음)
sudo apt install -y curl wget git

# 설치 확인
git --version
curl --version

출력:

git version 2.34.1
curl 7.81.0

Julia: "Git이 잘 설치되었네요!"

4단계: Node.js 및 nvm 설치​

태훈: "Node.js는 nvm으로 설치하는 게 좋아요. 버전 관리가 쉽거든요."

# nvm 설치 스크립트 다운로드 및 실행
curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.39.0/install.sh | bash

# nvm 환경 변수 로드 (터미널 재시작 대신)
export NVM_DIR="$HOME/.nvm"
[ -s "$NVM_DIR/nvm.sh" ] && \. "$NVM_DIR/nvm.sh"

# 설치 확인
nvm --version

태훈: "이제 Node.js LTS 버전을 설치합시다."

# Node.js LTS 설치
nvm install --lts

# 현재 설치된 Node.js 버전 확인
nvm list

# Node.js 및 npm 버전 확인
node --version
npm --version

출력:

v20.10.0
10.2.3

Julia: "와! Node.js 20이 설치되었어요. 최신 버전이네요!"

태훈: "필요하면 여러 버전을 동시에 설치하고 전환할 수 있어요."

# 추가 예제: 다른 버전도 설치
nvm install 18
nvm install 16

# 버전 전환
nvm use 20

# 기본 버전 설정
nvm alias default 20

5단계: VS Code 및 WSL 확장 설치​

태훈: "이제 VS Code를 WSL과 연결해봅시다."

Windows에서:

  1. VS Code 다운로드 및 설치 (Windows용)
  2. VS Code 실행
  3. 확장 마켓플레이스에서 "WSL" 검색
  4. "WSL" 확장 설치 (Microsoft 제공)

WSL Ubuntu에서:

# 프로젝트 디렉토리 생성
mkdir -p ~/projects
cd ~/projects

# VS Code 실행 (WSL에서)
code .

Julia: "어? VS Code가 Windows에서 열리는데 WSL에 연결되었다고 나와요!"

태훈: "맞아요! 왼쪽 하단에 'WSL: Ubuntu' 표시가 보이죠? 이제 VS Code에서 WSL의 파일을 편집할 수 있어요."

6단계: Git 설정​

태훈: "Git을 설정하고 SSH 키를 생성합시다."

# Git 사용자 정보 설정
git config --global user.name "Julia Kim"
git config --global user.email "julia@technova.com"

# 기본 브랜치 이름 설정
git config --global init.defaultBranch main

# 에디터 설정 (VS Code)
git config --global core.editor "code --wait"

# 설정 확인
git config --list

Julia: "설정 완료했어요!"

태훈: "이제 GitHub용 SSH 키를 만들어봅시다."

# SSH 키 생성
ssh-keygen -t ed25519 -C "julia@technova.com"

# 엔터 3번 (기본 위치, 비밀번호 없음)

출력:

Generating public/private ed25519 key pair.
Enter file in which to save the key (/home/julia/.ssh/id_ed25519):
Enter passphrase (empty for no passphrase):
Enter same passphrase again:
Your identification has been saved in /home/julia/.ssh/id_ed25519
Your public key has been saved in /home/julia/.ssh/id_ed25519.pub
# SSH 에이전트 시작
eval "$(ssh-agent -s)"

# SSH 키 추가
ssh-add ~/.ssh/id_ed25519

# 공개 키 복사 (클립보드로)
cat ~/.ssh/id_ed25519.pub

태훈: "이 공개 키를 GitHub 설정에 추가하세요."

GitHub에서:

  1. 설정 → SSH and GPG keys
  2. New SSH key
  3. 복사한 키 붙여넣기
  4. Add SSH key

Julia: "추가했어요!"

# SSH 연결 테스트
ssh -T git@github.com

출력:

Hi Julia! You've successfully authenticated, but GitHub does not provide shell access.

Julia: "성공! GitHub에 연결되었어요!"

7단계: 프로젝트 클론 및 실행​

태훈: "이제 실제 프로젝트를 클론해봅시다."

# projects 디렉토리로 이동
cd ~/projects

# 프로젝트 클론 (SSH)
git clone git@github.com:technova/shop-platform.git

# 프로젝트 디렉토리로 이동
cd shop-platform

# 프로젝트 구조 확인
ls -la

출력:

.
..
.git
backend/
frontend/
README.md
docker-compose.yml

Julia: "오! 모노레포 구조네요. backend와 frontend가 분리되어 있어요."

8단계: 백엔드 설정 및 실행​

태훈: "먼저 백엔드부터 실행해볼까요?"

# 백엔드 디렉토리로 이동
cd backend

# 의존성 설치
npm install

# 환경 변수 파일 복사
cp .env.example .env

# .env 파일 편집 (VS Code로)
code .env

.env 파일:

PORT=3001
DATABASE_URL=postgresql://localhost:5432/shop
JWT_SECRET=your-secret-key
NODE_ENV=development
# 개발 서버 실행
npm run dev

출력:

[nodemon] starting `node src/server.js`
Server is running on http://localhost:3001
Database connected successfully

Julia: "백엔드가 3001 포트에서 실행 중이에요!"

9단계: 프론트엔드 설정 및 실행​

새로운 터미널 탭에서 (Ctrl + Shift + T):

# 프로젝트 루트에서 프론트엔드로 이동
cd ~/projects/shop-platform/frontend

# 의존성 설치
npm install

# 환경 변수 설정
cp .env.example .env.local
code .env.local

.env.local 파일:

NEXT_PUBLIC_API_URL=http://localhost:3001
NEXT_PUBLIC_APP_NAME=TechNova Shop
# 개발 서버 실행
npm run dev

출력:

ready - started server on 0.0.0.0:3000, url: http://localhost:3000
event - compiled client and server successfully

Julia: "프론트엔드도 3000 포트에서 실행 중이에요!"

태훈: "이제 Windows 브라우저에서 http://localhost:3000 열어보세요!"

Julia: "와! 사이트가 로드되었어요! Windows에서 WSL의 서버에 바로 접속되네요!"

10단계: VS Code에서 개발하기​

태훈: "이제 VS Code로 코드를 수정해봅시다."

# 프로젝트 루트에서 VS Code 열기
cd ~/projects/shop-platform
code .

VS Code에서:

  1. frontend/src/pages/index.tsx 열기
  2. 코드 수정:
// 수정 전
<h1>Welcome to TechNova Shop</h1>

// 수정 후
<h1>Welcome to TechNova Shop - Julia's First Edit!</h1>
  1. 파일 저장 (Ctrl + S)

Julia: "어? 저장하자마자 브라우저가 자동으로 새로고침되었어요!"

태훈: "맞아요! Hot Module Replacement(HMR) 기능이에요. WSL에서도 완벽하게 작동하죠."

11단계: 터미널 분할 및 로그 모니터링​

태훈: "VS Code의 통합 터미널을 사용하면 더 편해요."

VS Code에서:

  1. `Ctrl + `` (백틱) - 터미널 열기
  2. 터미널 분할 (우측 상단 분할 아이콘)
  3. 한쪽에서는 백엔드, 다른 쪽에서는 프론트엔드 실행

Julia: "이렇게 하니까 훨씬 편하네요! 모든 로그를 한눈에 볼 수 있어요."

✅ 검증​

Julia의 개발 환경이 완벽하게 설정되었는지 확인해봅시다:

# Node.js 및 npm 버전
node --version && npm --version

# Git 설정 확인
git config user.name
git config user.email

# SSH 연결 확인
ssh -T git@github.com

# 실행 중인 프로세스 확인
ps aux | grep node

# 포트 확인
netstat -tuln | grep -E '3000|3001'

모든 확인 사항:

  • ✅ Node.js 20.x 실행 중
  • ✅ Git 설정 완료
  • ✅ GitHub SSH 연결 성공
  • ✅ 백엔드 (3001) 실행 중
  • ✅ 프론트엔드 (3000) 실행 중
  • ✅ VS Code WSL 연결
  • ✅ Hot Reload 작동

🎓 배운 내용​

WSL 2 개발 환경​

  1. nvm으로 Node.js 관리

    nvm install <version>
    nvm use <version>
    nvm alias default <version>

  2. VS Code WSL 통합

    • WSL 확장 설치
    • code . 명령으로 실행
    • 완벽한 IntelliSense 및 디버깅

  3. Git 및 SSH 설정

    git config --global
    ssh-keygen -t ed25519
    ssh-add

  4. 포트 포워딩

    • WSL의 포트가 자동으로 Windows에 포워딩됨
    • localhost로 바로 접근 가능

프로젝트 구조​

shop-platform/
├── backend/
│   ├── src/
│   ├── package.json
│   └── .env
├── frontend/
│   ├── src/
│   ├── package.json
│   └── .env.local
└── docker-compose.yml

💡 실전 팁​

1. nvm 기본 버전 설정​

# 새 터미널마다 자동으로 LTS 사용
nvm alias default lts/*

2. npm 글로벌 패키지​

# 자주 사용하는 도구 글로벌 설치
npm install -g typescript
npm install -g nodemon
npm install -g pm2
npm install -g @nestjs/cli

3. VS Code 터미널 기본값을 WSL로​

VS Code 설정 (Ctrl + ,):

{
  "terminal.integrated.defaultProfile.windows": "Ubuntu (WSL)"
}

4. Git Credential Helper​

# Windows Credential Manager 사용
git config --global credential.helper "/mnt/c/Program\ Files/Git/mingw64/bin/git-credential-manager-core.exe"

5. WSL 메모리 제한 (.wslconfig)​

Windows에서 C:\Users\YourName\.wslconfig 생성:

[wsl2]
memory=8GB
processors=4
swap=2GB

6. 빠른 프로젝트 시작 스크립트​

~/.bashrc에 추가:

# 프로젝트 빠른 시작
alias proj="cd ~/projects/shop-platform"
alias start-backend="cd ~/projects/shop-platform/backend && npm run dev"
alias start-frontend="cd ~/projects/shop-platform/frontend && npm run dev"

# 개발 환경 한 번에 시작
dev() {
    proj
    code .
    # 백엔드와 프론트엔드를 백그라운드로 실행
    cd backend && npm run dev &
    cd ../frontend && npm run dev &
}

사용:

# 터미널에서
dev  # 모든 것을 한 번에 시작!

🐛 문제 해결​

npm install 시 권한 오류​

# npm 글로벌 디렉토리 변경
mkdir ~/.npm-global
npm config set prefix '~/.npm-global'

# PATH에 추가 (~/.bashrc에)
echo 'export PATH=~/.npm-global/bin:$PATH' >> ~/.bashrc
source ~/.bashrc

파일 변경 감지 안 됨​

# inotify 한도 증가
echo fs.inotify.max_user_watches=524288 | sudo tee -a /etc/sysctl.conf
sudo sysctl -p

포트가 이미 사용 중​

# 특정 포트를 사용하는 프로세스 찾기
lsof -i :3000

# 프로세스 종료
kill -9 <PID>
