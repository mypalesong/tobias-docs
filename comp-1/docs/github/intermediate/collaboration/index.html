협업 워크플로우

팀에서 Git과 GitHub를 효과적으로 사용하는 다양한 워크플로우를 배웁니다.

주요 워크플로우 비교​

워크플로우복잡도적합한 팀 크기배포 주기
GitHub Flow낮음소규모빈번함
Git Flow높음대규모정기적
Trunk-based중간모든 규모매우 빈번함
Forking중간오픈소스다양함

1. GitHub Flow​

가장 간단하고 널리 사용되는 워크플로우입니다.

원칙​

  1. main 브랜치는 항상 배포 가능한 상태
  2. 새 기능은 브랜치에서 개발
  3. Pull Request로 코드 리뷰
  4. main에 병합 후 즉시 배포

실습​

# 1. 최신 main 받기
git checkout main
git pull

# 2. feature 브랜치 생성
git checkout -b feature/add-payment

# 3. 작업 및 커밋
echo "payment code" > payment.js
git add payment.js
git commit -m "feat: add payment integration"

# 4. 푸시
git push -u origin feature/add-payment

# 5. PR 생성
gh pr create --title "Add payment integration" --body "Implements Stripe payment"

# 6. 리뷰 및 병합 후 브랜치 삭제
git checkout main
git pull
git branch -d feature/add-payment

브랜치 명명 규칙​

feature/user-authentication
feature/add-dark-mode
fix/login-bug
fix/memory-leak
hotfix/critical-security-issue
docs/update-readme
refactor/cleanup-auth

2. Git Flow​

복잡한 릴리스 관리가 필요한 프로젝트에 적합합니다.

브랜치 구조​

main (production)
├── develop (integration)
│   ├── feature/payment
│   ├── feature/notifications
│   └── feature/dashboard
├── release/v2.0
└── hotfix/security-patch

브랜치 타입​

main: 프로덕션 코드

  • 항상 안정적
  • 태그로 버전 관리

develop: 개발 브랜치

  • 다음 릴리스 준비
  • 모든 feature 병합

feature: 기능 개발

  • develop에서 분기
  • develop으로 병합

release: 릴리스 준비

  • develop에서 분기
  • main과 develop으로 병합

hotfix: 긴급 수정

  • main에서 분기
  • main과 develop으로 병합

실습​

# Git Flow 초기화
git flow init

# Feature 시작
git flow feature start user-profile
# ... 작업 ...
git flow feature finish user-profile

# Release 시작
git flow release start 1.0.0
# ... 버그 수정 ...
git flow release finish 1.0.0

# Hotfix
git flow hotfix start security-patch
# ... 수정 ...
git flow hotfix finish security-patch

수동으로 하려면:

# Feature 브랜치
git checkout develop
git checkout -b feature/user-profile
# ... 작업 ...
git checkout develop
git merge --no-ff feature/user-profile
git branch -d feature/user-profile

# Release 브랜치
git checkout develop
git checkout -b release/1.0.0
# ... 버그 수정 ...
git checkout main
git merge --no-ff release/1.0.0
git tag -a v1.0.0 -m "Release version 1.0.0"
git checkout develop
git merge --no-ff release/1.0.0
git branch -d release/1.0.0

3. Trunk-based Development​

매우 짧은 수명의 브랜치를 사용합니다.

원칙​

  • 모든 개발자가 trunk(main)에 자주 커밋
  • 브랜치 수명: 1-2일
  • Feature flags로 미완성 기능 숨김
  • CI/CD 필수

실습​

# 1. 짧은 브랜치 생성
git checkout -b short-lived-feature

# 2. 작은 변경사항 커밋
git commit -m "feat: add button (behind feature flag)"

# 3. 즉시 푸시 및 병합
git push -u origin short-lived-feature
gh pr create --title "Add button" --body "Small change, ready to merge"

# 4. 빠른 리뷰 후 병합
gh pr merge --squash

# 5. 로컬 정리
git checkout main
git pull
git branch -d short-lived-feature

Feature Flags 예시​

// config/features.js
export const features = {
  newPaymentFlow: process.env.FEATURE_NEW_PAYMENT === 'true',
  darkMode: process.env.FEATURE_DARK_MODE === 'true',
};

// 사용
import { features } from './config/features';

function PaymentPage() {
  if (features.newPaymentFlow) {
    return <NewPaymentFlow />;
  }
  return <OldPaymentFlow />;
}

4. Forking Workflow​

오픈소스 프로젝트에 주로 사용됩니다.

프로세스​

# 1. 원본 저장소 Fork (GitHub 웹에서)

# 2. Fork한 저장소 Clone
git clone https://github.com/YOUR_USERNAME/project.git
cd project

# 3. 원본 저장소를 upstream으로 추가
git remote add upstream https://github.com/ORIGINAL_OWNER/project.git

# 4. 브랜치 생성
git checkout -b feature/my-contribution

# 5. 작업 및 커밋
git add .
git commit -m "feat: add new feature"

# 6. 내 Fork에 푸시
git push -u origin feature/my-contribution

# 7. 원본 저장소에 PR 생성
gh pr create --repo ORIGINAL_OWNER/project

# 8. 원본 저장소의 최신 변경사항 동기화
git fetch upstream
git checkout main
git merge upstream/main
git push origin main

Upstream 동기화​

# 정기적으로 실행
git fetch upstream
git checkout main
git rebase upstream/main
git push origin main --force-with-lease

팀 협업 Best Practices​

1. 명확한 커밋 메시지​

# ❌ 나쁜 예
git commit -m "fix"
git commit -m "update"
git commit -m "changes"

# ✅ 좋은 예
git commit -m "fix: resolve login timeout issue"
git commit -m "feat: add user profile page"
git commit -m "docs: update API documentation"

2. 작은 PR​

# ❌ 나쁜 예: 1000줄 변경
# ✅ 좋은 예: 여러 개의 작은 PR

# PR 1: 기본 구조
# PR 2: UI 컴포넌트
# PR 3: 비즈니스 로직
# PR 4: 테스트

3. 코드 리뷰 문화​

## 좋은 리뷰 코멘트

✅ "이 부분은 별도 함수로 추출하면 어떨까요?"
✅ "null 체크를 추가하는 것이 안전할 것 같습니다."
✅ "좋은 구현입니다! 다만 성능을 위해 캐싱을 고려해보세요."

## 피해야 할 코멘트

❌ "이 코드는 별로네요."
❌ "왜 이렇게 했어요?"
❌ "다시 작성하세요."

4. 브랜치 보호 규칙​

# GitHub Settings → Branches → Branch protection rules

Required status checks:
  - ✅ Require status checks to pass before merging
  - ✅ Require branches to be up to date before merging

Required approvals:
  - ✅ Require pull request reviews before merging
  - Number of approvals: 2

Other:
  - ✅ Require conversation resolution before merging
  - ✅ Require linear history
  - ✅ Do not allow bypassing the above settings

도구 및 자동화​

Pre-commit Hooks​

# .git/hooks/pre-commit
#!/bin/sh

# 린트 검사
npm run lint || exit 1

# 테스트 실행
npm test || exit 1

# 커밋 메시지 검증
# (commitlint 사용)

CI/CD 파이프라인​

# .github/workflows/ci.yml
name: CI

on:
  pull_request:
    branches: [ main, develop ]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - name: Install dependencies
        run: npm install
      - name: Run tests
        run: npm test
      - name: Run linter
        run: npm run lint

자동 병합​

# .github/workflows/auto-merge.yml
name: Auto Merge

on:
  pull_request:
    types: [ labeled ]

jobs:
  auto-merge:
    if: contains(github.event.pull_request.labels.*.name, 'automerge')
    runs-on: ubuntu-latest
    steps:
      - uses: pascalgn/automerge-action@v0.15.6
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
