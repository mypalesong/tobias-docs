충돌 해결 (Conflict Resolution)

Git에서 충돌이 발생하는 이유와 효과적으로 해결하는 방법을 배웁니다.

충돌이란?​

두 개 이상의 브랜치가 같은 파일의 같은 부분을 다르게 수정했을 때 발생합니다. Git이 자동으로 병합할 수 없어 수동 개입이 필요합니다.

충돌 발생 시나리오​

시나리오 1: Merge 중 충돌​

# main 브랜치
git checkout main
echo "Hello from main" > file.txt
git add file.txt
git commit -m "Update from main"

# feature 브랜치
git checkout feature
echo "Hello from feature" > file.txt
git add file.txt
git commit -m "Update from feature"

# 병합 시도
git checkout main
git merge feature
# CONFLICT (content): Merge conflict in file.txt

시나리오 2: Pull 중 충돌​

# 원격에서 변경
# (다른 개발자가 푸시)

# 로컬에서도 같은 파일 변경
echo "Local changes" > file.txt
git add file.txt
git commit -m "Local update"

# Pull 시도
git pull
# CONFLICT (content): Merge conflict in file.txt

시나리오 3: Rebase 중 충돌​

git checkout feature
git rebase main
# CONFLICT (content): Merge conflict in file.txt

충돌 표시 이해하기​

충돌이 발생하면 파일에 다음과 같은 마커가 표시됩니다:

<<<<<<< HEAD
현재 브랜치의 내용
=======
병합하려는 브랜치의 내용
>>>>>>> feature-branch

예시:

function greet() {
<<<<<<< HEAD
  return "Hello from main";
=======
  return "Hello from feature";
>>>>>>> feature-branch
}

충돌 해결 방법​

방법 1: 수동 편집​

# 1. 충돌 파일 확인
git status

# 2. 파일 열어서 수정
code file.txt

# 3. 마커 제거하고 원하는 내용으로 수정
function greet() {
  return "Hello from both branches";
}

# 4. 스테이징
git add file.txt

# 5. 병합 완료
git commit

방법 2: 한쪽 선택​

# 현재 브랜치 (ours) 선택
git checkout --ours file.txt
git add file.txt

# 병합하려는 브랜치 (theirs) 선택
git checkout --theirs file.txt
git add file.txt

# 병합 완료
git commit

방법 3: Merge Tool 사용​

# 기본 merge tool 실행
git mergetool

# 특정 tool 지정
git mergetool --tool=vimdiff

IDE/에디터 도구 활용​

VS Code​

VS Code는 충돌을 시각적으로 표시하고 쉽게 해결할 수 있는 UI를 제공합니다:

<<<<<<< HEAD (Current Change)
현재 브랜치의 내용
=======
병합하려는 브랜치의 내용
>>>>>>> feature-branch (Incoming Change)

[Accept Current Change] [Accept Incoming Change]
[Accept Both Changes] [Compare Changes]

Git Merge Tool 설정​

# VS Code
git config --global merge.tool vscode
git config --global mergetool.vscode.cmd 'code --wait $MERGED'

# Sublime Merge
git config --global merge.tool smerge
git config --global mergetool.smerge.cmd 'smerge mergetool "$BASE" "$LOCAL" "$REMOTE" -o "$MERGED"'

# Beyond Compare
git config --global merge.tool bc
git config --global mergetool.bc.path "C:/Program Files/Beyond Compare 4/bcomp.exe"

충돌 해결 전략​

1. 단순 충돌​

// 충돌 발생
<<<<<<< HEAD
const MAX_COUNT = 10;
=======
const MAX_COUNT = 20;
>>>>>>> feature-branch

// 해결: 더 큰 값 선택
const MAX_COUNT = 20;

2. 두 변경사항 모두 유지​

// 충돌 발생
<<<<<<< HEAD
function setupApp() {
  initAuth();
}
=======
function setupApp() {
  initDatabase();
}
>>>>>>> feature-branch

// 해결: 두 초기화 모두 필요
function setupApp() {
  initAuth();
  initDatabase();
}

3. 완전히 새로운 해결책​

// 충돌 발생
<<<<<<< HEAD
const config = { api: 'v1' };
=======
const config = { endpoint: '/api' };
>>>>>>> feature-branch

// 해결: 두 설정 병합 + 개선
const config = {
  api: 'v1',
  endpoint: '/api/v1'
};

Rebase 충돌 해결​

# Rebase 시작
git rebase main
# CONFLICT (content): Merge conflict in file.txt

# 충돌 해결
code file.txt
# ... 수정 ...
git add file.txt

# Rebase 계속
git rebase --continue

# 또는 포기
git rebase --abort

# 또는 현재 커밋 건너뛰기
git rebase --skip

대규모 충돌 처리​

전략 1: 단계별 병합​

# 큰 feature 브랜치를 작은 단위로 병합
git checkout feature
git rebase -i main

# 커밋을 그룹화하여 하나씩 병합

전략 2: Rerere (Reuse Recorded Resolution)​

# rerere 활성화
git config --global rerere.enabled true

# 동일한 충돌이 다시 발생하면 자동으로 해결
git merge feature
# (충돌 해결)
git add .
git commit

# 나중에 같은 충돌 발생 시 자동 해결됨

충돌 디버깅​

충돌 원인 찾기​

# 양쪽 버전 비교
git diff HEAD...MERGE_HEAD

# 공통 조상과 비교
git diff $(git merge-base HEAD MERGE_HEAD) HEAD
git diff $(git merge-base HEAD MERGE_HEAD) MERGE_HEAD

# 충돌 파일만 보기
git diff --name-only --diff-filter=U

누가 변경했는지 확인​

# 각 라인의 마지막 수정자 확인
git blame file.txt

# 두 브랜치의 로그 비교
git log --oneline --graph --all

실습 예제​

예제 1: 간단한 충돌 해결​

# 저장소 초기화
git init conflict-demo
cd conflict-demo

# main 브랜치에서 파일 생성
echo "Version 1" > file.txt
git add file.txt
git commit -m "Initial commit"

# feature 브랜치 생성
git checkout -b feature

# 두 브랜치에서 같은 파일 수정
git checkout main
echo "Main version" > file.txt
git commit -am "Update on main"

git checkout feature
echo "Feature version" > file.txt
git commit -am "Update on feature"

# 병합 시도
git checkout main
git merge feature
# 충돌 발생!

# 충돌 확인
cat file.txt

# 수동 해결
echo "Merged version" > file.txt
git add file.txt
git commit -m "Resolve conflict"

예제 2: 여러 파일 충돌​

# 충돌 파일 목록
git status

# 각 파일 해결
for file in $(git diff --name-only --diff-filter=U); do
  echo "Resolving $file"
  git checkout --ours "$file"  # 또는 --theirs
  git add "$file"
done

# 커밋
git commit

예제 3: Rebase 충돌 해결​

# Rebase 시작
git rebase main

# 충돌 해결
git status
code conflicted-file.txt
git add conflicted-file.txt
git rebase --continue

# 추가 충돌이 있다면 반복
# 모든 커밋이 적용될 때까지

충돌 예방 팁​

1. 자주 Pull​

# 매일 아침 최신 코드 받기
git checkout main
git pull
git checkout feature
git rebase main

2. 작은 단위로 작업​

# 큰 기능을 작은 PR로 분할
# 각각 빠르게 병합

3. 코드 영역 분리​

// ❌ 나쁜 예: 모두 같은 파일
// app.js - 1000 lines

// ✅ 좋은 예: 기능별로 분리
// auth.js
// database.js
// api.js

4. 커뮤니케이션​

# Slack/Discord에 알림
"file.txt를 수정 중입니다. 잠시 후 커밋할 예정입니다."

고급 도구​

Git Diff3​

# diff3 스타일 활성화 (3-way diff)
git config --global merge.conflictstyle diff3

충돌 표시가 더 상세해집니다:

<<<<<<< HEAD
현재 브랜치
||||||| merged common ancestors
공통 조상
=======
병합하려는 브랜치
>>>>>>> feature-branch

Git Log로 충돌 분석​

# 두 브랜치의 차이 커밋 보기
git log --merge --left-right HEAD...MERGE_HEAD

# 충돌 파일의 히스토리
git log --merge -p file.txt

일반적인 실수​

1. ❌ 충돌 마커 커밋​

// 잘못된 커밋
function example() {
<<<<<<< HEAD
  return "oops";
=======
  return "forgot to resolve";
>>>>>>> feature
}

해결: 커밋 전 항상 git diff로 확인

2. ❌ 잘못된 선택​

# 무조건 --ours 사용
git checkout --ours .  # 위험!

해결: 각 충돌을 개별적으로 검토

3. ❌ 테스트 없이 병합​

# 충돌 해결 후 바로 커밋
git add .
git commit  # 위험!

해결: 항상 테스트 후 커밋

git add .
npm test
git commit

충돌 해결 체크리스트​

  • 충돌 파일 모두 확인
  • 코드 문맥 이해
  • 양쪽 변경사항 검토
  • 적절한 해결책 선택
  • 마커 완전히 제거
  • 코드 실행/테스트
  • 린트 통과 확인
  • 팀원에게 리뷰 요청
