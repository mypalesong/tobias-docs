Git Rebase 소개

Rebase를 사용하여 깔끔한 커밋 히스토리를 유지하는 방법을 배웁니다.

Rebase란?​

Rebase는 커밋들을 다른 베이스 위로 재배치하는 것입니다. 히스토리를 선형으로 유지하여 더 깔끔하게 만들 수 있습니다.

Merge vs Rebase​

Merge​

git merge feature-branch

히스토리:

*   Merge branch 'feature'
|\
| * Feature commit 2
| * Feature commit 1
|/
* Main commit

Rebase​

git rebase main

히스토리:

* Feature commit 2
* Feature commit 1
* Main commit

기본 Rebase​

사용법​

# feature 브랜치에서
git checkout feature-branch
git rebase main

# 또는 한 줄로
git rebase main feature-branch

단계별 예제​

# 1. 상황 설정
git checkout main
echo "main work" > main.txt
git add main.txt
git commit -m "Main work"

git checkout -b feature
echo "feature work" > feature.txt
git add feature.txt
git commit -m "Feature work"

# main에 새 커밋 추가
git checkout main
echo "more main work" >> main.txt
git commit -am "More main work"

# 2. Rebase 실행
git checkout feature
git rebase main

# 3. 결과 확인
git log --oneline --graph

Interactive Rebase​

히스토리를 수정할 수 있는 강력한 도구입니다.

사용법​

# 마지막 3개 커밋을 대화형으로 수정
git rebase -i HEAD~3

# 또는 특정 커밋까지
git rebase -i abc123

명령어​

에디터에서 사용 가능한 명령어:

  • pick (p): 커밋 유지
  • reword (r): 커밋 메시지 수정
  • edit (e): 커밋 수정
  • squash (s): 이전 커밋과 합치기
  • fixup (f): squash와 같지만 메시지 버림
  • drop (d): 커밋 삭제
  • exec (x): 쉘 명령 실행

예제: 커밋 합치기 (Squash)​

# 1. Interactive rebase 시작
git rebase -i HEAD~3

# 2. 에디터에서:
pick abc123 First commit
squash def456 Second commit
squash ghi789 Third commit

# 3. 새 커밋 메시지 작성
Combined commit message

- Feature A
- Feature B
- Feature C

# 4. 저장 및 종료

예제: 커밋 메시지 수정​

# 1. Interactive rebase
git rebase -i HEAD~3

# 2. 에디터에서:
reword abc123 Old commit message
pick def456 Keep this message
pick ghi789 Keep this too

# 3. 새 메시지 입력
feat: add user authentication (improved message)

예제: 커밋 순서 변경​

# 1. Interactive rebase
git rebase -i HEAD~3

# 2. 에디터에서 줄 순서 변경:
pick ghi789 Third commit
pick abc123 First commit
pick def456 Second commit

Rebase 충돌 해결​

# 1. Rebase 시작
git rebase main
# CONFLICT (content): Merge conflict in file.txt

# 2. 충돌 파일 확인
git status

# 3. 충돌 해결
code file.txt
# ... 수정 ...

# 4. 스테이징
git add file.txt

# 5. Rebase 계속
git rebase --continue

# 또는 포기
git rebase --abort

# 또는 현재 커밋 건너뛰기
git rebase --skip

실습 예제​

예제 1: Feature 브랜치 최신화​

# 1. Feature 브랜치에서 작업
git checkout -b feature/new-ui
echo "UI changes" > ui.txt
git add ui.txt
git commit -m "feat: update UI"

# 2. Main에 새 커밋이 추가됨
git checkout main
git pull

# 3. Feature 브랜치를 main 위로 rebase
git checkout feature/new-ui
git rebase main

# 4. 강제 푸시 (이미 푸시한 경우)
git push --force-with-lease

예제 2: 여러 커밋을 하나로 합치기​

# 1. 여러 작은 커밋 생성
git commit -m "Add function"
git commit -m "Fix typo"
git commit -m "Add tests"
git commit -m "Fix tests"

# 2. Interactive rebase로 합치기
git rebase -i HEAD~4

# 3. 에디터에서:
pick abc123 Add function
fixup def456 Fix typo
fixup ghi789 Add tests
fixup jkl012 Fix tests

# 결과: 하나의 깔끔한 커밋

예제 3: 커밋 분할​

# 1. 너무 큰 커밋 선택
git rebase -i HEAD~3

# 2. 에디터에서:
edit abc123 Big commit to split
pick def456 Other commit

# 3. 커밋 되돌리기
git reset HEAD^

# 4. 작은 단위로 커밋
git add file1.txt
git commit -m "Part 1: Add feature A"

git add file2.txt
git commit -m "Part 2: Add feature B"

# 5. Rebase 계속
git rebase --continue

Golden Rule of Rebase​

위험

절대 규칙: 이미 푸시한 커밋을 rebase하지 마세요!

❌ 위험한 경우​

# main 브랜치에서 rebase (다른 사람도 사용 중)
git checkout main
git rebase feature  # 위험!

✅ 안전한 경우​

# 내 개인 feature 브랜치에서 rebase
git checkout my-feature
git rebase main  # 안전

Rebase 옵션​

--onto​

다른 베이스로 변경:

# topic 브랜치를 old-base에서 new-base로 이동
git rebase --onto new-base old-base topic

--autosquash​

자동으로 fixup/squash:

# Commit 1
git commit -m "feat: add feature"

# Commit 2 (fixup 표시)
git commit -m "fixup! feat: add feature"

# Rebase 시 자동으로 합침
git rebase -i --autosquash HEAD~2

--exec​

각 커밋마다 명령 실행:

# 각 커밋마다 테스트 실행
git rebase -i --exec "npm test" main

Pull with Rebase​

# Pull 시 merge 대신 rebase
git pull --rebase

# 기본값으로 설정
git config --global pull.rebase true

# 로컬 변경사항 자동 stash
git config --global rebase.autostash true

고급 패턴​

패턴 1: 클린업 Before PR​

# PR 생성 전 커밋 정리
git rebase -i main

# Squash, reword, reorder
# 결과: 깔끔한 히스토리

패턴 2: 작은 단위로 Rebase​

# 한 번에 하나씩 커밋 rebase
git rebase --interactive --rebase-merges main

패턴 3: Fixup Commits​

# 1. 원래 커밋
git commit -m "feat: add login"

# 2. 나중에 수정사항 발견
git commit --fixup HEAD

# 3. Rebase로 자동 합침
git rebase -i --autosquash HEAD~2

일반적인 문제​

문제 1: 충돌이 너무 많음​

# 해결: Rebase 중단하고 merge 사용
git rebase --abort
git merge main

문제 2: 이미 푸시한 브랜치 rebase​

# 강제 푸시 필요 (팀과 협의 후)
git push --force-with-lease

# 팀원들은 다음 명령 필요:
git fetch
git reset --hard origin/branch-name

문제 3: Rebase 중 잘못된 선택​

# Reflog에서 이전 상태 찾기
git reflog

# 원하는 시점으로 복구
git reset --hard HEAD@{2}

Rebase Checklist​

  • 개인 브랜치에서만 사용
  • 푸시 전에 rebase
  • 충돌을 각각 해결
  • Rebase 후 테스트 실행
  • Force push는 신중하게
  • 팀원과 소통

유용한 Alias​

# Rebase 관련 alias
git config --global alias.rb 'rebase'
git config --global alias.rbi 'rebase -i'
git config --global alias.rbc 'rebase --continue'
git config --global alias.rba 'rebase --abort'

# 사용
git rbi HEAD~3
git rbc

Merge vs Rebase 선택 가이드​

상황권장 방법
Feature 브랜치 최신화Rebase
Main에 Feature 병합Merge
개인 브랜치 정리Rebase
팀 브랜치 병합Merge
히스토리 보존 중요Merge
히스토리 단순화 중요Rebase
