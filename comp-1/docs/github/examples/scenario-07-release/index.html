시나리오 7: 릴리스 데이

현우 PM이 v2.0.0 릴리스를 준비하고 배포하는 하루를 따라가봅니다.

📌 상황 설명​

3개월간의 개발 끝에, 드디어 메이저 버전 v2.0.0을 릴리스할 날이 왔습니다. 현우는 릴리스 매니저로서 안전하고 체계적인 배포를 진행해야 합니다.

🎯 목표​

  • Release 브랜치 생성 및 관리
  • Semantic Versioning 이해
  • Changelog 자동 생성
  • GitHub Release 생성
  • 자동화된 배포 파이프라인
  • Rollback 계획 수립

💬 대화​

현우 (아침 9시): "팀 여러분, 오늘이 v2.0.0 릴리스 데이입니다! 계획대로 진행하겠습니다."

수현 (DevOps): "CI/CD 파이프라인 준비 완료했습니다. 체크리스트 확인해주세요."

준호: "마지막 버그 수정 PR도 어제 머지됐습니다. 준비됐어요!"

민지: "릴리스 과정을 배우고 싶어요. 제가 도울 수 있는 게 있을까요?"

현우: "좋아요! 같이 진행하면서 배워봅시다."

Day -7: 릴리스 계획 (일주일 전)​

1단계: 릴리스 체크리스트 작성​

# v2.0.0 Release Checklist

## Pre-Release (1 week before)
- [ ] Feature freeze
- [ ] Create release branch
- [ ] Update dependencies
- [ ] Run security audit
- [ ] Performance testing
- [ ] Documentation review

## Release Day
- [ ] Final testing on staging
- [ ] Update version numbers
- [ ] Generate changelog
- [ ] Create GitHub release
- [ ] Deploy to production
- [ ] Smoke testing
- [ ] Monitor for 24h

## Post-Release
- [ ] Announcement
- [ ] Update marketing site
- [ ] Close related issues
- [ ] Retrospective meeting

Day -7: Release 브랜치 생성​

# 최신 develop 가져오기
git checkout develop
git pull origin develop

# Release 브랜치 생성
git checkout -b release/v2.0.0

# 원격에 푸시
git push -u origin release/v2.0.0

현우: "Feature freeze! 이제부터 새 기능은 v2.1.0으로 연기됩니다."

2단계: 버전 번호 업데이트​

# package.json
npm version 2.0.0 --no-git-tag-version

# version.py (Python 프로젝트)
echo "__version__ = '2.0.0'" > src/version.py

# build.gradle (Java 프로젝트)
# version = '2.0.0'
# 버전 업데이트 커밋
git add package.json

  --head release/v2.0.0 \
  --title "Release v2.0.0" \

# Release v2.0.0

## Testing

- [x] Integration tests pass
- [x] E2E tests pass
- [x] Performance tests pass
- [x] Security scan clean
- [x] Staging deployment successful

## Approvals Required
- [ ] @junho (Tech Lead)
- [ ] @suhyun (DevOps)
- [ ] @current-ceo (CEO Approval)

## Deployment Plan
1. Merge to main
2. Create Git tag v2.0.0
3. Create GitHub Release
4. Deploy to production (blue-green)
5. Monitor for 1 hour
6. Switch traffic to new version
7. Monitor for 24 hours

## Rollback Plan

1. Switch traffic back to v1.5.0

3. Fix and re-deploy

)"

9단계: 자동 배포​

.github/workflows/release.yml:

name: Release Deployment

on:
  release:
    types: [published]

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
        with:
          ref: ${{ github.event.release.tag_name }}

      - name: Setup Node.js
        uses: actions/setup-node@v3
        with:
          node-version: '20'

      - name: Install dependencies
        run: npm ci

      - name: Run tests
        run: npm test

      - name: Build
        run: npm run build
        env:
          NODE_ENV: production

      - name: Deploy to production
        env:
          AWS_ACCESS_KEY_ID: ${{ secrets.AWS_ACCESS_KEY_ID }}
          AWS_SECRET_ACCESS_KEY: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
        run: |
          npm run deploy:production

      - name: Slack notification
        uses: 8398a7/action-slack@v3
        with:
          status: ${{ job.status }}
          text: 'v2.0.0 deployed to production!'
          webhook_url: ${{ secrets.SLACK_WEBHOOK }}

      - name: Create deployment
        uses: chrnorm/deployment-action@v2
        with:
          token: ${{ secrets.GITHUB_TOKEN }}
          environment: production
          ref: ${{ github.event.release.tag_name }}

🎓 배운 내용​

Semantic Versioning​

MAJOR.MINOR.PATCH
  │     │     │
  │     │     └─ 버그 수정 (1.0.0 → 1.0.1)
  │     └─────── 기능 추가 (1.0.0 → 1.1.0)
  └───────────── Breaking changes (1.0.0 → 2.0.0)

Release 브랜치 전략​

develop ─┬─ release/v2.0.0 ─┬─ main (v2.0.0)
         │                  │
         └─ feature/new     └─ develop (머지백)

배포 전략​

  1. Blue-Green: 구버전과 신버전 동시 운영
  2. Canary: 일부 사용자에게만 먼저 배포
  3. Rolling: 서버를 하나씩 순차적으로 업데이트
