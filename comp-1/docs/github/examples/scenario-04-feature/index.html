시나리오 4: 대규모 기능 개발

민지가 2주짜리 대형 프로젝트를 맡게 됩니다. 결제 시스템 전면 개편!

📌 상황 설명​

월요일 오전, 현우 PM이 민지에게 큰 프로젝트를 맡깁니다. 기존 결제 시스템을 Stripe으로 전환하는 작업입니다. 예상 기간은 2주이며, 이 기간 동안 main 브랜치는 계속 업데이트될 것입니다.

🎯 목표​

  • 장기 feature 브랜치 관리
  • 정기적인 업데이트 반영 (Rebase)
  • 작업을 작은 PR로 분할
  • 충돌 최소화 전략
  • 효과적인 코드 리뷰 받기

💬 대화​

현우 (PM): "민지님, 큰 프로젝트를 맡아주세요. Stripe 결제 시스템 통합입니다."

민지: "2주면... 꽤 큰 작업이네요. 어떻게 관리하는 게 좋을까요?"

준호: "큰 feature는 전략이 필요해요. 작게 나누고, 자주 동기화하고, 단계별로 PR을 만드는 게 핵심입니다."

⚡ 실습​

Day 1: 프로젝트 계획 및 브랜치 전략​

1단계: 작업 분할 계획​

준호: "먼저 전체 작업을 작은 단위로 나눠봅시다."

# Stripe 결제 시스템 통합 계획

## Phase 1: 기초 설정 (Day 1-2)
- [ ] Stripe SDK 설치
- [ ] 환경 설정
- [ ] 기본 구조 생성

## Phase 2: 백엔드 통합 (Day 3-5)
- [ ] Payment Intent API
- [ ] Webhook 설정
- [ ] 에러 핸들링

## Phase 3: 프론트엔드 (Day 6-8)
- [ ] 결제 폼 컴포넌트
- [ ] 카드 입력 UI
- [ ] 결제 플로우

## Phase 4: 테스트 및 문서화 (Day 9-10)
- [ ] E2E 테스트
- [ ] 단위 테스트
- [ ] API 문서

2단계: Feature 브랜치 생성​

# 최신 코드 받기
git checkout main
git pull origin main

# Long-lived feature 브랜치 생성
git checkout -b feature/stripe-integration

# 원격에 푸시 (백업 목적)
git push -u origin feature/stripe-integration

Day 1-2: Phase 1 시작​

3단계: 작은 단위로 커밋​

# Stripe SDK 설치
npm install @stripe/stripe-js @stripe/react-stripe-js

# 설치만 커밋
git add package.json package-lock.json
git commit -m "build: install Stripe SDK dependencies"
# 환경 설정
echo "STRIPE_PUBLIC_KEY=pk_test_..." >> .env.example
echo "STRIPE_SECRET_KEY=" >> .env.example

git add .env.example
git commit -m "config: add Stripe environment variables template"
# 기본 구조 생성
mkdir -p src/services/payment
touch src/services/payment/stripe.js
touch src/services/payment/index.js

git add src/services/payment/
git commit -m "feat: create payment service structure"

민지: "작은 단위로 커밋하니까 히스토리가 명확하네요!"

Day 3: 첫 번째 PR 생성​

4단계: Phase 1을 별도 PR로​

# Phase 1 작업을 별도 브랜치로
git checkout -b feature/stripe-setup
git cherry-pick <첫번째-커밋>..<마지막-커밋>

# PR 생성
gh pr create \
  --title "Phase 1: Stripe Integration Setup" \
  --body "## Changes
- Install Stripe SDK
- Add environment variables
- Create payment service structure

Part of #245 (Stripe Integration)

## Next Steps
Phase 2 will implement actual payment logic" \
  --base main

준호: "좋아요! 큰 feature를 작은 PR로 나누면 리뷰하기 훨씬 쉽죠."

Day 4: Main 브랜치 업데이트 반영​

준호 (Slack): "Main에 중요한 보안 업데이트가 머지됐어요. Feature 브랜치를 업데이트하세요."

5단계: Rebase로 최신 코드 반영​

# Main 브랜치 최신화
git checkout main
git pull origin main

# Feature 브랜치로 돌아가기
git checkout feature/stripe-integration

# Main의 변경사항을 rebase로 반영
git rebase main

# 충돌 없이 성공!
# Successfully rebased and updated refs/heads/feature/stripe-integration

# 강제 푸시 (개인 브랜치이므로 안전)
git push --force-with-lease

팁: 매일 아침 main을 rebase하여 최신 상태 유지!

Day 5: 충돌 발생!​

다른 개발자가 같은 파일을 수정했습니다.

6단계: Rebase 충돌 해결​

# Rebase 시도
git rebase main

# CONFLICT (content): Merge conflict in src/services/payment/index.js
# Could not apply abc1234... feat: add payment service

충돌 파일 내용:

<<<<<<< HEAD (main의 변경)
export { processRefund } from './refund';
export { validateCard } from './validation';
=======
export { createPayment } from './stripe';  (내 변경)
export { handleWebhook } from './webhook';
>>>>>>> feat: add payment service

해결:

// 모두 필요하므로 합치기
export { processRefund } from './refund';
export { validateCard } from './validation';
export { createPayment } from './stripe';
export { handleWebhook } from './webhook';
# 충돌 해결 후
git add src/services/payment/index.js

# Rebase 계속
git rebase --continue

# 다음 커밋 적용...
# 완료될 때까지 반복

Day 7: Phase 2 완료 및 PR​

7단계: 누적 커밋 정리 (Squash)​

# 현재 브랜치의 커밋들
git log --oneline origin/main..HEAD

# abc1234 feat: implement webhook handler
# def5678 fix: webhook validation
# ghi9012 fix: typo in webhook
# jkl3456 feat: add Payment Intent API
# mno7890 fix: error handling
# pqr1234 fix: improve error messages

# 너무 많은 작은 커밋들...

준호: "PR 전에 관련 커밋들을 정리하면 리뷰하기 좋아요."

# Interactive rebase로 정리
git rebase -i origin/main

# 에디터에서:
pick jkl3456 feat: add Payment Intent API
fixup mno7890 fix: error handling
fixup pqr1234 fix: improve error messages
pick abc1234 feat: implement webhook handler
fixup def5678 fix: webhook validation
fixup ghi9012 fix: typo in webhook

# 결과: 2개의 깔끔한 커밋
# jkl3456 feat: add Payment Intent API
# abc1234 feat: implement webhook handler

Day 10: 대규모 변경사항 통합​

8단계: Feature Toggle로 안전하게 배포​

// src/config/features.js
export const FEATURES = {
  USE_STRIPE_PAYMENT: process.env.REACT_APP_USE_STRIPE === 'true',
};

// src/components/Checkout.jsx
import { FEATURES } from '../config/features';
import { OldPayment } from './OldPayment';
import { StripePayment } from './StripePayment';

export function Checkout() {
  return FEATURES.USE_STRIPE_PAYMENT
    ? <StripePayment />
    : <OldPayment />;
}

민지: "이렇게 하면 프로덕션에 머지해도 안전하겠네요!"

git branch -a | grep stripe-integration

🎓 배운 내용​

대규모 Feature 관리 전략​

1. 작업 분할​

큰 프로젝트
  ├─ Phase 1: 기초 (PR #1)
  ├─ Phase 2: 백엔드 (PR #2)
  ├─ Phase 3: 프론트 (PR #3)
  └─ Phase 4: 테스트 (PR #4)

2. 정기적인 Rebase​

# 매일 아침 루틴
git checkout main
git pull
git checkout feature/my-feature
git rebase main

3. 커밋 정리​

# 작업 중: 작게 자주 커밋
git commit -m "wip: add function"
git commit -m "wip: fix bug"
git commit -m "wip: add test"

# PR 전: 의미있게 정리
git rebase -i origin/main
# → 1개의 완성된 커밋

4. Feature Toggle 패턴​

// 안전한 배포
if (FEATURES.NEW_FEATURE) {
  // 새 기능
} else {
  // 기존 기능
}

💡 실전 팁​

1. Daily Sync 루틴​

매일 작업 시작 전:

#!/bin/bash
# daily-sync.sh

echo "🔄 Syncing with main..."
git checkout main
git pull origin main

echo "🔄 Updating feature branch..."
git checkout feature/my-feature
git rebase main

if [ $? -eq 0 ]; then
  echo "✅ Sync successful!"
else
  echo "⚠️  Conflicts detected. Please resolve."
fi

2. PR 분할 전략​

❌ 나쁜 예: 1개의 거대한 PR (2000 lines)

✅ 좋은 예: 4개의 작은 PR
- PR #1: Setup (100 lines)
- PR #2: Backend (400 lines)
- PR #3: Frontend (500 lines)
- PR #4: Tests & Docs (200 lines)

3. WIP 커밋 활용​

# 퇴근 전
git add .
git commit -m "WIP: payment form - 70% complete"
git push

# 다음날
git reset HEAD~1  # WIP 커밋 취소
# 작업 계속...

4. 충돌 예방​

# 다른 사람이 작업 중인 파일 확인
gh pr list --state open
gh pr diff <PR-number>
