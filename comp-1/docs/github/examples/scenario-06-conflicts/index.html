시나리오 6: 팀 협업의 충돌

3명의 개발자가 같은 파일을 동시에 수정하면서 복잡한 충돌이 발생합니다.

📌 상황 설명​

월요일 스프린트 시작, 민지, 준호, 새로 합류한 지훈이 함께 대시보드 개편 작업을 시작합니다. 하지만 커뮤니케이션 부족으로 모두 같은 Dashboard.jsx 파일을 수정하게 됩니다!

🎯 목표​

  • 복잡한 3-way merge 이해하기
  • 효과적인 충돌 해결 전략
  • 팀원과 커뮤니케이션
  • 충돌 예방 방법 익히기
  • Visual merge tool 활용

💬 대화​

현우 (월요일 아침 스탠드업): "이번 주는 대시보드 개편입니다. 민지님은 차트, 준호님은 테이블, 지훈님은 필터 기능을 맡아주세요."

민지: "네, 차트 컴포넌트 작업하겠습니다!"

준호: "테이블 정렬 기능 추가할게요."

지훈: "필터 UI 만들겠습니다!"

3일 후...

민지 (Slack): "Dashboard.jsx 푸시했는데... 충돌이 엄청 많이 나요! 😱"

준호: "저도 막 푸시하려고 했는데..."

지훈: "앗, 저도 같은 파일 수정했어요..."

VS Code Merge Editor 활용​

지훈: "충돌이 너무 복잡해요... 😭"

준호: "VS Code의 Merge Editor를 사용해보세요. 시각적으로 보면 더 쉬워요."

VS Code에서 충돌 파일을 열면:

[Accept Current Change] [Accept Incoming Change]
[Accept Both Changes] [Compare Changes]

지훈, 3-way merge editor 사용:

┌─────────────────────────┬─────────────────────────┬─────────────────────────┐
│   Current (main)        │   Base (공통 조상)      │   Incoming (내 변경)    │
├─────────────────────────┼─────────────────────────┼─────────────────────────┤
│ charts + table          │ 원본 코드               │ filters                 │
└─────────────────────────┴─────────────────────────┴─────────────────────────┘
                                    ↓
                            [Result (최종)]

3. 작은 PR, 빠른 병합​

✅ Good Practice:
- 하루에 1-2개의 작은 PR
- 빠른 리뷰 및 병합
- 매일 main을 rebase

❌ Bad Practice:
- 1주일 동안 작업 후 한 번에 PR
- 큰 파일을 여러 명이 동시 수정

4. Pre-merge Hook​

# .git/hooks/pre-commit
#!/bin/sh

# 큰 파일 동시 수정 경고
LARGE_FILES="src/components/Dashboard.jsx"

for file in $LARGE_FILES; do
  if git diff --cached --name-only | grep -q "$file"; then
    echo "⚠️  Warning: Modifying large shared file: $file"
    echo "Please coordinate with team members!"
    echo "Continue? (y/n)"
    read answer
    if [ "$answer" != "y" ]; then
      exit 1
    fi
  fi
done

🎓 배운 내용​

3-Way Merge 이해​

       Base (공통 조상)
       /              \
  Current (HEAD)   Incoming (변경사항)
       \              /
         Result (병합 결과)

충돌 해결 전략​

상황해결 방법
간단한 충돌수동 편집
복잡한 충돌Merge tool 사용
전체 파일 선택git checkout --ours/--theirs
부분 선택Cherry-pick 특정 변경사항

VS Code Merge Editor 단축키​

- Alt + Left/Right: 변경사항 탐색
- Ctrl + Shift + P: Merge editor 열기
- Accept Current: 현재 브랜치 변경 수용
- Accept Incoming: 들어오는 변경 수용
- Accept Both: 둘 다 수용

💡 실전 팁​

1. 충돌 미리 감지​

# PR 생성 전 충돌 확인
git fetch origin
git merge-base HEAD origin/main
git diff --name-only origin/main...HEAD

# 다른 사람의 PR과 겹치는 파일 확인
gh pr list --state open | grep "Dashboard"

2. Merge vs Rebase 선택​

# Merge: 히스토리 보존, 충돌 한 번만
git merge origin/main

# Rebase: 깔끔한 히스토리, 각 커밋마다 충돌 가능
git rebase origin/main

3. 충돌 해결 템플릿​

4. 팀 협업 규칙​

# COLLABORATION.md

## 파일 수정 규칙
1. 큰 파일 수정 전 Slack에 공지
2. 동일 파일 작업 시 시간 조율
3. 가능하면 컴포넌트 분리
4. 하루 1회 이상 main rebase

## PR 규칙
1. 200줄 이하 유지
2. 24시간 내 리뷰
3. 충돌 발생 시 작성자가 해결
4. Squash merge 권장

5. 자동화 도구​

# .github/workflows/conflict-check.yml
name: Conflict Check

on:
  pull_request:
    types: [opened, synchronize]

jobs:
  check-conflicts:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - name: Check for conflicts
        run: |
          git fetch origin main
          if ! git merge-base --is-ancestor origin/main HEAD; then
            git merge --no-commit --no-ff origin/main || {
              echo "⚠️ This PR has conflicts with main"
              exit 1
            }
          fi
