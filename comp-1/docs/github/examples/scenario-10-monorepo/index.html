시나리오 10: Monorepo 관리

여러 프로젝트를 하나의 저장소로 통합하여 효율적으로 관리합니다.

📌 상황 설명​

코드크래프트가 성장하면서 프로젝트가 10개로 늘어났습니다. 각각 별도 저장소로 관리하다보니 공통 코드 동기화, 버전 관리, CI/CD 설정이 복잡해졌습니다. Monorepo로 전환하기로 결정!

🎯 목표​

  • Monorepo 구조 설계
  • 효율적인 빌드 전략
  • 선택적 체크아웃 (Sparse Checkout)
  • 독립적인 버전 관리
  • CI/CD 최적화

💬 대화​

현우 (PM): "10개 저장소 관리하느라 힘들지 않나요?"

수현: "맞아요. 공통 라이브러리 업데이트할 때마다 10개 저장소에 다 반영해야 해요."

준호: "Monorepo로 가면 좋을 것 같아요. Google, Facebook도 다 그렇게 하잖아요."

민지: "Monorepo가 뭔가요?"

준호: "여러 프로젝트를 하나의 저장소에서 관리하는 거예요. 보여드릴게요!"

⚡ 실습​

Phase 1: Monorepo 구조 설계​

1단계: 저장소 구조 계획​

# 기존 구조 (10개의 별도 저장소)
shop-frontend/
shop-backend/
shop-mobile-ios/
shop-mobile-android/
shared-ui-components/
shared-utils/
admin-dashboard/
analytics-service/
notification-service/
payment-service/

# 새 Monorepo 구조
codecraft-monorepo/
├── apps/
│   ├── web/              # shop-frontend
│   ├── mobile-ios/       # shop-mobile-ios
│   ├── mobile-android/   # shop-mobile-android
│   ├── admin/            # admin-dashboard
│   └── backend/          # shop-backend
├── services/
│   ├── analytics/        # analytics-service
│   ├── notification/     # notification-service
│   └── payment/          # payment-service
├── packages/
│   ├── ui-components/    # shared-ui-components
│   ├── utils/            # shared-utils
│   ├── types/            # 공통 타입 정의
│   └── config/           # 공통 설정
├── tools/
│   └── scripts/          # 빌드/배포 스크립트
└── package.json

Phase 2: 기존 저장소 통합​

2단계: Git History 보존하며 통합​

# 새 Monorepo 생성
mkdir codecraft-monorepo
cd codecraft-monorepo
git init

# 1번째 프로젝트 추가 (history 보존)
git remote add shop-frontend ../shop-frontend
git fetch shop-frontend
git merge --allow-unrelated-histories shop-frontend/main

# apps/web으로 이동
mkdir -p apps
git mv -k * apps/web/
git commit -m "chore: migrate shop-frontend to monorepo"

# 2번째 프로젝트 추가
git remote add shop-backend ../shop-backend
git fetch shop-backend

# Subtree로 추가 (history 보존)
git subtree add --prefix=apps/backend shop-backend main

# 반복...

3단계: Workspace 설정 (npm/yarn/pnpm)​

// package.json (root)
{
  "name": "codecraft-monorepo",
  "private": true,
  "workspaces": [
    "apps/*",
    "services/*",
    "packages/*"
  ],
  "scripts": {
    "build": "turbo run build",
    "test": "turbo run test",
    "lint": "turbo run lint",
    "dev": "turbo run dev --parallel"
  },
  "devDependencies": {
    "turbo": "^1.10.0",
    "prettier": "^3.0.0",
    "eslint": "^8.50.0"
  }
}
// turbo.json - 빌드 최적화
{
  "$schema": "https://turbo.build/schema.json",
  "pipeline": {
    "build": {
      "dependsOn": ["^build"],
      "outputs": ["dist/**", ".next/**"]
    },
    "test": {
      "dependsOn": ["build"],
      "cache": false
    },
    "lint": {
      "outputs": []
    },
    "dev": {
      "cache": false,
      "persistent": true
    }
  }
}

Phase 3: 선택적 체크아웃 (대용량 저장소)​

4단계: Sparse Checkout 설정​

# Sparse checkout 활성화
git sparse-checkout init --cone

# 특정 디렉토리만 체크아웃
git sparse-checkout set apps/web packages/ui-components

# 현재 설정 확인
git sparse-checkout list
# apps/web
# packages/ui-components

# 필요할 때 추가
git sparse-checkout add apps/mobile-ios

# 전체 체크아웃
git sparse-checkout disable

민지의 워크스테이션 (프론트엔드 개발자):

# 민지는 웹과 모바일만 필요
git sparse-checkout set apps/web apps/mobile-ios packages/ui-components

준호의 워크스테이션 (백엔드 개발자):

# 준호는 백엔드와 서비스만 필요
git sparse-checkout set apps/backend services packages/utils

Phase 5: 효율적인 CI/CD​

6단계: 변경된 패키지만 빌드​

# .github/workflows/ci.yml
name: CI

on:
  pull_request:
  push:
    branches: [main]

jobs:
  changes:
    runs-on: ubuntu-latest
    outputs:
      packages: ${{ steps.filter.outputs.changes }}
    steps:
      - uses: actions/checkout@v3
      - uses: dorny/paths-filter@v2
        id: filter
        with:
          filters: |
            web:
              - 'apps/web/**'
            backend:
              - 'apps/backend/**'
            ui-components:
              - 'packages/ui-components/**'

  build-web:
    needs: changes
    if: contains(needs.changes.outputs.packages, 'web')
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - name: Build Web
        run: |
          cd apps/web
          npm run build

  build-backend:
    needs: changes
    if: contains(needs.changes.outputs.packages, 'backend')
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - name: Build Backend
        run: |
          cd apps/backend
          npm run build

  # Turbo를 사용한 최적화
  turbo-build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - name: Setup Node
        uses: actions/setup-node@v3
      - name: Install dependencies
        run: npm install
      - name: Build
        run: npx turbo run build --filter=[HEAD^1]
        # 변경된 패키지만 빌드!

Phase 6: 공통 코드 관리​

7단계: Internal Packages​

# packages/utils/package.json
{
  "name": "@codecraft/utils",
  "version": "1.0.0",
  "main": "dist/index.js",
  "types": "dist/index.d.ts"
}

# apps/web/package.json
{
  "dependencies": {
    "@codecraft/utils": "*"  # Monorepo 내부 참조
  }
}
// packages/utils/src/index.ts
export function formatCurrency(amount: number) {
  return new Intl.NumberFormat('ko-KR', {
    style: 'currency',
    currency: 'KRW'
  }).format(amount);
}

// apps/web/src/components/Price.tsx
import { formatCurrency } from '@codecraft/utils';

export function Price({ amount }) {
  return <span>{formatCurrency(amount)}</span>;
}

Phase 7: Git 최적화​

8단계: Shallow Clone & Partial Clone​

# Shallow clone (빠른 CI)
git clone --depth 1 https://github.com/codecraft/monorepo.git

# Partial clone (대용량 파일 제외)
git clone --filter=blob:none https://github.com/codecraft/monorepo.git

# 특정 브랜치만
git clone --single-branch --branch main https://github.com/codecraft/monorepo.git

9단계: Git LFS for Binary Files​

# Git LFS 설치
git lfs install

# 추적할 파일 타입 지정
git lfs track "*.psd"
git lfs track "*.ai"
git lfs track "apps/mobile-ios/Assets/**"

# .gitattributes
cat .gitattributes
# *.psd filter=lfs diff=lfs merge=lfs -text
# *.ai filter=lfs diff=lfs merge=lfs -text

Phase 8: 문서화​

10단계: Monorepo 가이드 작성​

# Codecraft Monorepo 가이드

## 구조
- `apps/`: 실행 가능한 애플리케이션
- `services/`: 마이크로서비스
- `packages/`: 공유 라이브러리
- `tools/`: 개발 도구

## 시작하기

\`\`\`bash
# 전체 설치
npm install

# 개발 서버 (모든 앱)
npm run dev

# 특정 앱만
npm run dev --filter=web
\`\`\`

## 새 패키지 추가

\`\`\`bash
# 1. 디렉토리 생성
mkdir packages/new-package

# 2. package.json 생성
cd packages/new-package
npm init -y

# 3. Root에서 설치
cd ../..
npm install
\`\`\`

## 의존성 관리

\`\`\`bash
# 특정 패키지에 의존성 추가
npm install react --workspace=packages/ui-components

# 모든 패키지 업데이트
npm update --workspaces
\`\`\`

## 빌드 & 테스트

\`\`\`bash
# 모든 패키지 빌드
npm run build

# 변경된 패키지만
npm run build --filter=[HEAD^1]

# 특정 패키지와 의존하는 패키지
npm run build --filter=...ui-components
\`\`\`

✅ 검증​

# Workspace 구조 확인
npm ls --depth=0

# 의존성 그래프
npx turbo run build --dry=json --graph

# 빌드 시간 비교
# Before (10개 저장소): 각각 3분 = 30분
# After (Monorepo + Turbo): 8분 (캐시 사용 시 2분)

Monorepo vs Polyrepo​

특성MonorepoPolyrepo
저장소 수1개여러 개
공통 코드쉬움어려움
버전 관리통합독립적
CI/CD복잡하지만 최적화 가능간단
Clone 시간길 수 있음짧음
