시나리오 3: 실수했을 때

민지가 실수로 잘못된 작업을 하고, 당황하며 준호에게 도움을 요청합니다.

📌 상황 설명​

금요일 오후 4시, 민지는 피곤한 상태에서 여러 실수를 연속으로 저지릅니다:

  1. 잘못된 브랜치(main)에 커밋
  2. 중요한 설정 파일 삭제
  3. 이미 푸시한 커밋에 민감한 정보 포함
  4. Merge 대신 Rebase를 잘못 사용

🎯 목표​

  • 잘못된 커밋 되돌리기
  • 삭제한 파일 복구하기
  • 민감한 정보가 담긴 커밋 처리
  • Git 실수 복구 방법 익히기
  • 실수 방지 습관 기르기

💬 대화​

민지 (패닉 상태): "준호님! 큰일났어요! Main 브랜치에 직접 커밋했어요!"

준호: "괜찮아요, 침착하세요. 아직 푸시 안 했죠?"

민지: "아니요, 푸시는 안 했어요. 그리고... 설정 파일도 실수로 지웠어요. 😭"

준호: "다행히 Git에서 모든 걸 복구할 수 있어요. 하나씩 해결해봅시다."

⚡ 실습​

실수 1: 잘못된 브랜치에 커밋​

민지의 상황:

# 현재 상태 확인
git status
# On branch main  ← 😱 main에 있었다!
# Your branch is ahead of 'origin/main' by 1 commit.

git log --oneline -3
# abc1234 (HEAD -> main) feat: add user profile page  ← 이 커밋이 문제
# def5678 (origin/main) fix: correct payment calculation
# ghi9012 feat: add welcome message

준호: "걱정 마세요. 커밋을 새 브랜치로 옮기면 됩니다."

해결 방법 1: 브랜치를 만들고 Reset​

# 1. 현재 위치에서 새 브랜치 생성 (커밋 포함)
git branch feature/user-profile

# 2. Main을 origin/main으로 되돌리기
git reset --hard origin/main

# 3. 새 브랜치로 전환
git checkout feature/user-profile

# 4. 확인
git log --oneline -3
# abc1234 (HEAD -> feature/user-profile) feat: add user profile page  ← 여기에 있음
# def5678 (origin/main, main) fix: correct payment calculation
# ghi9012 feat: add welcome message

민지: "와! 커밋이 옮겨졌어요!"

실수 2: 중요한 파일 삭제​

민지: "그런데... 작업하다가 config/database.js 파일을 실수로 삭제했어요."

# 현재 상태
git status
# On branch feature/user-profile
# Changes not staged for commit:
#   deleted:    config/database.js

준호: "커밋 전이니까 쉽게 복구할 수 있어요."

해결 방법: Restore로 복구​

# 삭제한 파일 복구
git restore config/database.js

# 또는 (구버전 Git)
git checkout -- config/database.js

# 확인
ls config/
# database.js ← 복구됨!

git status
# On branch feature/user-profile
# nothing to commit, working tree clean

민지: "파일이 돌아왔어요! 정말 신기해요!"

실수 3: 커밋한 파일 삭제 복구​

민지: "아, 그런데 어제 커밋한 파일도 오늘 지웠는데, 그것도 복구할 수 있나요?"

# 2일 전에 커밋된 파일을 오늘 삭제
git log --oneline -- src/components/OldFeature.jsx
# xyz9876 feat: add old feature

# 파일이 언제 삭제되었는지 확인
git log --all --full-history -- src/components/OldFeature.jsx

해결 방법: 특정 커밋에서 파일 복구​

# 파일이 있던 마지막 커밋에서 복구
git restore --source=xyz9876 src/components/OldFeature.jsx

# 확인
ls src/components/
# OldFeature.jsx ← 복구됨!

실수 4: 잘못된 커밋 메시지​

민지: "아, 그리고 커밋 메시지에 오타가 있어요. 'proflie' 대신 'profile'이어야 하는데..."

git log --oneline -1
# abc1234 feat: add user proflie page  ← 오타!

해결 방법: Commit Amend​

# 마지막 커밋 메시지 수정
git commit --amend -m "feat: add user profile page"

# 확인
git log --oneline -1
# abc1234 feat: add user profile page  ← 수정됨!

준호: "아직 푸시 안 했으니까 amend가 안전해요. 푸시한 후엔 사용하면 안 됩니다!"

실수 5: 커밋에 파일 추가 깜빡​

민지: "아! CSS 파일을 커밋하는 걸 깜빡했어요!"

# 빠진 파일 추가
git add src/styles/UserProfile.css

# 마지막 커밋에 포함시키기
git commit --amend --no-edit

# 확인
git show --name-only
# commit abc1234
# feat: add user profile page
#
# src/components/UserProfile.jsx
# src/styles/UserProfile.css  ← 추가됨!

실수 6: 민감한 정보 커밋 (푸시 전)​

민지 (얼굴이 하얗게 질리며): "큰일났어요! API 키를 커밋했어요!"

git log --oneline -1
# abc1234 feat: add user profile with API key  ← 😱

# 파일 내용 확인
git show abc1234:src/config/api.js
# export const API_KEY = "sk-1234567890abcdef";  ← 민감한 정보!

준호: "다행히 아직 푸시 전이에요. Reset으로 해결할게요."

해결 방법: Soft Reset으로 커밋 취소​

# 커밋 취소 (파일은 유지)
git reset --soft HEAD~1

# 상태 확인
git status
# Changes to be committed:
#   modified:   src/components/UserProfile.jsx
#   modified:   src/config/api.js

# API 키 제거
code src/config/api.js
# API_KEY를 환경 변수로 변경

# .env 파일로 이동
echo "API_KEY=sk-1234567890abcdef" >> .env

# .gitignore에 추가
echo ".env" >> .gitignore

# 다시 커밋
git add .
git commit -m "feat: add user profile page

Use environment variables for API key"

실수 7: 이미 푸시한 민감한 정보​

다음날...

민지: "준호님, 어제 다른 브랜치에서 API 키를 푸시해버렸어요! 😭"

준호: "이건 좀 더 복잡해요. 히스토리를 다시 써야 합니다."

# 현재 상태
git log origin/feature/api-integration --oneline -3
# aaa1111 feat: API integration
# bbb2222 feat: add API key (← 이 커밋이 문제)
# ccc3333 feat: initial setup

해결 방법: Interactive Rebase로 커밋 수정​

# Interactive rebase 시작
git checkout feature/api-integration
git rebase -i HEAD~3

# 에디터가 열림:
# pick ccc3333 feat: initial setup
# edit bbb2222 feat: add API key  ← pick을 edit으로 변경
# pick aaa1111 feat: API integration

# 파일 수정
code src/config/api.js
# API_KEY 제거하고 환경 변수로 변경

# 변경사항 추가
git add src/config/api.js

# Rebase 계속
git commit --amend --no-edit
git rebase --continue

# 강제 푸시 (⚠️ 주의 필요)
git push --force-with-lease

준호: "팀원들에게 알려야 해요. 브랜치를 다시 받아야 합니다."

실수 8: 잘못된 Rebase​

민지: "실수로 rebase를 잘못했어요. 커밋들이 엉망이 됐어요!"

git log --oneline -5
# ??? 이상한 커밋들이...

해결 방법: Reflog로 복구​

# Reflog로 이전 상태 찾기
git reflog
# abc1234 HEAD@{0}: rebase: aborting
# def5678 HEAD@{1}: rebase: checkout main
# ghi9012 HEAD@{2}: commit: feat: add feature  ← 여기로 돌아가고 싶음

# 이전 상태로 복구
git reset --hard HEAD@{2}

# 확인
git log --oneline -3
# ghi9012 feat: add feature
# 정상 상태로 복구됨!

민지: "Reflog는 타임머신 같아요!"

✅ 검증​

복구 확인 체크리스트​

# 1. 브랜치 위치 확인
git branch --show-current
# feature/user-profile

# 2. Main이 깨끗한지 확인
git checkout main
git log --oneline -3
# origin/main과 동일해야 함

# 3. 민감한 정보 제거 확인
git log --all --full-history -S "API_KEY" --source --all
# .env와 .gitignore에만 있어야 함

# 4. 삭제한 파일 복구 확인
ls config/database.js
# 파일이 존재해야 함

🎓 배운 내용​

커밋 취소 방법​

명령어효과사용 시기
git reset --soft HEAD~1커밋만 취소, 파일 유지커밋 다시 하기
git reset --mixed HEAD~1커밋+스테이징 취소파일 재선택
git reset --hard HEAD~1모든 변경사항 삭제완전히 되돌리기
git revert HEAD새 커밋으로 취소푸시 후 취소

파일 복구 방법​

# 작업 디렉토리 변경 취소
git restore <file>

# 스테이징 취소
git restore --staged <file>

# 특정 커밋에서 복구
git restore --source=<commit> <file>

# 삭제된 파일 찾기
git log --all --full-history -- <file>

Reflog - 최후의 수단​

# 모든 작업 이력
git reflog

# 특정 시점으로 복구
git reset --hard HEAD@{n}

💡 실전 팁​

1. 실수 방지 Pre-commit Hook​

.git/hooks/pre-commit:

#!/bin/sh

# Main 브랜치 직접 커밋 방지
branch=$(git symbolic-ref HEAD | sed -e 's,.*/\(.*\),\1,')
if [ $branch = "main" ]; then
  echo "❌ Direct commit to main is not allowed!"
  echo "Create a feature branch instead."
  exit 1
fi

# 민감한 정보 검사
if git diff --cached | grep -E "API_KEY|PASSWORD|SECRET"; then
  echo "❌ Warning: Possible sensitive information detected!"
  exit 1
fi

exit 0

2. 안전한 작업 습관​

# 작업 전 항상 브랜치 확인
git branch --show-current

# 푸시 전 내용 확인
git log origin/branch..HEAD --oneline
git diff origin/branch..HEAD

# 중요한 작업 전 백업
git branch backup-$(date +%Y%m%d)

3. Alias로 실수 복구 쉽게​

# 유용한 alias 설정
git config --global alias.undo 'reset --soft HEAD~1'
git config --global alias.unstage 'restore --staged'
git config --global alias.last 'log -1 HEAD --stat'
git config --global alias.visual 'log --oneline --graph --all'

# 사용
git undo  # 마지막 커밋 취소
git unstage file.txt  # 스테이징 취소

4. 민감한 정보 완전 삭제​

이미 푸시한 경우:

# BFG Repo-Cleaner 사용
brew install bfg

# 민감한 정보 제거
bfg --replace-text passwords.txt
git reflog expire --expire=now --all
git gc --prune=now --aggressive
git push --force
