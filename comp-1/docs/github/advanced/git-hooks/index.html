Git Hooks

Git Hooks를 사용하여 Git 이벤트에 따라 자동화된 스크립트를 실행하는 방법을 배웁니다.

Git Hooks란?​

Git Hooks는 Git 이벤트가 발생할 때 자동으로 실행되는 스크립트입니다. 코드 품질 검증, 커밋 메시지 검사, 자동 배포 등에 활용됩니다.

Hooks 위치​

# 로컬 hooks
.git/hooks/

# 샘플 파일들
.git/hooks/pre-commit.sample
.git/hooks/commit-msg.sample
.git/hooks/pre-push.sample

Client-side Hooks​

pre-commit​

커밋 전에 실행됩니다. 코드 품질 검사에 주로 사용합니다.

.git/hooks/pre-commit:

#!/bin/sh

# 린트 검사
echo "Running linter..."
npm run lint
if [ $? -ne 0 ]; then
  echo "❌ Lint failed. Please fix errors before committing."
  exit 1
fi

# 테스트 실행
echo "Running tests..."
npm test
if [ $? -ne 0 ]; then
  echo "❌ Tests failed. Please fix before committing."
  exit 1
fi

echo "✅ All checks passed!"
exit 0

실행 권한 부여:

chmod +x .git/hooks/pre-commit

prepare-commit-msg​

커밋 메시지를 작성하기 전에 실행됩니다.

.git/hooks/prepare-commit-msg:

#!/bin/sh

COMMIT_MSG_FILE=$1
COMMIT_SOURCE=$2
SHA1=$3

# 브랜치 이름을 커밋 메시지에 추가
BRANCH_NAME=$(git symbolic-ref --short HEAD)

# 이미 메시지가 있으면 건너뛰기
if [ -z "$COMMIT_SOURCE" ]; then
  echo "[$BRANCH_NAME] $(cat $COMMIT_MSG_FILE)" > $COMMIT_MSG_FILE
fi

commit-msg​

커밋 메시지를 검증합니다.

.git/hooks/commit-msg:

#!/bin/sh

COMMIT_MSG_FILE=$1
COMMIT_MSG=$(cat $COMMIT_MSG_FILE)

# Conventional Commits 검증
PATTERN="^(feat|fix|docs|style|refactor|test|chore)(\(.+\))?: .{10,}"

if ! echo "$COMMIT_MSG" | grep -qE "$PATTERN"; then
  echo "❌ Invalid commit message format!"
  echo "Format: <type>(<scope>): <subject>"
  echo "Example: feat(auth): add login functionality"
  exit 1
fi

echo "✅ Commit message is valid"
exit 0

pre-push​

푸시 전에 실행됩니다.

.git/hooks/pre-push:

#!/bin/sh

# 보호된 브랜치에 푸시 방지
protected_branch='main'
current_branch=$(git symbolic-ref HEAD | sed -e 's,.*/\(.*\),\1,')

if [ $current_branch = $protected_branch ]; then
  echo "❌ Direct push to $protected_branch is not allowed!"
  echo "Please create a Pull Request instead."
  exit 1
fi

# 모든 테스트 실행
echo "Running all tests before push..."
npm test
if [ $? -ne 0 ]; then
  echo "❌ Tests failed. Push aborted."
  exit 1
fi

echo "✅ All checks passed. Pushing..."
exit 0

Server-side Hooks​

pre-receive​

서버가 푸시를 받기 전에 실행됩니다.

#!/bin/sh

# 커밋 크기 제한
while read oldrev newrev refname; do
  # 큰 파일 검사
  if git diff --name-only $oldrev $newrev | xargs git show :| grep -q "\.zip\|\.exe"; then
    echo "❌ Binary files are not allowed!"
    exit 1
  fi
done

update​

각 브랜치마다 실행됩니다.

#!/bin/sh

refname=$1
oldrev=$2
newrev=$3

# main 브랜치는 fast-forward만 허용
if [ "$refname" = "refs/heads/main" ]; then
  if ! git merge-base --is-ancestor $oldrev $newrev; then
    echo "❌ Only fast-forward merges are allowed on main!"
    exit 1
  fi
fi

exit 0

post-receive​

푸시가 완료된 후 실행됩니다. 배포에 주로 사용합니다.

#!/bin/sh

# 자동 배포
while read oldrev newrev refname; do
  branch=$(git rev-parse --symbolic --abbrev-ref $refname)

  if [ "$branch" = "main" ]; then
    echo "Deploying to production..."
    cd /var/www/app
    git pull
    npm install
    npm run build
    pm2 restart app
  fi
done

실습 예제​

예제 1: 린터 자동 실행​

.git/hooks/pre-commit:

#!/bin/sh

FILES=$(git diff --cached --name-only --diff-filter=ACM | grep '\.js$\|\.ts$')

if [ -n "$FILES" ]; then
  echo "Running ESLint..."
  echo "$FILES" | xargs npx eslint

  if [ $? -ne 0 ]; then
    echo "❌ ESLint failed. Please fix errors."
    exit 1
  fi

  echo "Running Prettier..."
  echo "$FILES" | xargs npx prettier --check

  if [ $? -ne 0 ]; then
    echo "❌ Code formatting issues. Run 'npm run format'"
    exit 1
  fi
fi

echo "✅ All checks passed!"
exit 0

예제 2: 커밋 메시지에 Issue 번호 필수​

.git/hooks/commit-msg:

#!/bin/sh

COMMIT_MSG=$(cat $1)

# Issue 번호 확인 (#123 형식)
if ! echo "$COMMIT_MSG" | grep -qE "#[0-9]+"; then
  echo "❌ Commit message must include issue number (e.g., #123)"
  exit 1
fi

exit 0

예제 3: 민감한 정보 검사​

.git/hooks/pre-commit:

#!/bin/sh

# 민감한 정보 패턴
PATTERNS=(
  "API_KEY"
  "SECRET_KEY"
  "PASSWORD"
  "aws_secret"
  "private_key"
)

for pattern in "${PATTERNS[@]}"; do
  if git diff --cached | grep -iE "$pattern"; then
    echo "❌ Warning: Possible secret detected: $pattern"
    echo "Please remove sensitive information before committing."
    exit 1
  fi
done

exit 0

Hooks 공유하기​

Git hooks는 기본적으로 .git/hooks/에 있어 버전 관리되지 않습니다. 팀과 공유하려면:

방법 1: 스크립트 디렉토리​

# 1. 프로젝트 루트에 hooks 디렉토리 생성
mkdir .githooks

# 2. hooks 작성
cat > .githooks/pre-commit << 'EOF'
#!/bin/sh
npm run lint && npm test
EOF

# 3. 실행 권한
chmod +x .githooks/pre-commit

# 4. Git 설정
git config core.hooksPath .githooks

# 5. README에 안내
echo "Run: git config core.hooksPath .githooks" >> README.md

방법 2: Husky 사용 (Node.js)​

# 설치
npm install --save-dev husky

# 초기화
npx husky install

# Hook 추가
npx husky add .husky/pre-commit "npm test"
npx husky add .husky/commit-msg 'npx commitlint --edit $1'

# package.json에 추가
npm pkg set scripts.prepare="husky install"

package.json:

{
  "scripts": {
    "prepare": "husky install"
  },
  "devDependencies": {
    "husky": "^8.0.0",
    "@commitlint/cli": "^17.0.0",
    "@commitlint/config-conventional": "^17.0.0"
  }
}

방법 3: pre-commit Framework​

# 설치
pip install pre-commit

# .pre-commit-config.yaml 생성
cat > .pre-commit-config.yaml << 'EOF'
repos:
  - repo: https://github.com/pre-commit/pre-commit-hooks
    rev: v4.4.0
    hooks:
      - id: trailing-whitespace
      - id: end-of-file-fixer
      - id: check-yaml
      - id: check-added-large-files

  - repo: https://github.com/pre-commit/mirrors-eslint
    rev: v8.44.0
    hooks:
      - id: eslint
        files: \.[jt]sx?$
        types: [file]
EOF

# 설치
pre-commit install

# 수동 실행
pre-commit run --all-files

고급 Hooks​

조건부 실행​

#!/bin/sh

# CI 환경에서는 건너뛰기
if [ -n "$CI" ]; then
  echo "Skipping hooks in CI environment"
  exit 0
fi

# 브랜치별로 다른 검사
BRANCH=$(git symbolic-ref --short HEAD)

if [[ $BRANCH == feature/* ]]; then
  npm run lint
elif [[ $BRANCH == hotfix/* ]]; then
  npm test -- --critical-only
fi

여러 명령어 실행​

#!/bin/sh

run_check() {
  echo "Running $1..."
  $2
  if [ $? -ne 0 ]; then
    echo "❌ $1 failed!"
    exit 1
  fi
  echo "✅ $1 passed"
}

run_check "Linter" "npm run lint"
run_check "Type Check" "npm run type-check"
run_check "Tests" "npm test"
run_check "Build" "npm run build"

echo "✅ All checks passed!"
exit 0

성능 최적화​

#!/bin/sh

# 변경된 파일만 검사
STAGED_FILES=$(git diff --cached --name-only --diff-filter=ACM | grep '\.js$')

if [ -z "$STAGED_FILES" ]; then
  echo "No JS files to check"
  exit 0
fi

# 병렬 실행
echo "$STAGED_FILES" | xargs -P 4 -n 10 npx eslint

exit $?

디버깅​

Hook 실행 확인​

# Hook 실행 여부 확인
GIT_TRACE=1 git commit -m "test"

# Hook 스크립트 디버그
#!/bin/sh
set -x  # 명령어 출력
# ... hooks 코드 ...

Hook 일시 비활성화​

# 한 번만 건너뛰기
git commit --no-verify -m "skip hooks"

# 또는
HUSKY=0 git commit -m "skip husky hooks"

Best Practices​

1. 빠르게 유지​

# ❌ 느린 hook
npm test  # 모든 테스트 (5분)

# ✅ 빠른 hook
npm run test:changed  # 변경된 파일만 (10초)

2. 명확한 에러 메시지​

# ❌ 나쁜 예
exit 1

# ✅ 좋은 예
echo "❌ Lint failed. Run 'npm run lint:fix' to auto-fix issues."
echo "Affected files:"
echo "$FILES"
exit 1

3. 선택적 실행​

# 환경 변수로 제어
if [ "$SKIP_HOOKS" != "true" ]; then
  npm run lint
fi

4. 문서화​

# Hooks

## Pre-commit
- Runs ESLint
- Runs Prettier
- Runs type checking

## Commit-msg
- Validates conventional commit format

## Pre-push
- Runs all tests
- Checks for TODO comments in main branch

## Skip hooks
\`\`\`bash
git commit --no-verify
# or
SKIP_HOOKS=true git commit
\`\`\`
