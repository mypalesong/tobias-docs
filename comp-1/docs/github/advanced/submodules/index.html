Git Submodules

Git Submodules를 사용하여 프로젝트 내에서 다른 Git 저장소를 관리하는 방법을 배웁니다.

Submodules란?​

Submodule은 Git 저장소 안에 다른 Git 저장소를 포함시키는 방법입니다. 라이브러리, 공통 코드, 의존성 관리에 사용됩니다.

기본 사용법​

Submodule 추가​

# 기본 추가
git submodule add https://github.com/user/library.git lib

# 특정 브랜치 지정
git submodule add -b main https://github.com/user/library.git lib

# 특정 경로에 추가
git submodule add https://github.com/user/library.git path/to/lib

이 명령은:

  1. lib 디렉토리에 저장소를 클론
  2. .gitmodules 파일 생성/업데이트
  3. Submodule을 현재 저장소에 등록

Submodule이 있는 저장소 클론​

# 방법 1: 클론 후 초기화
git clone https://github.com/user/project.git
cd project
git submodule init
git submodule update

# 방법 2: 한 번에
git clone --recursive https://github.com/user/project.git

# 방법 3: 이미 클론한 경우
git submodule update --init --recursive

Submodule 업데이트​

# 모든 submodule 업데이트
git submodule update --remote

# 특정 submodule만 업데이트
git submodule update --remote lib

# 최신 버전으로 업데이트
git submodule update --remote --merge

.gitmodules 파일​

[submodule "lib"]
    path = lib
    url = https://github.com/user/library.git
    branch = main

실습 예제​

예제 1: 라이브러리 추가​

# 1. 프로젝트 생성
mkdir my-project
cd my-project
git init

# 2. Submodule 추가
git submodule add https://github.com/user/ui-library.git libs/ui

# 3. 커밋
git add .gitmodules libs/ui
git commit -m "Add UI library as submodule"

# 4. 사용
import { Button } from './libs/ui/button'

예제 2: 팀원이 프로젝트 클론​

# 1. 저장소 클론
git clone https://github.com/team/project.git
cd project

# 2. Submodule 초기화
git submodule init

# 3. Submodule 내용 가져오기
git submodule update

# 또는 한 번에
git submodule update --init --recursive

예제 3: Submodule 업데이트​

# 1. Submodule 디렉토리로 이동
cd libs/ui

# 2. 최신 변경사항 가져오기
git fetch
git merge origin/main

# 3. 부모 저장소로 돌아가기
cd ../..

# 4. 변경사항 커밋
git add libs/ui
git commit -m "Update UI library to latest version"
git push

Submodule 작업하기​

Submodule에서 직접 작업​

# 1. Submodule 디렉토리로 이동
cd libs/ui

# 2. 브랜치 생성
git checkout -b feature/new-button

# 3. 작업 및 커밋
echo "new button" > button.js
git add button.js
git commit -m "Add new button"

# 4. Submodule 저장소에 푸시
git push origin feature/new-button

# 5. 부모 저장소로 돌아가기
cd ../..

# 6. Submodule 포인터 업데이트
git add libs/ui
git commit -m "Update UI library submodule"

특정 커밋으로 고정​

# Submodule을 특정 커밋으로 설정
cd libs/ui
git checkout abc123
cd ../..
git add libs/ui
git commit -m "Pin UI library to version abc123"

고급 기능​

병렬로 Submodule 업데이트​

# 여러 submodule을 병렬로 업데이트
git submodule update --init --recursive --jobs 4

Submodule 상태 확인​

# 모든 submodule 상태
git submodule status

# 출력 예시:
#  abc1234 libs/ui (v1.0.0)
# +def5678 libs/auth (v2.1.0-dirty)
# -ghi9012 libs/utils

# Legend:
# ' ' - 동기화됨
# '+' - 다른 커밋 체크아웃
# '-' - 초기화 안 됨
# 'U' - 병합 충돌

Submodule 변경사항 확인​

# Submodule의 변경사항 확인
git diff --submodule

# 더 상세하게
git config --global diff.submodule log

Submodule 제거​

# 1. Submodule 등록 해제
git submodule deinit libs/ui

# 2. .gitmodules에서 제거
git rm libs/ui

# 3. .git/modules 정리
rm -rf .git/modules/libs/ui

# 4. 커밋
git commit -m "Remove UI library submodule"

실전 패턴​

패턴 1: 공통 라이브러리 관리​

# 프로젝트 구조
project/
├── app/
├── libs/
│   ├── ui-components/     (submodule)
│   ├── utils/             (submodule)
│   └── api-client/        (submodule)
└── .gitmodules

# Submodule 추가
git submodule add https://github.com/org/ui-components.git libs/ui-components
git submodule add https://github.com/org/utils.git libs/utils
git submodule add https://github.com/org/api-client.git libs/api-client

패턴 2: 모노레포의 일부 공유​

# main-app에서 shared-lib를 submodule로
cd main-app
git submodule add https://github.com/org/monorepo.git shared
cd shared
git sparse-checkout init --cone
git sparse-checkout set packages/shared-lib

패턴 3: 테마/플러그인 시스템​

# WordPress 스타일
project/
├── themes/
│   ├── theme1/    (submodule)
│   └── theme2/    (submodule)
└── plugins/
    ├── plugin1/   (submodule)
    └── plugin2/   (submodule)

자동화​

CI/CD에서 Submodule 처리​

# .github/workflows/ci.yml
name: CI

on: [push]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
        with:
          submodules: recursive

      - name: Run tests
        run: npm test

Pre-commit Hook​

# .git/hooks/pre-commit
#!/bin/sh

# Submodule이 최신인지 확인
git submodule foreach 'git fetch && [ $(git rev-parse HEAD) = $(git rev-parse @{u}) ] || exit 1'

if [ $? -ne 0 ]; then
  echo "Warning: Some submodules are not up to date!"
  echo "Run: git submodule update --remote"
  exit 1
fi

Submodules vs Alternatives​

Submodules​

# 장점:
# - Git 네이티브 지원
# - 정확한 버전 고정
# - 각 저장소 독립적 관리

# 단점:
# - 복잡한 워크플로우
# - 초보자에게 어려움
# - 쉽게 동기화 누락

Git Subtree​

# 대안: Git Subtree
git subtree add --prefix=lib https://github.com/user/library.git main

# 장점:
# - 단순한 워크플로우
# - 서브트리 내용이 메인 저장소에 포함

# 단점:
# - 히스토리가 복잡해짐
# - 업데이트가 어려움

Package Manager​

# 대안: npm, yarn, pip 등
npm install @org/ui-library

# 장점:
# - 간단한 사용
# - 버전 관리 용이

# 단점:
# - 패키지 레지스트리 필요
# - 소스 코드 접근 제한적

트러블슈팅​

문제 1: Detached HEAD​

# Submodule이 detached HEAD 상태
cd libs/ui

# 브랜치로 전환
git checkout main

# 또는 설정으로 자동화
git config submodule.libs/ui.branch main

문제 2: 변경사항 손실​

# Submodule에 커밋 안 한 변경사항이 있을 때
git submodule update
# warning: unable to rmdir 'libs/ui': Directory not empty

# 해결: 먼저 커밋하거나 stash
cd libs/ui
git stash
cd ../..
git submodule update
cd libs/ui
git stash pop

문제 3: URL 변경​

# Submodule URL이 변경된 경우
# .gitmodules 수정
[submodule "lib"]
    url = https://new-url.git

# 동기화
git submodule sync
git submodule update --init --recursive

Best Practices​

1. README 문서화​

# Setup

Clone with submodules:
\`\`\`bash
git clone --recursive https://github.com/user/project.git
\`\`\`

Update submodules:
\`\`\`bash
git submodule update --remote
\`\`\`

2. 브랜치 고정​

[submodule "lib"]
    path = lib
    url = https://github.com/user/library.git
    branch = stable  # 안정 브랜치 사용

3. 정기적 업데이트​

# 매주 업데이트 확인
git submodule update --remote
git submodule status

4. CI 설정​

# 항상 최신 submodule 사용
steps:
  - uses: actions/checkout@v3
    with:
      submodules: true
