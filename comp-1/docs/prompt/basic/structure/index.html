프롬프트 구조

효과적인 프롬프트를 구성하는 방법

좋은 내용도 잘못된 구조로 전달하면 효과가 반감됩니다. 이 가이드에서는 프롬프트를 논리적이고 명확하게 구조화하는 방법을 배웁니다.

🎯 학습 목표​

  • 프롬프트의 핵심 구성 요소 이해
  • 다양한 구조 패턴과 적용 사례
  • 복잡한 요청을 체계적으로 정리하는 방법
  • 재사용 가능한 프롬프트 템플릿 작성

📦 프롬프트의 핵심 구성 요소​

기본 구조​

prompt_structure = {
    "1. 역할/맥락": "AI의 역할이나 상황 설정",
    "2. 작업": "수행해야 할 구체적인 작업",
    "3. 요구사항": "작업 수행 시 지켜야 할 조건",
    "4. 출력 형식": "결과물의 형태",
    "5. 제약사항": "하지 말아야 할 것, 제한사항",
    "6. 예제": "(선택) 참고할 수 있는 예시"
}

각 요소의 역할​

구성 요소목적필수 여부
역할/맥락AI의 전문성 방향 설정선택 (권장)
작업무엇을 해야 하는지 명확히✅ 필수
요구사항어떻게 해야 하는지 상세화✅ 필수
출력 형식결과물의 구조 정의권장
제약사항피해야 할 것 명시선택
예제패턴 학습 지원선택

# 프로젝트: 모놀리식 → 마이크로서비스 전환

## 🎯 목표
기존 Django 모놀리식 애플리케이션을 마이크로서비스 아키텍처로 전환

---

## 📊 현재 상황

### 시스템 구성
- **아키텍처**: Django 모놀리식
- **데이터베이스**: PostgreSQL (단일 DB)
- **사용자 수**: 일일 50,000명
- **주요 기능**:
  1. 사용자 인증/관리
  2. 제품 카탈로그
  3. 주문 처리
  4. 결제
  5. 배송 관리

### 문제점
- 주문 처리 시 전체 시스템 느려짐 (응답 시간 5초+)
- 결제 모듈 장애 시 전체 서비스 다운
- 배포 시 전체 서비스 중단 필요 (다운타임 30분)
- 특정 기능만 스케일링 불가능

---

## 🎯 전환 목표

### 비즈니스 목표
1. 주문 처리 성능 개선 (응답 시간 1초 이하)
2. 부분 장애 격리 (한 서비스 장애 시 다른 서비스 정상 작동)
3. 무중단 배포 가능
4. 트래픽에 따른 개별 서비스 스케일링

### 기술 목표
- 서비스 간 느슨한 결합
- 독립적인 데이터베이스
- API Gateway 도입
- 이벤트 기반 통신

---

## 🛠️ 기술 환경

### 현재 스택
- Backend: Django 4.2
- Database: PostgreSQL 15
- Server: AWS EC2
- Deployment: Git push → Manual deployment

### 전환 예정 스택
- Backend: FastAPI (새 서비스)
- Database: PostgreSQL (서비스별 분리)
- Container: Docker
- Orchestration: AWS ECS
- Message Queue: ?  (추천 필요)
- API Gateway: ?  (추천 필요)

---

## 📋 요청사항

### 1단계: 아키텍처 설계
다음을 포함한 마이크로서비스 아키텍처를 설계해주세요:

#### 서비스 분할
- 어떤 기준으로 서비스를 나눌 것인가?
- 각 서비스의 책임 범위는?
- 서비스 간 의존성은?

#### 데이터 관리
- 데이터베이스 분리 전략
- 데이터 일관성 유지 방법
- 분산 트랜잭션 처리

#### 통신 방식
- 동기 vs 비동기 통신 선택 기준
- Message Queue 추천 (RabbitMQ vs Kafka vs AWS SQS)
- API Gateway 추천 (Kong vs AWS API Gateway)

### 2단계: 마이그레이션 계획
#### 단계별 전환 로드맵
- 어떤 서비스부터 분리할 것인가?
- 각 단계별 소요 시간 추정
